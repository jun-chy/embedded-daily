## 一、项目简介

**tinyrtos** 是一个面向 ARM Cortex-M 架构的紧凑型抢占式优先级 RTOS 内核，完全从零开始用 C 语言编写。项目初衷是深入理解 RTOS 内部原理，逐步演化为一个追求确定性、静态分配、可读性和低开销的实用嵌入式内核。

与 FreeRTOS、Zephyr 等成熟 RTOS 不同，tinyrtos 刻意贴近硬件，直接暴露底层执行模型，不追求广泛的可移植性或庞大的中间件生态。核心理念是：**确定性、小内存占用、静态分配、可理解的架构、实用的嵌入式可用性、最小的抽象开销。**

### 核心特性一览

| 特性 | 描述 |
|------|------|
| 抢占式优先级调度 | 高优先级任务自动抢占低优先级任务 |
| 同优先级轮转调度 | 同优先级任务之间轮转执行 |
| PendSV 上下文切换 | 利用 Cortex-M PendSV 异常实现低开销切换 |
| SVC 任务启动 | 通过 SVC 异常启动第一个任务 |
| 独立 PSP 任务栈 | 每个任务使用独立的 Process Stack Pointer |
| 阻塞与睡眠 | 支持任务阻塞和定时唤醒 |
| 计数信号量 | ISR 安全的信号量同步 |
| 互斥锁 | 支持互斥访问共享资源 |
| 队列 | 任务间数据传递，支持 ISR 版本 |
| 栈水位检测 | 使用 DEADBEEF 模式检测栈使用量 |
| HardFault 诊断 | 自动转储寄存器并解码故障原因 |
| OSI 运行时 Shell | UART 串口运行时诊断接口 |

### 内核 footprint

## 四、核心代码深度分析

### 4.1 任务控制块（TCB）数据结构

任务控制块是 RTOS 的核心数据结构，定义在 `include/tinyrtos/kernel/task.h` 中：

```c
typedef struct {
    uint32_t * sp;                          // 任务栈指针，切换时加载到 PSP
    uint32_t stack[RTOSK_TASK_STACK_WORDS]; // 任务栈缓冲区（256 words = 1KB）
    rtosk_task_state_t state;               // 当前任务状态
    uint32_t wake_tick;                     // 阻塞到期唤醒 tick
    uint32_t priority;                      // 静态优先级（数字越大优先级越高）
    const char * name;                      // 任务名称
} rtosk_task_t;
```

**逐字段解析：**

- `sp`：保存任务被切换出时的栈指针。当调度器选中该任务时，此值会被加载到 PSP 寄存器。由于 Cortex-M 的硬件自动保存 R0-R3、R12、LR、PC、xPSR（8 个字），软件还需保存 R4-R11（8 个字），因此 `sp` 始终指向软件保存帧的起始位置。
- `stack[256]`：每个任务静态分配 256 个 32 位字（1KB）的栈空间。所有任务栈在编译时确定，**零动态分配**。
- `state`：任务状态枚举，包括 `READY`、`BLOCKED`（睡眠）、`BLOCKED_ON_SEMAPHORE`、`BLOCKED_ON_QUEUE`、`BLOCKED_ON_MUTEX`、`IDLE`。
- `wake_tick`：当任务因 `sleep_ms` 阻塞时，记录应被唤醒的绝对 tick 值。SysTick 中断中会比较此值来决定是否唤醒。
- `priority`：静态优先级。**数字越大优先级越高**，调度器总是选择就绪状态中优先级最高的任务。

任务状态的枚举定义：

```c
typedef enum {
    RTOSK_TASK_READY = 0U,              // 就绪，可被调度
    RTOSK_TASK_BLOCKED,                 // 睡眠阻塞，等待 wake_tick
    RTOSK_TASK_BLOCKED_ON_SEMAPHORE,    // 等待信号量
    RTOSK_TASK_BLOCKED_ON_QUEUE,        // 等待队列数据
    RTOSK_TASK_BLOCKED_ON_MUTEX,        // 等待互斥锁
    RTOSK_TASK_IDLE                     // 空闲任务
} rtosk_task_state_t;
```

内核使用静态数组存储所有任务：

```c
static rtosk_task_t RTOSK_TASKS[RTOSK_TOTAL_TASK_SLOTS]; // 5个槽位（4用户+1空闲）
static uint32_t RTOSK_TASK_COUNT   = 0UL;  // 已创建任务数
static uint32_t RTOSK_CURRENT_TASK = 0UL;  // 当前运行任务索引
static uint32_t RTOSK_IDLE_READY   = 0UL;  // 空闲任务是否已初始化
```

宏定义限制了最大任务数为 4，加上空闲任务共 5 个槽位：

```c
#define RTOSK_MAX_TASKS         4UL
#define RTOSK_IDLE_TASK_INDEX   RTOSK_MAX_TASKS  // 空闲任务在索引 4
#define RTOSK_TOTAL_TASK_SLOTS  (RTOSK_MAX_TASKS + 1UL)  // 共 5 个槽位
```

### 4.2 任务栈帧构建 —— Cortex-M 异常帧模拟

这是 tinyrtos 最精妙的部分之一。当一个新任务被创建时，需要手动构建一个合法的 Cortex-M 异常栈帧，使得 SVC_Handler 返回时能"假装"该任务正在被从中断恢复执行。

```c
static uint32_t * rtosk_task_build_stack(rtosk_task_t * task, rtosk_task_func_t task_func) {
    // Cortex-M 栈向下生长，指针从栈顶开始
    uint32_t * sp = &task->stack[RTOSK_TASK_STACK_WORDS];
    sp = rtosk_port_align_stack_pointer(sp);  // 8 字节对齐

    // 构建 8 字硬件异常帧
    *(--sp) = RTOSK_XPSR_T_BIT;                    // xPSR: 设置 Thumb bit (bit 24 = 1)
    *(--sp) = (uint32_t)task_func;                  // PC: 任务函数入口地址
    *(--sp) = (uint32_t)rtosk_task_exit_trap;       // LR: 任务返回地址（死循环陷阱）
    *(--sp) = 0x00000000UL;                         // R12
    *(--sp) = 0x00000000UL;                         // R3
    *(--sp) = 0x00000000UL;                         // R2
    *(--sp) = 0x00000000UL;                         // R1
    *(--sp) = 0x00000000UL;                         // R0

    // 构建 8 字软件保存帧（R4-R11）
    *(--sp) = 0x00000000UL;  // R11
    *(--sp) = 0x00000000UL;  // R10
    *(--sp) = 0x00000000UL;  // R9
    *(--sp) = 0x00000000UL;  // R8
    *(--sp) = 0x00000000UL;  // R7
    *(--sp) = 0x00000000UL;  // R6
    *(--sp) = 0x00000000UL;  // R5
    *(--sp) = 0x00000000UL;  // R4

    return sp;  // 返回 R4 的地址（软件帧起始）
}
```

**逐行深度解析：**

1. **栈方向**：Cortex-M 的栈是**向下生长**的（full-descending stack），所以 `sp` 从数组末尾 `&task->stack[256]` 开始，通过 `*(--sp)` 逐字递减填充。

2. **8 字节对齐**：`rtosk_port_align_stack_pointer` 通过 `sp & ~0x7UL` 将指针对齐到 8 字节边界。AAPCS（ARM 构架过程调用标准）要求栈指针在函数调用时必须 8 字节对齐，否则可能触发对齐故障。

3. **xPSR（程序状态寄存器）**：`RTOSK_XPSR_T_BIT` 设置了 Thumb bit（bit 24）。Cortex-M 只支持 Thumb 指令集，如果此位为 0，尝试从该栈帧恢复 PC 时会触发 InvState HardFault。

4. **PC = task_func**：将任务函数地址放入 PC 位置。当异常返回时，CPU 从栈中弹出此值到 PC，开始执行任务函数。

5. **LR = rtosk_task_exit_trap**：如果任务函数不小心 return 了（而不是无限循环），LR 会被加载到 PC，进入一个死循环陷阱，防止任务返回到不可预测的地址。

6. **R0-R3, R12 = 0**：这些寄存器初始值清零。在实际任务中，它们会被任务函数内的代码覆盖。

7. **R4-R11 = 0**：软件保存帧，这些寄存器在 Cortex-M 异常进出时不会被硬件自动保存/恢复，需要 PendSV_Handler 中的汇编代码手动处理。

构建完成后的栈帧布局：

```
高地址（栈底）
  ┌──────────┐
  │  未使用    │ ← DEADBEEF 水印区域
  │  ...      │
  ├──────────┤
  │ R4       │ ← sp 指向这里（软件帧起始）
  │ R5       │
  │ R6       │
  │ R7       │
  │ R8       │
  │ R9       │
  │ R10      │
  │ R11      │
  │ R0       │ ← 硬件帧起始
  │ R1       │
  │ R2       │
  │ R3       │
  │ R12      │
  │ LR       │ = rtosk_task_exit_trap
  │ PC       │ = task_func
  │ xPSR     │ = Thumb bit set
  └──────────┘
低地址（栈顶）
```

### 4.3 SVC_Handler —— 启动第一个任务

内核启动时，`rtosk_kernel_start()` 调用 `rtosk_port_start_first_task()`，后者执行 `svc #0` 指令触发 SVC 异常。SVC_Handler 负责从第一个任务的栈帧恢复执行：

```c
__attribute__((naked)) void SVC_Handler(void) {
    __asm volatile (
        // 调用 C 函数获取当前任务的保存栈指针
        "ldr r0, =rtosk_task_get_stack_pointer \n"
        "blx r0                                \n"
        // 从任务栈恢复软件保存的 R4-R11
        "ldmia r0!, {r4-r11}                   \n"
        // r0 现在指向硬件异常帧，设置 PSP
        "msr psp, r0                           \n"
        // 读取 CONTROL 寄存器
        "mrs r0, control                       \n"
        // 设置 CONTROL.SPSEL = 1（Thread 模式使用 PSP）
        "orr r0, r0, #2                        \n"
        // 写回 CONTROL
        "msr control, r0                       \n"
        // 指令同步屏障，确保 CONTROL 变更立即生效
        "isb                                   \n"
        // 加载异常返回码：返回 Thread 模式，使用 PSP
        "ldr lr, =0xFFFFFFFD                   \n"
        // 异常返回，CPU 从 PSP 恢复 R0-R3/R12/LR/PC/xPSR
        "bx lr                                 \n"
    );
}
```

**逐行解析：**

1. `naked` 属性：告诉编译器不要为此函数生成 prologue/epilogue（不需要 push/pop 栈帧），因为我们完全用内联汇编控制。

2. `ldr r0, =rtosk_task_get_stack_pointer` + `blx r0`：调用 C 函数 `rtosk_task_get_stack_pointer()`，返回值（当前任务的 sp 指针）放入 R0。

3. `ldmia r0!, {r4-r11}`：**Load Multiple Increment After**，从 R0 指向的地址依次加载 R4-R11（8 个寄存器），同时 R0 自增 32 字节。执行后 R0 指向硬件异常帧的起始（R0 的位置）。

4. `msr psp, r0`：将 R0（现在指向硬件帧）写入 PSP（Process Stack Pointer）。这样异常返回时 CPU 会从 PSP 恢复硬件帧。

5. `mrs r0, control` + `orr r0, r0, #2` + `msr control, r0`：设置 CONTROL 寄存器的 SPSEL 位（bit 1）为 1，使 Thread 模式使用 PSP 而非 MSP。这是多任务的关键——每个任务有自己的 PSP。

6. `isb`：指令同步屏障，确保 CONTROL 寄存器的修改在下一条指令执行前生效。

7. `ldr lr, =0xFFFFFFFD`：加载特殊返回值到 LR。`0xFFFFFFFD` 表示返回 Thread 模式并使用 PSP（bit 0=1 表示 Handler 模式→Thread 模式，bit 2=1 表示使用 PSP，bit 3=1 表示 Thread 模式）。

8. `bx lr`：异常返回。CPU 读取 PSP 处的硬件帧，恢复 R0-R3、R12、LR、PC、xPSR，然后跳转到 PC（即 task_func）开始执行任务。

### 4.4 PendSV_Handler —— 上下文切换核心

PendSV（Pending SV）是 Cortex-M 专门为 RTOS 上下文切换设计的异常。它被配置为最低优先级，确保不会打断其他中断。每次 SysTick 中断或任务主动 yield 时，都会挂起 PendSV：

```c
__attribute__((naked)) void PendSV_Handler(void) {
    __asm volatile (
        // 读取当前 PSP（指向硬件保存帧）
        "mrs r0, psp                            \n"
        // 将 R4-R11 保存到当前任务栈
        "stmdb r0!, {r4-r11}                    \n"
        // 保存 LR 并 push r4 保持栈对齐
        "push {r4, lr}                          \n"
        // 调用 C 函数保存当前任务的 SP
        "ldr r1, =rtosk_task_save_stack_pointer \n"
        "blx r1                                 \n"
        // 调用调度器选择下一个任务
        "ldr r1, =rtosk_scheduler_select_next   \n"
        "blx r1                                 \n"
        // 获取下一个任务的 SP
        "ldr r1, =rtosk_task_get_stack_pointer  \n"
        "blx r1                                 \n"
        // 恢复 LR（r4 仅用于平衡 push）
        "pop {r4, lr}                           \n"
        // 从新任务栈恢复 R4-R11
        "ldmia r0!, {r4-r11}                    \n"
        // 设置 PSP 为新任务的硬件帧
        "msr psp, r0                            \n"
        // 异常返回，CPU 恢复新任务的硬件帧并继续执行
        "bx lr                                  \n"
    );
}
```

**逐行深度解析：**

1. `mrs r0, psp`：读取当前 PSP。进入 PendSV 时，硬件已自动将 R0-R3、R12、LR、PC、xPSR 保存到 PSP 指向的栈中。此时 PSP 指向硬件帧的起始。

2. `stmdb r0!, {r4-r11}`：**Store Multiple Decrement Before**，将 R4-R11 保存到 PSP 之上（地址递减方向）。`!` 表示回写，R0 更新为保存后的地址。现在 R0 指向 R4 的位置（软件帧起始），这就是要保存到 TCB 的 SP 值。

3. `push {r4, lr}`：将 R4 和 LR 压入当前栈（MSP）。因为接下来要调用 C 函数，LR（EXC_RETURN）会被覆盖，需要保存。push R4 是为了保持 8 字节栈对齐。

4. `ldr r1, =rtosk_task_save_stack_pointer` + `blx r1`：调用 C 函数 `rtosk_task_save_stack_pointer(r0)`，参数 R0 是当前任务的保存栈指针。该函数将其存入 `RTOSK_TASKS[current].sp`。

5. `ldr r1, =rtosk_scheduler_select_next` + `blx r1`：调用调度器，选择下一个要运行的任务。该函数会更新 `RTOSK_CURRENT_TASK`。

6. `ldr r1, =rtosk_task_get_stack_pointer` + `blx r1`：获取新选中任务的保存栈指针，返回值在 R0 中。

7. `pop {r4, lr}`：恢复之前保存的 LR（EXC_RETURN）和 R4。

8. `ldmia r0!, {r4-r11}`：从新任务的栈中恢复 R4-R11。R0 自增后指向硬件帧起始。

9. `msr psp, r0`：将 PSP 设置为新任务的硬件帧地址。

10. `bx lr`：异常返回。CPU 从新任务的 PSP 恢复 R0-R3/R12/LR/PC/xPSR，跳转到该任务上次被切换出时的 PC 继续执行。

**上下文切换时序图：**

```
SysTick IRQ ──► rtosk_kernel_tick()
                    │
                    ├─ RTOSK_SYSTICKS++
                    ├─ rtosk_task_update_blocked()  // 唤醒到期任务
                    └─ rtosk_port_yield()
                         │
                         ├─ SCB->ICSR = PENDSVSET  // 挂起 PendSV
                         └─ __DSB(); __ISB()
                              │
                    ┌─────────┘
                    │
              SysTick ISR 返回
                    │
              PendSV ISR 触发（最低优先级，最后执行）
                    │
              ├─ 保存当前任务 R4-R11 到其栈
              ├─ 保存当前任务 SP 到 TCB
              ├─ 调用调度器选择下一任务
              ├─ 从新任务 TCB 加载 SP
              ├─ 恢复新任务 R4-R11
              ├─ 设置 PSP
              └─ 异常返回到新任务
```

### 4.5 调度器实现 —— 优先级调度算法

调度器代码极其简洁，位于 `src/kernel/scheduler.c`：

```c
void rtosk_scheduler_select_next(void) {
    uint32_t task_count = rtosk_task_get_count();
    if(task_count == 0UL) {
        return;  // 没有任务，直接返回
    }

    uint32_t current = rtosk_task_get_current_index();
    if(current >= task_count) {
        current = 0UL;  // 当前是空闲任务，从 0 开始搜索
    }

    uint32_t best_index = RTOSK_IDLE_TASK_INDEX;
    uint32_t best_priority = 0UL;
    uint32_t found_ready = 0UL;

    // 遍历所有任务，寻找就绪状态中优先级最高的
    for(uint32_t offset = 0UL; offset < task_count; offset++) {
        uint32_t index = current + offset;
        if(index >= task_count) {
            index -= task_count;  // 环形回绕
        }

        rtosk_task_t * task = rtosk_task_get(index);
        if(task == 0 || task->state != RTOSK_TASK_READY) {
            continue;  // 跳过非就绪任务
        }

        if(found_ready == 0UL || task->priority > best_priority) {
            best_index = index;
            best_priority = task->priority;
            found_ready = 1UL;
        }
    }

    if(found_ready != 0UL) {
        rtosk_task_set_current_index(best_index);      // 选中最高优先级就绪任务
    } else if(rtosk_task_is_idle_ready() != 0UL) {
        rtosk_task_set_current_index(RTOSK_IDLE_TASK_INDEX);  // 没有就绪任务，运行空闲任务
    }
}
```

**算法解析：**

1. **环形遍历**：从当前任务索引开始，环形遍历所有任务。`index = current + offset`，当超出任务数时回绕。这实现了同优先级任务的轮转调度（round-robin）——如果两个任务优先级相同，先遍历到的会被选中，下次 SysTick 时从它开始遍历，另一个会被选中。

2. **优先级比较**：`task->priority > best_priority` 使用严格大于（`>`），意味着同优先级时，先找到的任务保持选中（因为后来的不会替换它）。结合环形遍历，实现了同优先级轮转。

3. **空闲任务降级**：如果没有任务处于 READY 状态，调度器选择空闲任务（`RTOSK_IDLE_TASK_INDEX`）。空闲任务执行 `wfi`（Wait For Interrupt）指令，降低功耗。

### 4.6 内核初始化与启动流程

```c
void rtosk_kernel_systick_init(void) {
    if(RTOSK_KERNEL_SYSTICK_INIT) return;
    rtosk_port_configure_exceptions();  // 配置 PendSV/SysTick 优先级
    SysTick->LOAD = (SystemCoreClock / 1000UL) - 1UL;  // 1ms tick
    SysTick->VAL = 0UL;
    SysTick->CTRL = SysTick_CTRL_CLKSOURCE_Msk |
                    SysTick_CTRL_TICKINT_Msk   |
                    SysTick_CTRL_ENABLE_Msk;
    RTOSK_KERNEL_SYSTICK_INIT = 1U;
}

void rtosk_kernel_start(void) {
    if(RTOSK_KERNEL_INIT) return;
    if(rtosk_task_get_count() == 0UL) {
        while(1) {}  // 没有任务，死循环
    }
    rtosk_port_configure_faults();  // 启用除零和对齐故障陷阱
    rtosk_task_init_idle();         // 初始化空闲任务
    RTOSK_KERNEL_INIT = 1U;
    rtosk_port_start_first_task();  // 触发 SVC，启动第一个任务
}
```

**启动流程：**

1. `rtosk_port_configure_exceptions()`：设置 PendSV 优先级为最低（0xFF），SysTick 优先级次低（0xFE）。这确保 PendSV 不会打断任何其他中断，上下文切换总是在所有中断处理完成后才发生。

2. SysTick 配置为 1ms 周期中断：`SystemCoreClock / 1000 - 1`。对于 216MHz 的 F756ZG，reload 值为 215999。

3. `rtosk_port_configure_faults()`：启用 `DIV_0_TRP`（除零陷阱）和 `UNALIGN_TRP`（对齐陷阱），帮助开发者在调试阶段尽早发现错误。

4. `rtosk_task_init_idle()`：构建空闲任务的栈帧。空闲任务函数简单执行 `wfi` 指令进入低功耗等待。

5. `rtosk_port_start_first_task()` 执行 `svc #0`，进入 SVC_Handler 启动第一个任务。

### 4.7 SysTick 中断处理

```c
void rtosk_kernel_tick(void) {
    RTOSK_SYSTICKS++;  // 全局 tick 计数器递增
    if(RTOSK_KERNEL_INIT == 0U) {
        return;  // 内核未启动时不处理
    }
    rtosk_task_update_blocked(RTOSK_SYSTICKS);  // 唤醒到期任务
    rtosk_port_yield();  // 请求上下文切换
}
```

`rtosk_task_update_blocked` 检查所有阻塞任务是否到期：

```c
uint32_t rtosk_task_update_blocked(uint32_t current_tick) {
    uint32_t woke_task = 0UL;
    for(uint32_t i = 0UL; i < RTOSK_TASK_COUNT; i++) {
        if(RTOSK_TASKS[i].state == RTOSK_TASK_BLOCKED) {
            // 使用有符号比较处理 tick 回绕
            if((int32_t)(current_tick - RTOSK_TASKS[i].wake_tick) >= 0) {
                RTOSK_TASKS[i].state = RTOSK_TASK_READY;
                woke_task = 1UL;
            }
        }
    }
    return woke_task;
}
```

**关键点 —— tick 回绕处理**：`(int32_t)(current_tick - wake_tick) >= 0` 使用有符号减法。当 tick 计数器从 `0xFFFFFFFF` 回绕到 `0` 时，减法结果仍正确。例如 `current_tick = 2`，`wake_tick = 0xFFFFFFFE`，差值为 `4`（有符号），仍判断为到期。

### 4.8 任务睡眠实现

```c
void rtosk_kernel_sleep_ms(uint32_t ms) {
    rtosk_kernel_enter_critical();
    uint32_t wake_tick = RTOSK_SYSTICKS + ms;
    rtosk_task_block_current_until(wake_tick);  // 设置阻塞状态和唤醒时间
    rtosk_kernel_exit_critical();
    rtosk_kernel_yield();  // 立即触发上下文切换
}
```

**解析**：

1. 进入临界区保护 wake_tick 计算和状态修改的原子性。
2. `wake_tick = RTOSK_SYSTICKS + ms`：计算绝对唤醒时间。
3. `rtosk_task_block_current_until` 将当前任务状态设为 `BLOCKED`，记录 `wake_tick`。
4. 退出临界区后调用 `rtosk_kernel_yield()` 触发 PendSV，让 CPU 切换到其他就绪任务。
5. 当 SysTick tick 达到 `wake_tick` 时，`rtosk_task_update_blocked` 会将任务状态恢复为 `READY`，调度器便可再次选中它。

### 4.9 临界区管理 —— 嵌套安全

```c
static uint32_t RTOSK_CRITICAL_NESTING = 0UL;
static uint32_t RTOSK_CRITICAL_PRIMASK = 0UL;

void rtosk_kernel_enter_critical(void) {
    uint32_t primask = rtosk_port_irq_save();  // 读取并禁用中断
    if(RTOSK_CRITICAL_NESTING == 0UL) {
        RTOSK_CRITICAL_PRIMASK = primask;  // 仅最外层保存原始状态
    }
    RTOSK_CRITICAL_NESTING++;
}

void rtosk_kernel_exit_critical(void) {
    if(RTOSK_CRITICAL_NESTING == 0UL) {
        return;  // 防止下溢
    }
    RTOSK_CRITICAL_NESTING--;
    if(RTOSK_CRITICAL_NESTING == 0UL) {
        rtosk_port_irq_restore(RTOSK_CRITICAL_PRIMASK);  // 仅最外层恢复中断
    }
}
```

**嵌套安全设计**：使用 `RTOSK_CRITICAL_NESTING` 计数器跟踪临界区嵌套深度。只有最外层（nesting 从 0→1）保存原始 PRIMASK，只有最外层退出（nesting 从 1→0）恢复中断。这允许在信号量、队列等同步原语内部安全地嵌套调用 `enter_critical` / `exit_critical`。

### 4.10 队列实现 —— 环形缓冲 + 阻塞接收

```c
void rtosk_queue_init(rtosk_queue_t * queue, void * buffer, uint32_t item_size, uint32_t capacity) {
    queue->buffer = (uint8_t *)buffer;
    queue->item_size = item_size;
    queue->capacity = capacity;
    queue->head = 0UL;  // 写入位置
    queue->tail = 0UL;  // 读取位置
    queue->count = 0UL;
    queue->waiting_task = RTOSK_QUEUE_NO_WAITER;  // 无等待者
}

uint32_t rtosk_queue_send(rtosk_queue_t * queue, const void * item) {
    rtosk_kernel_enter_critical();
    if(queue->count >= queue->capacity) {
        rtosk_kernel_exit_critical();
        return 0UL;  // 队列满，发送失败
    }
    // 写入数据到 head 位置
    uint8_t * dst = &queue->buffer[queue->head * queue->item_size];
    rtosk_queue_copy(dst, (const uint8_t *)item, queue->item_size);
    queue->head++;
    if(queue->head >= queue->capacity) {
        queue->head = 0UL;  // 环形回绕
    }
    queue->count++;
    // 如果有任务在等待队列数据，唤醒它
    if(queue->waiting_task != RTOSK_QUEUE_NO_WAITER) {
        rtosk_task_set_ready(queue->waiting_task);
        queue->waiting_task = RTOSK_QUEUE_NO_WAITER;
    }
    rtosk_kernel_exit_critical();
    rtosk_kernel_yield();  // 触发切换，让被唤醒的任务有机会运行
    return 1UL;
}

uint32_t rtosk_queue_receive(rtosk_queue_t * queue, void * item) {
    for(;;) {  // 无限循环直到获取到数据
        rtosk_kernel_enter_critical();
        if(queue->count > 0UL) {
            // 从 tail 位置读取数据
            uint8_t * src = &queue->buffer[queue->tail * queue->item_size];
            rtosk_queue_copy((uint8_t *)item, src, queue->item_size);
            queue->tail++;
            if(queue->tail >= queue->capacity) {
                queue->tail = 0UL;  // 环形回绕
            }
            queue->count--;
            rtosk_kernel_exit_critical();
            return 1UL;  // 成功接收
        }
        // 队列为空，阻塞当前任务
        queue->waiting_task = rtosk_task_get_current_index();
        rtosk_task_block_current_on_queue();
        rtosk_kernel_exit_critical();
        rtosk_kernel_yield();  // 切换到其他任务
        // 被唤醒后回到循环顶部重试
    }
}
```

**设计要点**：

- **环形缓冲**：使用 head（写指针）和 tail（读指针）实现 FIFO。当指针到达 capacity 时回绕到 0。
- **单等待者模型**：`waiting_task` 只记录一个等待者，简化了实现。如果有多个任务等待同一队列，只有最后一个注册的会被唤醒。
- **阻塞-唤醒机制**：接收时如果队列为空，任务被阻塞（`BLOCKED_ON_QUEUE`），当发送者写入数据后会唤醒等待者并 yield。
- **ISR 安全版本**：`rtosk_queue_send_from_isr` 不使用临界区（因为 ISR 中不会被抢占），直接操作队列。

### 4.11 互斥锁实现

```c
void rtosk_mutex_lock(rtosk_mutex_t * mutex) {
    for(;;) {
        rtosk_kernel_enter_critical();
        uint32_t current = rtosk_task_get_current_index();
        if(mutex->owner == RTOSK_MUTEX_NO_OWNER) {
            mutex->owner = current;  // 获取锁
            rtosk_kernel_exit_critical();
            return;
        }
        if(mutex->owner == current) {
            rtosk_kernel_exit_critical();
            return;  // 已持有锁，直接返回（不可重入但不会死锁）
        }
        // 锁被其他任务持有，阻塞等待
        mutex->waiting_task = current;
        rtosk_task_block_current_on_mutex();
        rtosk_kernel_exit_critical();
        rtosk_kernel_yield();  // 切换到其他任务
    }
}

void rtosk_mutex_unlock(rtosk_mutex_t * mutex) {
    rtosk_kernel_enter_critical();
    uint32_t current = rtosk_task_get_current_index();
    if(mutex->owner != current) {
        rtosk_kernel_exit_critical();
        return;  // 不是持有者，不能解锁
    }
    mutex->owner = RTOSK_MUTEX_NO_OWNER;
    // 唤醒等待者
    if(mutex->waiting_task != RTOSK_MUTEX_NO_WAITER) {
        rtosk_task_set_ready(mutex->waiting_task);
        mutex->waiting_task = RTOSK_MUTEX_NO_WAITER;
    }
    rtosk_kernel_exit_critical();
    rtosk_kernel_yield();  // 让被唤醒的等待者有机会获取锁
}
```

**注意**：此互斥锁尚未实现优先级继承（priority inheritance），这意味着可能出现优先级反转问题。作者在 README 中明确标注了这是已知限制和未来开发方向。

### 4.12 HardFault 诊断 —— 寄存器转储与原因解码

```c
__attribute__((naked)) void HardFault_Handler(void) {
    __asm volatile (
        "tst lr, #4      \n"  // 测试 EXC_RETURN bit 2
        "ite eq          \n"  // If-Then-Else 条件执行
        "mrseq r0, msp   \n"  // bit2=0: 使用 MSP
        "mrsne r0, psp   \n"  // bit2=1: 使用 PSP
        "mov r1, lr      \n"  // 第二参数 = EXC_RETURN
        "b rtosk_fault_hardfault_handler \n"  // 跳转到 C 处理函数
    );
}
```

HardFault_Handler 首先判断故障发生在哪种栈上（MSP 还是 PSP），然后将栈指针和 EXC_RETURN 传递给 C 函数 `rtosk_fault_hardfault_handler`：

```c
__attribute__((noreturn)) void rtosk_fault_hardfault_handler(uint32_t * stacked_registers, uint32_t exc_return) {
    __disable_irq();  // 禁用所有中断
    // 输出寄存器转储
    fault_write("========================================\r\n");
    fault_write("           TinyRTOS HARDFAULT           \r\n");
    fault_write("========================================\r\n");
    fault_write_reg("EXC_RETURN", exc_return);
    fault_write_reg("R0 ", stacked_registers[0]);
    fault_write_reg("R1 ", stacked_registers[1]);
    fault_write_reg("R2 ", stacked_registers[2]);
    fault_write_reg("R3 ", stacked_registers[3]);
    fault_write_reg("R12", stacked_registers[4]);
    fault_write_reg("LR ", stacked_registers[5]);
    fault_write_reg("PC ", stacked_registers[6]);   // 故障指令地址
    fault_write_reg("xPSR", stacked_registers[7]);

    uint32_t cfsr = SCB->CFSR;
    uint32_t hfsr = SCB->HFSR;
    // 解码 CFSR（可配置故障状态寄存器）
    fault_decode_cfsr(cfsr);
    // 解码 HFSR（硬故障状态寄存器）
    fault_decode_hfsr(hfsr);
    // 输出可能原因
    fault_print_fault_summary(cfsr, hfsr);

    fault_write("SYSTEM HALTED.\r\n");
    for(;;) {
        __asm volatile ("bkpt #0");  // 触发断点，方便调试器暂停
    }
}
```

`fault_decode_cfsr` 逐位解码故障标志，例如：
- `IACCVIOL`（bit 0）：指令访问违规
- `DACCVIOL`（bit 1）：数据访问违规
- `UNDEFINSTR`（bit 16）：未定义指令
- `INVSTATE`（bit 17）：无效 Thumb 状态
- `UNALIGNED`（bit 24）：未对齐访问
- `DIVBYZERO`（bit 25）：除零错误

`fault_print_fault_summary` 根据标志组合给出人类可读的可能原因，例如 "divide by zero." 或 "invalid exception return or corrupted stack frame."。

### 4.13 栈水位检测

```c
#define RTOSK_STACK_WATERMARK 0xDEADBEEFUL

// 初始化时用 DEADBEEF 填充整个栈
static void rtosk_task_watermark_stack(rtosk_task_t * task) {
    for(uint32_t i = 0UL; i < RTOSK_TASK_STACK_WORDS; i++) {
        task->stack[i] = RTOSK_STACK_WATERMARK;
    }
}

// 计算仍为 DEADBEEF 的连续字数（即未使用的栈空间）
static uint32_t rtosk_task_get_stack_free_words(rtosk_task_t * task) {
    uint32_t free_words = 0UL;
    while(free_words < RTOSK_TASK_STACK_WORDS && task->stack[free_words] == RTOSK_STACK_WATERMARK) {
        free_words++;
    }
    return free_words;
}
```

**原理**：任务创建时用 `0xDEADBEEF` 填充整个栈。任务运行时，栈向下生长会覆盖这些水印值。通过扫描仍为 `DEADBEEF` 的连续区域，可以测量栈的**最大使用深度**——这是一个保守估计，因为函数返回后栈空间被释放但水印不会被恢复。

OSI Shell 的 `tasks` 命令会调用 `rtosk_task_get_info` 获取每个任务的栈使用情况：

```c
uint32_t rtosk_task_get_info(uint32_t index, rtosk_task_info_t * info) {
    rtosk_task_t * task = &RTOSK_TASKS[index];
    uint32_t free_words = rtosk_task_get_stack_free_words(task);
    info->stack_free_words = free_words;
    info->stack_used_words = RTOSK_TASK_STACK_WORDS - free_words;
    // ...
    return 1UL;
}
```

### 4.14 示例应用 —— 双 LED 闪烁

```c
#include <tinyrtos/kernel/kernel.h>
#include <tinyrtos/osi/osi.h>
#include <tinyrtos/bsp/f756zg/led.h>

// Task0: 每 1 秒切换绿色 LED
static void led0_task(void) {
    while(1) {
        rtosk_bsp_f756zg_toggle_on_board_led(LED_0);
        rtosk_kernel_sleep_ms(1000UL);  // 阻塞 1 秒，允许其他任务运行
    }
}

// Task1: 忙等待切换蓝色 LED（故意不 yield，证明抢占有效）
static void led1_task(void) {
    while(1) {
        rtosk_bsp_f756zg_toggle_on_board_led(LED_1);
        // 从不 yield！如果 task0 仍然运行，证明抢占式调度有效
        for(volatile uint32_t i = 0UL; i < 3000000UL; i++) {}
    }
}

int main(void) {
    rtosk_bsp_f756zg_on_board_led_init();
    rtosk_osi_init(115200UL, 3UL);      // 初始化 OSI Shell（115200 波特率）
    rtosk_kernel_systick_init();         // 初始化 SysTick（1ms tick）
    rtosk_kernel_create_task(led0_task, 2UL, "led0");  // 优先级 2（高）
    rtosk_kernel_create_task(led1_task, 1UL, "led1");  // 优先级 1（低）
    rtosk_kernel_start();                // 启动内核（永不返回）
    for(;;) {}
    return 0;
}
```

**关键演示点**：

- `led0_task` 优先级为 2（高于 `led1_task` 的优先级 1），但它每秒会 sleep，所以 `led1_task` 有机会运行。
- `led1_task` 故意不 yield，执行忙等待循环。但由于 `led0_task` 优先级更高，当它从 sleep 中唤醒时，SysTick 中断会触发调度器抢占 `led1_task`，让 `led0_task` 执行。
- 如果两个 LED 都在闪烁，说明抢占式调度工作正常。

## 五、API 使用指南

### 内核 API

```c
#include <tinyrtos/kernel/kernel.h>

// 初始化 SysTick 定时器（1ms tick）
rtosk_kernel_systick_init();

// 创建任务（静态分配，最多 4 个）
rtosk_kernel_create_task(task_func, priority, "name");

// 启动内核（触发 SVC，永不返回）
rtosk_kernel_start();

// 当前任务睡眠 ms 毫秒（阻塞式）
rtosk_kernel_sleep_ms(1000UL);

// 主动让出 CPU（触发 PendSV）
rtosk_kernel_yield();

// 获取系统 tick 计数
uint32_t ticks = rtosk_kernel_get_ticks();
```

### 信号量 API

```c
#include <tinyrtos/kernel/semaphore.h>

rtosk_semaphore_t sem;
rtosk_semaphore_init(&sem, 0);     // 初始计数 0
rtosk_semaphore_take(&sem);        // 计数减 1（如果为 0 则阻塞）
rtosk_semaphore_give(&sem);        // 计数加 1（唤醒等待者）
rtosk_semaphore_give_from_isr(&sem); // ISR 安全版本
```

### 互斥锁 API

```c
#include <tinyrtos/kernel/mutex.h>

rtosk_mutex_t mutex;
rtosk_mutex_init(&mutex);
rtosk_mutex_lock(&mutex);
// ... 临界区 ...
rtosk_mutex_unlock(&mutex);
```

### 队列 API

```c
#include <tinyrtos/kernel/queue.h>

rtosk_queue_t queue;
uint32_t buffer[16];  // 4 个元素，每个 4 字节

rtosk_queue_init(&queue, buffer, sizeof(uint32_t), 4);

// 发送端
uint32_t data = 42;
rtosk_queue_send(&queue, &data);
// ISR 版本
rtosk_queue_send_from_isr(&queue, &data);

// 接收端（阻塞直到有数据）
uint32_t received;
rtosk_queue_receive(&queue, &received);
```

## 七、项目亮点与适用场景

### 技术亮点

1. **完整的从零实现**：不依赖任何第三方 RTOS 代码，从 PendSV 汇编到调度算法全部手写，是学习 RTOS 内部原理的绝佳教材。

2. **极简 footprint**：完整内核仅约 9KB Flash（`-Os`），适合资源受限的 Cortex-M 设备。

3. **静态分配一切**：任务栈、TCB、同步原语全部静态分配，零堆使用，消除内存碎片风险。

4. **精细的栈帧构建**：手动构造 Cortex-M 异常栈帧，精确模拟硬件中断保存行为，是理解 ARM 异常模型的极佳示例。

5. **完善的故障诊断**：HardFault 处理器自动转储所有寄存器，解码 CFSR/HFSR 故障标志，并给出可能原因，大幅缩短调试时间。

6. **栈水位检测**：使用 DEADBEEF 模式填充栈空间，运行时可查询每个任务的最大栈使用深度，帮助优化栈大小配置。

7. **OSI 运行时 Shell**：通过 UART 提供运行时诊断能力，无需调试器即可查看任务状态、栈使用、系统频率等信息。

### 适用场景

| 场景 | 说明 |
|------|------|
| RTOS 教学 | 理解抢占式调度、上下文切换、同步原语的实现原理 |
| Cortex-M 架构学习 | 深入理解 PSP/MSP、异常返回、NVIC 优先级等硬件特性 |
| 轻量嵌入式应用 | 资源受限场景下的简单多任务调度 |
| RTOS 二次开发 | 作为基础框架，添加 MPU 隔离、软件定时器等功能 |
| 代码审计参考 | 学习嵌入式 C 的安全编码实践（静态分配、临界区保护） |

### 已知限制

- 未实现优先级继承（可能发生优先级反转）
- 不支持动态任务创建/删除
- 无栈溢出陷阱（仅有水位检测）
- 无 MPU 内存隔离
- 不支持 tickless idle
- 无软件定时器和事件组
- 当前仅支持 STM32F7（Cortex-M7）

## 八、总结

tinyrtos 是一个难得的 RTOS 教学与实践项目。它没有 FreeRTOS 的庞大生态，也没有 Zephyr 的丰富中间件，但它用最少的代码、最清晰的结构，完整展示了 RTOS 内核的核心机制：从 PendSV 上下文切换到优先级调度，从栈帧构建到故障诊断。

对于嵌入式开发者来说，阅读 tinyrtos 源码的价值在于：你将真正理解当 `rtosk_kernel_sleep_ms(1000)` 被调用时，从 SysTick 中断到 PendSV 上下文切换再到任务唤醒的完整链路。这种底层理解，是从"会用 RTOS"到"理解 RTOS"的关键一步。

项目作者 Ian Wilkey 计划在未来添加优先级继承、事件组、软件定时器、MPU 隔离等功能，并在朝着"真正实用的嵌入式 RTOS"方向演进。对于关注 RTOS 内部原理的开发者，这是一个值得持续关注的项目。
