## 一、项目简介

Amast（A Minimalist Asynchronous Toolkit，极简异步工具包）是一个用 C99 编写的嵌入式异步编程框架。它把 **Actor 模型**与 **层次状态机（HSM）** 结合起来，为资源受限的 MCU 提供一套构建实时、事件驱动系统的方法论与代码库。

与传统的「裸机 while(1) + 标志位」或「RTOS + 互斥锁」两条路不同，Amast 走的是第三条路：**Active Object（主动对象）**。每个对象拥有自己的事件队列和状态机，对象之间只通过异步事件通信，天然避免了共享数据竞争，又比传统 RTOS 任务更轻量。

### 核心特性一览

| 特性 | 说明 |
|------|------|
| 有限状态机（FSM） | 平坦状态机，编译后仅约 0.88KB，适合极小场景 |
| 层次状态机（HSM） | 支持状态嵌套、行为继承、子机（submachine），约 2.66KB |
| Active Object | 抢占式 / 协作式两种调度，事件队列 + 发布订阅 |
| async/await | 基于 coroutine 的协作式异步，原生 C99 实现 |
| 事件系统 | 发布订阅、引用计数、内存池分配，ISR 安全 |
| 定时器 | 一次性 / 周期性，支持 tick 域 |
| 内存管理 | onesize 单尺寸分配器，无碎片、无临界区 |
| 平台抽象（PAL） | POSIX / libuv 移植，可运行于 x86、ARM MCU |
| 编译体积 | FSM 2.4KB / HSM 4.0KB / AO 10.6KB（x86 -Os -flto） |

## 四、核心代码深度分析

这是本文最重要的部分。我们从源码仓库逐段提取 Amast 的真实实现，并对每段代码进行逐行/逐段解析，覆盖初始化、状态分发算法、LCA 转换算法与 Active Object 事件循环。

### 4.1 FSM：状态分发算法（`libs/fsm/fsm.c`）

先看最基础的有限状态机。整个 FSM 的「心脏」是 `fsm_dispatch`——它决定了事件如何被派发给状态、状态切换时如何触发 entry/exit。

```c
static enum am_rc fsm_dispatch(
    struct am_fsm* fsm, const struct am_event* event
) {
    AM_ASSERT(fsm->state);
    am_fsm_state_fn src = fsm->state;
    enum am_rc rc = fsm->state(fsm, event);
    if ((AM_RC_HANDLED == rc) || (AM_RC_CORO_BUSY == rc) ||
        (AM_RC_CORO_DONE == rc)) {
        AM_ASSERT(fsm->state == src);
        return rc;
    }
    AM_ASSERT((AM_RC_TRAN == rc) || (AM_RC_TRAN_REDISPATCH == rc));
    am_fsm_state_fn dst = fsm->state;
    fsm->state = src;
    fsm_exit(fsm);
    fsm_enter(fsm, dst);
    return rc;
}
```

**逐段解析：**

- `am_fsm_state_fn src = fsm->state;` —— 先保存「源状态」。因为状态处理函数 `fsm->state(fsm, event)` 在内部可能通过 `AM_FSM_TRAN(target)` 修改 `fsm->state`，所以必须提前快照，否则切换后就找不到「从哪来」。
- `enum am_rc rc = fsm->state(fsm, event);` —— 真正调用当前状态的处理函数。返回值 `rc` 是一个枚举，表达了「事件被怎么处理了」：`HANDLED`（处理了，不切换）、`TRAN`（请求切换）、`CORO_BUSY/DONE`（协程相关）。
- `if ((AM_RC_HANDLED == rc) || ...)` —— 若事件被处理且没有触发切换，断言 `fsm->state == src`（即处理函数没乱改状态指针），直接返回。这是 Amast 的**防御式编程**：用断言保证状态机不变量。
- `am_fsm_state_fn dst = fsm->state;` —— 走到这里说明触发了切换，此时 `fsm->state` 已被 `AM_FSM_TRAN()` 改成了目标状态，取出作为 `dst`。
- `fsm->state = src;` —— **关键技巧**：临时把状态指针还原回源状态。因为接下来要调 `fsm_exit`，退出事件 `AM_EVT_EXIT` 必须发给「当前（源）状态」，而不是目标状态。
- `fsm_exit(fsm);` —— 触发源状态的退出钩子（发送 `AM_EVT_EXIT`）。
- `fsm_enter(fsm, dst);` —— 进入目标状态（发送 `AM_EVT_ENTRY`），并把 `fsm->state` 设为 `dst`。

这就是经典的 **entry/exit 状态嵌套语义**：每次切换都保证「先退后进」，让状态可以安全地申请/释放资源。

再看 entry/exit 的具体实现：

```c
static void fsm_enter(struct am_fsm* fsm, const am_fsm_state_fn state) {
    fsm->state = state;
    struct am_event entry = {.id = AM_EVT_ENTRY};
    enum am_rc rc = fsm->state(fsm, &entry);
    AM_ASSERT(AM_RC_HANDLED == rc);
}

static void fsm_exit(struct am_fsm* fsm) {
    struct am_event exit = {.id = AM_EVT_EXIT};
    enum am_rc rc = fsm->state(fsm, &exit);
    AM_ASSERT(AM_RC_HANDLED == rc);
}
```

**解析：** 这两个函数极其简洁——构造一个栈上的 `am_event`（只填 `id`，无载荷），调用当前状态处理函数，并断言返回 `HANDLED`。注意 `entry/exit` 事件**不允许触发转换**（`AM_FSM_TRAN` 文档明确禁止在 entry/exit 中使用），所以这里强制断言 `AM_RC_HANDLED`，从机制上杜绝了错误用法。

而对外暴露的 `am_fsm_dispatch` 在内部 `fsm_dispatch` 之上加了一层安全封装：

```c
void am_fsm_dispatch(struct am_fsm* fsm, const struct am_event* event) {
    AM_ASSERT(fsm);
    AM_ASSERT(fsm->state);
    AM_ASSERT(fsm->start_called);
    AM_ASSERT(!fsm->dispatch_in_progress);
    AM_ASSERT(event);
    AM_ASSERT(AM_EVENT_HAS_USER_ID(event));
    fsm->dispatch_in_progress = true;
    const int id = event->id;
    enum am_rc rc = fsm_dispatch(fsm, event);
    if (AM_RC_TRAN_REDISPATCH == rc) {
        rc = fsm_dispatch(fsm, event);
        AM_ASSERT(AM_RC_TRAN_REDISPATCH != rc);
    }
    fsm->dispatch_in_progress = false;
    AM_ASSERT(id == event->id);
}
```

**解析：**

- 五个 `AM_ASSERT` 构成前置条件检查：非空、已初始化、已 start、**没有重入**、事件 ID 合法。其中 `dispatch_in_progress` 标志位用来捕获「在状态处理函数里又 dispatch 自己」的重入错误。
- `fsm->dispatch_in_progress = true;` —— 置位重入保护。
- `const int id = event->id;` + 末尾 `AM_ASSERT(id == event->id);` —— 这是个巧妙的**事件悬挂/损坏检测**：如果用户在状态处理中错误地释放了栈外事件（如内存池事件忘了加引用计数），事件内存可能被复用导致 `id` 变化。注释里还贴心地给出了典型错误场景。
- `if (AM_RC_TRAN_REDISPATCH == rc)` —— 支持「转换并重派发」语义：切换到新状态后，把同一个事件再派发一次给新状态。这用于「事件在旧状态触发切换后，新状态也想立刻看到这个事件」的场景，且**最多重派发一次**（内部再次断言不再返回 REDISPATCH）。

### 4.2 FSM 状态转换宏（`libs/fsm/fsm.h`）

理解了 dispatch，再看用户写状态机时用到的两个核心宏——它们是 inline 函数：

```c
static inline enum am_rc am_fsm_handled(struct am_fsm* fsm) {
    (void)fsm;
    return AM_RC_HANDLED;
}

static inline enum am_rc am_fsm_tran(struct am_fsm* fsm, am_fsm_state_fn fn) {
    fsm->state = fn;
    return AM_RC_TRAN;
}
```

**解析：**

- `am_fsm_handled` —— 直接返回 `AM_RC_HANDLED`，告诉框架「事件我处理了，别再做什么」。`(void)fsm;` 消除未使用警告。
- `am_fsm_tran` —— **就地修改 `fsm->state` 为目标函数 `fn`**，然后返回 `AM_RC_TRAN`。这正是 4.1 中 `fsm_dispatch` 能在调用后读到新状态的原理——转换的「目标」是通过副作用写进 `fsm->state` 的，而非通过返回值。这种设计让 `AM_FSM_TRAN(target)` 用起来像语句，非常自然。

### 4.3 一个完整 FSM 示例（来自项目 README）

下面是官方给出的最小 FSM，两状态 A/B 互切：

```c
#define EVT_A AM_EVT_USER
#define EVT_B (AM_EVT_USER + 1)

struct app {
    struct am_fsm fsm;   /* 必须是第一个成员 */
} app;

static enum am_rc state_b(struct app *me, const struct am_event *event);

static enum am_rc state_a(struct app *me, const struct am_event *event) {
    switch (event->id) {
    case AM_EVT_ENTRY:
        am_printf("state_a entry\n");
        return AM_FSM_HANDLED();
    case AM_EVT_EXIT:
        am_printf("state_a exit\n");
        return AM_FSM_HANDLED();
    case EVT_B:
        return AM_FSM_TRAN(state_b);
    }
    return AM_FSM_HANDLED();
}

static enum am_rc state_b(struct app *me, const struct am_event *event) {
    switch (event->id) {
    case AM_EVT_ENTRY:
        am_printf("state_b entry\n");
        return AM_FSM_HANDLED();
    case AM_EVT_EXIT:
        am_printf("state_b exit\n");
        return AM_FSM_HANDLED();
    case EVT_A:
        return AM_FSM_TRAN(state_a);
    }
    return AM_FSM_HANDLED();
}

static enum am_rc init(struct app *me, const struct am_event *event) {
    return AM_FSM_TRAN(state_a);
}

int main(void) {
    am_fsm_ctor(&app.fsm, AM_FSM_STATE_CTOR(init));
    am_fsm_init(&app.fsm, /*init_event=*/NULL);
    am_fsm_dispatch(&app.fsm, &(struct am_event){.id = EVT_B});
    am_fsm_dispatch(&app.fsm, &(struct am_event){.id = EVT_A});
    return 0;
}
```

**逐段解析：**

- `struct app { struct am_fsm fsm; }` —— **必须把 `fsm` 放第一个成员**。这样 `struct app*` 和 `struct am_fsm*` 内存布局兼容，框架内部用 `am_fsm*` 操作时，可直接通过 C 的指针转型安全地「向下取回」`struct app*`，访问用户数据。这是 C 面向对象编程的经典手法。
- `AM_EVT_USER` —— 用户事件 ID 的起点，框架保留 `0..AM_EVT_USER-1` 给内部事件（ENTRY/EXIT/INIT/EMPTY）。
- 每个状态函数都是 `switch(event->id)`：处理 `ENTRY`/`EXIT` 做资源管理，处理业务事件返回 `AM_FSM_TRAN` 切换。
- `init` 状态只做一件事：返回 `AM_FSM_TRAN(state_a)`，即初始转换到 A。
- `main` 里：ctor 构造 → init 注册初始状态 → 第一次 dispatch EVT_B 触发 A→B → 第二次 dispatch EVT_A 触发 B→A。输出依次为 `state_a entry / state_a exit / state_b entry / state_b exit / state_a entry`，完美对应「先进后出再进」的语义。

### 4.4 HSM：层次状态机的核心算法（`libs/hsm/hsm.c`）

HSM 是 Amast 技术含量最高的部分。层次状态机的难点在于**状态转换时如何正确地退出/进入一串嵌套状态**，核心是计算 **LCA（Least Common Ancestor，最近公共祖先）**。

先看事件分发的「向上传播」逻辑：

```c
static enum am_rc hsm_dispatch(
    struct am_hsm* hsm, const struct am_event* event
) {
    struct am_hsm_state src = {.fn = NULL, .instance = 0};
    struct am_hsm_state state = hsm_get_state(hsm);
    enum am_rc rc = AM_RC_HANDLED;
    int cnt = AM_HSM_HIERARCHY_DEPTH_MAX;
    do {
        src = hsm_get_state(hsm);
        hsm_set_active_state(hsm, state);
        rc = src.fn(hsm, event);
        --cnt;
        AM_ASSERT(cnt);
    } while (AM_RC_SUPER == rc);
    bool tran = (AM_RC_TRAN == rc) || (AM_RC_TRAN_REDISPATCH == rc);
    if (!tran) {
        AM_ASSERT(
            (AM_RC_HANDLED == rc) || (AM_RC_CORO_BUSY == rc) ||
            (AM_RC_CORO_DONE == rc) || (AM_RC_SUPER == rc)
        );
        hsm_set_state(hsm, state);
        return rc;
    }
    struct am_hsm_state dst = hsm_get_state(hsm);
    AM_ASSERT(dst.fn != am_hsm_top);
    hsm_set_state(hsm, state);
    hsm_transition(hsm, src, dst);
    return rc;
}
```

**逐段解析：**

- `struct am_hsm_state state = hsm_get_state(hsm);` —— 保存当前活动叶子状态。
- `do { ... } while (AM_RC_SUPER == rc);` —— **行为继承的核心循环**：从最内层子状态开始处理事件，如果返回 `AM_RC_SUPER`（表示「我没处理，交给父状态」），就把事件**向上传播给超状态**，直到某个状态处理了它（`HANDLED`）或触发转换（`TRAN`）。这就是层次状态机的「行为继承」——子状态没处理的逻辑自动落到父状态。
- `hsm_set_active_state(hsm, state);` —— 每轮恢复活动状态指针，因为 `AM_HSM_SUPER(parent)` 宏会改写 `hsm->state_fn`。
- `--cnt; AM_ASSERT(cnt);` —— 防御性深度计数，避免层次超过 `AM_HSM_HIERARCHY_DEPTH_MAX`（默认 16）造成死循环。
- 若触发 `TRAN`：取出目标 `dst`，恢复到源状态，调用 `hsm_transition(hsm, src, dst)` 执行真正的层次进出。

接下来是整个框架最精妙的 **LCA 转换算法** `hsm_transition`：

```c
static void hsm_transition(
    struct am_hsm* hsm, struct am_hsm_state src, struct am_hsm_state dst
) {
    if (!am_hsm_state_is_eq(hsm, src)) {
        hsm_exit(hsm, /*until=*/src);
        hsm_set_state(hsm, src);
    }
    struct am_hsm_path path;
    if ((src.fn == dst.fn) && (src.instance == dst.instance)) {
        /* 转换到自身：先退出再进入 */
        path.state[0] = dst;
        path.len = 1;
        hsm_exit_state(hsm);
        hsm_enter_and_init(hsm, &path);
        return;
    }
    struct am_hsm_state until = am_hsm_state_make(am_hsm_top);
    hsm_build(hsm, &path, /*from=*/&dst, &until, /*till=*/&src);
    const struct am_hsm_state* end = &path.state[path.len - 1];
    if ((end->fn == src.fn) && (end->instance == src.instance)) {
        /* src 就是 LCA：只需从 LCA 进入到 dst */
        --path.len;
        hsm_enter_and_init(hsm, &path);
        return;
    }
    /*
     * 从 src 逐层退出到 am_hsm_top，沿途寻找 LCA。
     * 找到后不再退出 LCA，进入其所有子状态直到 dst。
     */
    int cnt = AM_HSM_HIERARCHY_DEPTH_MAX;
    while (hsm->state_fn != am_hsm_top) {
        --cnt;
        AM_ASSERT(cnt);
        if (hsm->hierarchy_level <= path.len) {
            int i = path.len - hsm->hierarchy_level;
            if (am_hsm_state_is_eq(hsm, path.state[i])) {
                /* LCA 找到，且不是 top */
                path.len = i;
                hsm_enter_and_init(hsm, &path);
                return;
            }
        }
        hsm_exit_state(hsm);
    }
    /* LCA 是 am_hsm_top() */
    hsm_enter_and_init(hsm, &path);
}
```

**逐段解析：**

这是状态机理论里 UML HSM 转换的实现。逻辑分四种情况：

1. **对齐源状态**：`if (!am_hsm_state_is_eq(hsm, src))` —— 由于事件向上传播时 `hsm->state_fn` 可能停在某个中间超态，先退出到真正的 `src`（触发转换的那个状态）。
2. **自转换**：`src == dst` 时，直接退出当前状态再重新进入，模拟 UML 的「external transition to self」。
3. **src 即 LCA**：`hsm_build` 从 `dst` 向上构建祖先链直到遇到 `src`。若链顶就是 `src`，说明 `src` 是 `dst` 的祖先（LCA），那只需「从 src 往下进入到 dst」，不退出 src。`--path.len` 把 src 自身从路径里剔除（因为它不退不进）。
4. **一般情况——边退出边找 LCA**：`while (hsm->state_fn != am_hsm_top)` 从 src 逐层退出，每退一层就用 `hierarchy_level`（预计算的层次深度）去 `path` 数组里对位比较——若当前退出到的状态正好在 dst 的祖先链上，它就是 LCA。找到后 `path.len = i` 截断路径，`hsm_enter_and_init` 从 LCA 往下进入。若一路退到 top 都没找到，则 LCA 就是 top。

> 这里的 `hierarchy_level` 是个性能优化：它缓存了每个状态的层次深度，使得 LCA 查找时能用 O(1) 的层级算术直接定位候选位置，而不必每次线性扫描整条路径。注释里的 `int i = path.len - hsm->hierarchy_level;` 就是这个对位计算。

接着看 `hsm_enter_and_init`——进入目标并递归处理初始转换：

```c
static void hsm_enter_and_init(struct am_hsm* hsm, struct am_hsm_path* path) {
    hsm_enter(hsm, path);
    hsm_set_state(hsm, path->state[0]);
    enum am_rc rc = AM_RC_OK;
    struct am_event init = {.id = AM_EVT_INIT};
    while ((rc = hsm->state_fn(hsm, &init)) == AM_RC_TRAN) {
        struct am_hsm_state until = path->state[0];
        struct am_hsm_state from = hsm_get_state(hsm);
        hsm_build(hsm, path, /*from=*/&from, &until, /*till=*/NULL);
        hsm_enter(hsm, path);
        hsm_set_state(hsm, path->state[0]);
    }
    AM_ASSERT((AM_RC_HANDLED == rc) || (AM_RC_SUPER == rc));
    hsm_set_state(hsm, path->state[0]);
}
```

**解析：**

- `hsm_enter(hsm, path)` —— 按 path 从最外层（LCA 子态）到最内层（dst）依次发 `ENTRY` 事件。
- `while (... == AM_RC_TRAN)` —— **递归初始转换**：进入 dst 后给它发 `INIT` 事件，若 dst 用 `AM_HSM_TRAN` 指定了更深的初始子状态，就再 build 一条新路径继续进入，直到某层不再要求初始转换。这正是 UML HSM「进入复合状态自动进入其默认子状态」的语义。

### 4.5 HSM 状态声明宏（`libs/hsm/hsm.h`）

用户写 HSM 时用到的几个关键 inline，理解它们就理解了「行为继承」的写法：

```c
static inline enum am_rc am_hsm_tran(struct am_hsm* hsm, am_hsm_state_fn fn) {
    hsm->state_fn = fn;
    hsm->state_instance = hsm->instance = 0;
    return AM_RC_TRAN;
}

static inline enum am_rc am_hsm_super(struct am_hsm* hsm, am_hsm_state_fn fn) {
    hsm->state_fn = fn;
    hsm->state_instance = hsm->instance = 0;
    return AM_RC_SUPER;
}
```

**解析：**

- `am_hsm_tran` —— 和 FSM 一样，靠副作用写目标到 `hsm->state_fn`，返回 `TRAN`。额外清零 `instance`（子机实例）。
- `am_hsm_super` —— **HSM 独有**：返回 `AM_RC_SUPER`，告诉框架「我没处理这个事件，请把它传给我的超状态 `fn`」。框架的 `hsm_dispatch` 循环正是靠这个返回值一层层向上走。注意它同样把 `hsm->state_fn` 改成了父状态——这就是 `hsm_dispatch` 里每轮要 `hsm_set_active_state(hsm, state)` 恢复的原因。

### 4.6 Active Object：事件队列与发布订阅（`libs/ao/ao.h`）

Active Object 把 HSM 包成「带事件队列的并发单元」。先看它的核心数据结构：

```c
typedef void (*am_ao_fn)(void* ctx, const struct am_event* event);

struct am_ao {
    am_ao_fn user_init_handler;        /* 用户 init 事件处理 */
    am_ao_fn user_event_handler;       /* 用户事件处理（通常是 hsm_dispatch） */
    void* ctx;                         /* 事件处理上下文（指向用户对象） */
    const char* name;                  /* AO 人类可读名 */
    struct am_event_queue event_queue; /* 事件队列 */
    int last_event;                    /* 最后处理的事件 */
    int task_id;                       /* 任务句柄 */
    struct am_ao_prio prio;            /* AO 优先级 */
    const struct am_event* init_event; /* 用户 AO init 事件 */
    bool init_called;
    bool running;
};
```

**解析：** AO 的精妙在于 `user_event_handler` —— 它是个函数指针，通常被设成 `am_hsm_dispatch`。这意味着 **AO 不绑定具体状态机实现**：你可以把 HSM 包进 AO，也可以把 FSM 包进 AO，甚至包自定义处理器。`ctx` 指向用户对象（含 `am_hsm`/`am_fsm` 成员），调度时框架调用 `user_event_handler(ctx, event)`，等于 `am_hsm_dispatch(my_obj, event)`。这是策略模式的 C 实现。

发布/订阅 API 的设计体现了「ISR 安全」与「事件所有权」两大考量：

```c
bool am_ao_publish_x(const struct am_event* event, int margin);
void am_ao_publish(const struct am_event* event);
bool am_ao_post_fifo_x(struct am_ao* ao, const struct am_event* event, int margin);
void am_ao_post_lifo(struct am_ao* ao, const struct am_event* event);
```

**解析：**

- `publish` 系列把事件投递给**所有订阅了该事件 ID 的 AO**；`post` 系列直接投给**指定 AO**。
- `_x` 后缀版本带 `margin` 参数：保证投递后队列仍留 `margin` 个空位，若放不下则**优雅返回 false**；不带 `_x` 的版本放不下直接 assert 崩溃。这给开发者两种选择——容错降级 or 严格不可丢。
- `fifo`（队尾）vs `lifo`（队头）：lifo 用于「紧急事件插队」，比如把高优先级中断事件塞到队列最前，下次调度立即处理。
- 文档反复强调「fast, thread safe and usable from ISR」——这些函数内部用临界区（`crit_enter/exit`）保护队列操作，因此中断里可以直接 publish/post。

### 4.7 Active Object 完整运行示例（来自项目 README）

下面这个例子展示了 AO 的完整生命周期：构造、定时器、发布订阅、ticker 喂入线程。这是理解 Amast 运行时的关键：

```c
struct app {
    struct am_hsm hsm;
    struct am_ao ao;
    struct am_timer *timer;
    struct am_timer_event timeout;
    int ticks;
};

static enum am_rc app_state_b(struct app *me, const struct am_event *event) {
    switch (event->id) {
    case AM_EVT_ENTRY:
        am_printf("state B\n");
        am_timer_arm(me->timer, &me->timeout.event, me->ticks, /*interval=*/0);
        return AM_HSM_HANDLED();
    case AM_EVT_EXIT:
        am_timer_disarm(me->timer, &me->timeout.event);
        return AM_HSM_HANDLED();
    case APP_EVT_SWITCH_MODE:
        return AM_HSM_TRAN(app_state_a);
    case APP_EVT_TIMER:
        am_printf("timer\n");
        am_timer_arm(me->timer, &me->timeout.event, me->ticks, /*interval=*/0);
        return AM_HSM_HANDLED();
    }
    return AM_HSM_SUPER(am_hsm_top);
}
```

**解析（state_b）：**

- `AM_EVT_ENTRY` 里 `am_timer_arm(..., ticks, /*interval=*/0)` —— 进入 B 就启动一次性定时器，`interval=0` 表示不周期。`me->ticks` 由 `am_time_get_tick_from_ms(..., 1000)` 算出，即 1000ms。
- `AM_EVT_EXIT` 里 `am_timer_disarm` —— 退出 B 时取消定时器，**保证不会在离开状态后还收到过期定时器事件**。这是状态机资源管理的黄金法则：entry 申请，exit 释放。
- `APP_EVT_TIMER` —— 定时器到期事件，处理后再 arm 一次，形成「软件周期定时」效果（用一次性定时器 + 手动重 arm 模拟周期）。

构造与启动部分：

```c
static void app_ctor(struct app *me, struct am_timer *timer) {
    memset(me, 0, sizeof(*me));
    am_ao_ctor(&me->ao, (am_ao_fn)am_hsm_init, (am_ao_fn)am_hsm_dispatch, me);
    am_hsm_ctor(&me->hsm, AM_HSM_STATE_CTOR(app_init));
    me->timer = timer;
    me->ticks = am_time_get_tick_from_ms(AM_PAL_TICK_DOMAIN_DEFAULT, 1000);
}
```

**解析：** `am_ao_ctor(&me->ao, init_fn, dispatch_fn, me)` —— 把 `am_hsm_init` 和 `am_hsm_dispatch` 作为 AO 的 init/dispatch 处理器注入，`me` 作为上下文。这一行就是把 HSM「装配」进 AO 的关键——AO 调度时实际跑的就是 HSM。

ticker 线程是整个运行时的「心跳」：

```c
static void ticker_task(void *param) {
    struct am_timer *timer = param;
    am_task_startup_gate_wait();
    while (am_ao_get_cnt() > 0) {
        am_sleep_ticks(AM_TICK_DOMAIN_DEFAULT, /*ticks=*/1);
        am_timer_tick_iterator_init(timer);
        struct am_timer_event* fired = NULL;
        while ((fired = am_timer_tick_iterator_next(timer)) != NULL) {
            void *owner = AM_CAST(struct am_timer_event_x *, fired)->ctx;
            if (owner) {
                am_ao_post_fifo(owner, &fired->event);
            } else {
                am_ao_publish(&fired->event);
            }
        }
    }
}
```

**解析：**

- `am_task_startup_gate_wait()` —— 等待所有任务同步启动，避免启动竞态。
- `while (am_ao_get_cnt() > 0)` —— 只要有存活的 AO 就一直跑，全退出则结束。
- `am_sleep_ticks(..., 1)` —— 每次睡 1 tick，构成系统节拍。
- 内层 `while` 用**迭代器**遍历本次 tick 到期的所有定时器：有 `owner` 的（定向定时器）用 `post_fifo` 投给该 AO；无 owner 的（广播定时器）用 `publish` 发给所有订阅者。这个 owner/non-owner 二分设计统一了「单播定时」和「广播定时」两种用法。

main 函数完成运行时初始化与启动：

```c
int main(void) {
    am_pal_ctor(/*arg=*/NULL);
    struct am_timer timer;
    am_timer_ctor(&timer);
    struct am_event_alloc alloc;
    am_event_alloc_init(&alloc);
    am_event_alloc_add_pool(&alloc, m_event_pool, sizeof(m_event_pool),
                            sizeof(m_event_pool[0]), AM_ALIGNOF(am_event_t));
    struct am_event_subscribe_list pubsub_list[APP_EVT_PUB_MAX];
    am_event_async_init(pubsub_list, AM_COUNTOF(pubsub_list), &alloc);
    struct am_ao_state_cfg cfg = {
        .crit_enter = am_crit_enter, .crit_exit = am_crit_exit, .alloc = &alloc
    };
    am_ao_state_ctor(&cfg);
    struct app m;
    app_ctor(&m, &timer);
    const struct am_event *m_queue[2];
    am_ao_start(&m.ao, (struct am_ao_prio){.ao = AM_AO_PRIO_MAX, .task = AM_AO_PRIO_MAX},
                m_queue, AM_COUNTOF(m_queue), NULL, 0, "app", NULL);
    am_task_create("ticker", AM_AO_PRIO_MIN, NULL, 0, ticker_task, 0, &timer);
    am_task_create("input", AM_AO_PRIO_MIN, NULL, 0, input_task, 0, &m);
    while (am_ao_get_cnt() > 0) {
        am_ao_run_all();
    }
    am_ao_state_dtor();
    am_pal_dtor();
    return 0;
}
```

**解析：** 初始化顺序很讲究——PAL → 定时器 → **事件内存池**（`am_event_alloc_add_pool`，所有动态事件从这里分配）→ 发布订阅链 → AO 全局状态（注入临界区回调与分配器）→ 构造用户 AO → 启动 AO（指定优先级、事件队列 `m_queue`）→ 创建 ticker/input 辅助任务 → 主循环 `am_ao_run_all()` 轮询所有 AO。其中 `m_queue[2]` 说明这个 AO 的事件队列只有 2 格——Amast 鼓励极小队列，因为事件处理是即时的。

```c
/* 1. 定义事件 ID（从 AM_EVT_USER 起） */
enum {
    APP_EVT_SENSOR_READY = AM_EVT_USER,
    APP_EVT_ALARM,
    APP_EVT_MAX
};

/* 2. 带载荷的事件结构 */
struct sensor_event {
    struct am_event event;   /* 必须是第一个成员 */

    float humidity;
};

/* 3. 静态事件直接 dispatch（无动态分配） */

am_hsm_dispatch(&my_hsm, &evt);

/* 4. 动态事件从池分配 + 引用计数 + 发布 */
struct sensor_event *se = (struct sensor_event *)
    am_event_allocate(&alloc, APP_EVT_SENSOR_READY, sizeof(*se));
se->temperature = 36.5f;
se->humidity    = 55.0f;
am_event_inc_ref_cnt(&se->event);   /* 发布前加引用 */
am_ao_publish(&se->event);          /* 投递给所有订阅者 */
am_event_dec_ref_cnt(&se->event);   /* 本地引用释放 */
```

| `AM_AO_PRIO_MAX` | AO 优先级上限 | `AM_AO_NUM_MAX - 1` |
| 调度模式 | `amast_preemptive.c` / `amast_cooperative.c` | 裸机协作 / RTOS 抢占 |

### 6.3 烧录到 MCU（通用流程）

Amast 不绑定特定烧录工具，移植到 MCU 时的通用步骤：

1. **实现 PAL 接口**：在你的 MCU 工程里实现 `am_task_create`、`am_sleep_ticks`、`am_crit_enter/exit`、`am_time_get_tick_from_ms` 等原语（参考 `amast_posix.c`）。
2. **加入 4 个核心文件**：`amast.h`、`amast_config.h`、`amast.c`、`amast_preemptive.c`（或 cooperative）。
3. **编译进固件**：用你熟悉的 IDE/Makefile/CMake 把上述文件编入工程。
4. **烧录**：用对应工具下载——
   - STM32：`openocd -f interface/stlink.cfg -f target/stm32f4x.cfg -c "program build/fw.elf reset exit"`
   - RP2040：`openocd -f interface/cmsis-dap.cfg -f target/rp2040.cfg -c "program fw.elf verify reset exit"`
   - ESP32：`idf.py flash`（需把 PAL 适配进 ESP-IDF 任务层）

### 6.4 x86 验证

最稳妥的入门方式是先在 Linux/WSL 上跑通，再移植到 MCU：

```bash
cd amast
pixi run all          # 编译并运行所有示例与测试
# 单独跑某个示例
./build/apps/examples/dpp/dpp    # 哲学家就餐（多 AO 并发）
./build/apps/examples/hsm/hsm    # HSM 层次演示
```

## 七、项目亮点与适用场景

### 7.1 技术亮点

| 亮点 | 技术内涵 |
|------|---------|
| 极致体积控制 | FSM 仅 0.88KB、HSM 2.66KB，AO 3.55KB，适合 Flash 受限 MCU |
| 行为继承 | HSM 的 `AM_HSM_SUPER` 让子状态自动复用父状态逻辑，消除 switch-case 重复 |
| LCA 算法 | `hsm_transition` 用预计算 `hierarchy_level` 做 O(depth) 转换，正确处理任意嵌套进出 |
| ISR 安全通信 | publish/post 全程临界区保护，中断里可直接发事件 |
| 事件所有权模型 | 引用计数 + 内存池，文档提供「事件所有权图」明确生命周期 |
| 协程 async/await | 纯 C99 实现协作式异步，无需 RTOS 也能写非阻塞流程 |
| 防御式编程 | 大量 `AM_ASSERT` 守护不变量：重入检测、entry/exit 禁转、事件损坏探测 |

### 7.2 适用场景

- **复杂状态逻辑的嵌入式设备**：打印机、家电控制面板、工业仪表——状态多且嵌套深，HSM 行为继承大幅降低复杂度。
- **多模块并发通信**：传感器采集 + 通信协议栈 + 用户界面各自封装为 AO，无锁消息通信避免竞争。
- **低功耗 MCU**：协作式调度 + 小体积，适合 Cortex-M0/M3 级别芯片。
- **需要可测试性的固件**：状态机可在 x86 上完整跑单测（无需硬件），CI 友好。
- **教学与研究**：UML 状态机理论的标准 C 实现，是学习 HSM/LCA/Actor 模型的优质参考。

### 7.3 二次开发方向

- 为特定 RTOS（FreeRTOS/ThreadX/Zephyr）实现抢占式 PAL 移植。
- 增加 MQTT/CoAP 事件桥接，让 AO 直接处理物联网消息。
- 结合 onesize 分配器做事件池大小静态分析工具。

## 八、总结

Amast 是一个「小而美」的嵌入式异步框架。它没有追求大而全的功能堆叠，而是把 **Actor 模型 + 层次状态机**这两件最有助于管理嵌入式复杂度的事做到极致：FSM/HSM 内核合计不到 4KB，却完整实现了 UML 状态机的行为继承、LCA 转换、递归初始转换等高级语义；Active Object 用事件队列 + 发布订阅把并发简化为「消息传递」，从机制上消灭共享数据竞争。

从源码质量看，Amast 堪称 C99 嵌入式工程的范本：层层断言守护不变量、inline 函数实现零开销抽象、文档与代码一一对应、x86 体积数据透明公开。对于任何在「裸机标志位」和「RTOS 互斥锁」之间犹豫的嵌入式开发者，Amast 提供了一条结构清晰、可测试、可移植的第三条路——值得一读其源码，更值得在下一个中等复杂度的固件项目中一试。
