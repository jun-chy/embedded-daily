## 项目简介与核心特性一览

Picoware 是一款为 PicoCalc、Cardputer ADV、Waveshare RP2350 系列触屏、Flipper Zero Video Game Module 等设备打造的开源自定义固件。它本质上是一个跑在 RP2040/RP2350 上的**迷你操作系统**：拥有桌面环境、视图管理器、文件管理器、应用商店、OTA 更新、游戏模拟器（GameBoy/Doom/Tetris），甚至自带 PSRAM 扩展、MP3 播放和社交网络功能。

项目提供两套固件形态：`src/MicroPython/` 是基于 MicroPython 的模块化固件（用户可用 Python 写应用），`src/SDK/Picoware/` 是基于 Pico C/C++ SDK 的纯原生固件（性能更高）。这种双轨设计兼顾了易用性与性能。

| 特性 | 说明 |
|------|------|
| 桌面环境 | 带动画的桌面、视图栈导航、双缓冲渲染 |
| 设备端代码编辑器 | 直接在设备上写代码 |
| 设备端 REPL | 交互式 Python 解释器 |
| 文件管理器 | 浏览 SD 卡文件 |
| 应用商店 | OTA 下载安装应用 |
| UF2 引导加载 | USB 拖拽刷写 |
| GameBoy 模拟器 | 运行 GB/GBC ROM |
| 游戏合集 | Doom、Tetris、Pong、贪吃蛇等 |
| PSRAM 支持 | PIO QSPI 驱动扩展 PicoCalc 内存 |
| MP3/WAV 播放 | PIO 音频输出 + minimp3 解码 |
| WiFi | 扫描、 captive portal 配网 |
| 蓝牙 | BLE 扫描与操作 |
| 多核支持 | Core 0 主循环 + Core 1 并行任务 |
| 桌面模拟器 | SDL2 在 Windows/Mac/Linux 运行 |

## 核心代码深度分析

### 一、系统启动与主循环（Picoware.cpp）

整个 SDK 固件的入口极其精炼，体现了「框架先行」的设计哲学：

```cpp
#include <stdio.h>
#include "pico/stdlib.h"
#include "hardware/clocks.h"
#include "src/applications/desktop/desktop.hpp"
#include "src/system/view_manager.hpp"

int main()
{
    stdio_init_all();
    set_sys_clock_khz(200000, true);   // 超频到 200MHz（默认 125MHz）
    ViewManager vm;                     // 构造视图管理器（初始化存储/绘制/输入/键盘）
    vm.add(&desktopView);               // 注册桌面视图
    vm.switchTo("Desktop");             // 切换到桌面
    while (true)
    {
        vm.run();                       // 永久主循环，驱动当前视图
    }
}
```

**逐段解析：**

- `set_sys_clock_khz(200000, true)`：把系统时钟从默认 125MHz 拉到 200MHz。第二个参数 `true` 表示「需要时自动调整外设分频」以保持外设时钟正确。RP2040 官方支持超频到 200MHz，对游戏和 GUI 渲染是必要的性能保障。
- `ViewManager vm`：构造时已完成大量初始化——创建 `picoware` 存储目录、初始化 `Draw` 绘图引擎（含字体）、`Input` 输入管理器、`Keyboard` 软键盘，并清屏。视图管理器是整个系统的中枢。
- `vm.add(&desktopView)`：`desktopView` 是一个全局视图对象（声明在 `desktop.hpp`），桌面是系统的根视图。
- `vm.switchTo("Desktop")`：切换并启动桌面视图，进入主循环前屏幕已显示桌面。
- `while (true) { vm.run(); }`：永不退出的主循环。`vm.run()` 每次调用执行当前视图的 `run()` 回调，处理一帧的输入、逻辑和渲染。这是典型的「裸金属事件循环」架构，没有 RTOS 调度开销。

### 二、视图管理器——视图栈导航（view_manager.cpp）

`ViewManager` 实现了类似移动端的**视图栈导航**：进入子界面时把当前视图压栈，返回时弹栈恢复。这是整个 GUI 系统的骨架。

```cpp
void ViewManager::switchTo(const char *viewName, bool clearStack, bool pushView)
{
    auto view = getView(viewName);
    if (view == nullptr) {
        printf("ViewManager: View '%s' not found or is NULL.\n", viewName);
        return;
    }
    // 切换前把当前视图压入栈（便于后续 back 返回）
    if (this->currentView != nullptr) {
        if (clearStack) {
            this->clearStack();               // 清空历史栈（跳转到根节点场景）
        }
        if (pushView) {
            this->_pushView(this->currentView);  // 当前视图入栈
        }
        this->currentView->stop(this);        // 停止当前视图（释放资源）
        this->clear();                        // 清屏（双缓冲）
    }
    this->currentView = view;
    if (!this->currentView->start(this)) {    // 启动新视图，失败则回退
        this->back();
    }
}
```

**逐段解析：**

- **三参数设计**：`clearStack` 控制是否清空导航历史（如从设置返回桌面应清栈），`pushView` 控制是否保留返回路径（模态弹窗可能不需要入栈）。
- `stop()` / `start()` 生命周期：每个视图有 `start`/`run`/`stop` 三段生命周期。切换时先 `stop` 旧视图释放资源，清屏，再 `start` 新视图。这避免了视图间资源泄漏。
- `start()` 失败回退：若新视图启动失败（如 SD 卡未就绪），自动调用 `back()` 返回上一视图，保证用户不会卡在黑屏。

返回逻辑 `back()` 处理了更复杂的情况——可选地移除当前视图：

```cpp
void ViewManager::back(bool removeCurrentView)
{
    if (this->stackDepth > 0) {
        const View *viewToRemove = nullptr;
        if (this->currentView != nullptr && removeCurrentView) {
            viewToRemove = this->currentView;     // 标记待移除视图（如临时设置页）
        }
        if (this->currentView != nullptr) {
            this->currentView->stop(this);
            this->clear();
        }
        this->stackDepth--;
        this->currentView = this->viewStack[this->stackDepth];  // 弹栈
        this->viewStack[this->stackDepth] = nullptr;
        if (this->currentView != nullptr) {
            if (!this->currentView->start(this)) {
                this->back(false);                // 启动失败递归回退（但不移除视图）
                return;
            }
        }
        // 可选地从视图数组中彻底移除临时视图
        if (viewToRemove != nullptr) {
            // ...从 views[] 数组和栈中清除所有该视图实例...
        }
    }
}
```

**逐段解析：**

- `removeCurrentView` 参数：临时视图（如一次性输入框）返回时应彻底销毁，常驻视图（如设置主页）则保留以便下次快速进入。
- **栈一致性维护**：移除视图时，不仅从 `views[]` 数组删除，还要遍历 `viewStack[]` 清除所有残留引用，并前移填补。注释 `j--` 确保移除后重新检查当前索引（因为前移后新元素落到了同一下标）——这是经典的删除遍历技巧。
- 递归 `back(false)` 处理启动失败链：若连续多个视图都无法启动，逐级回退直到找到一个能启动的，避免死锁。

### 三、NTP 时间获取与重试策略（view_manager.cpp）

`getTime()` 展示了在不可靠无线网络下的**指数退避式重试**设计：

```cpp
const char *ViewManager::getTime()
{
    static uint32_t lastSuccessfulUpdate = 0;
    static uint32_t lastFailedAttempt = 0;
    static char timeBuffer[9] = {0};                    // HH:MM:SS
    static const uint32_t MIN_RETRY_INTERVAL_MS = 5000; // 成功后至少 5 秒不重复请求
    static const uint32_t ERROR_COOLDOWN_MS = 30000;    // 失败后冷却 30 秒再重试

    if (!wifi.isConnected()) {
        return nullptr;
    }
    uint32_t currentTime = to_ms_since_boot(get_absolute_time());

    // 失败冷却期内直接返回缓存值
    if (lastFailedAttempt > 0 && (currentTime - lastFailedAttempt) < ERROR_COOLDOWN_MS) {
        return timeBuffer[0] != 0 ? timeBuffer : nullptr;
    }
    // 成功后短期内用本地系统时钟推算，不重复请求 NTP
    if (lastSuccessfulUpdate > 0 && (currentTime - lastSuccessfulUpdate) < MIN_RETRY_INTERVAL_MS) {
        time_t now = time(nullptr);
        if (now > 946684800) {                          // 大于 2000 年时间戳才算有效
            struct tm timeinfo;
            localtime_r(&now, &timeinfo);
            sprintf(timeBuffer, "%02d:%02d:%02d", timeinfo.tm_hour, timeinfo.tm_min, timeinfo.tm_sec);
            return timeBuffer;
        }
    }
    // 真正发起 NTP 请求
    struct tm timeinfo;
    if (wifi.setTime(timeinfo, 5000)) {                 // 5 秒超时
        sprintf(timeBuffer, "%02d:%02d:%02d", timeinfo.tm_hour, timeinfo.tm_min, timeinfo.tm_sec);
        lastSuccessfulUpdate = currentTime;
        lastFailedAttempt = 0;                          // 成功则重置失败计数
        return timeBuffer;
    } else {
        lastFailedAttempt = currentTime;
        return timeBuffer[0] != 0 ? timeBuffer : nullptr;  // 返回上次缓存的时间
    }
}
```

**逐段解析：**

- **三态时间策略**：未联网返回 `nullptr`；联网但近期成功过，用本地 `time()` 推算（避免频繁 NTP 请求）；联网且冷却期已过，才真正发 NTP。
- `946684800` 是 2000-01-01 的 Unix 时间戳，作为「有效时间」下限——未同步时系统时间可能是 1970 年，这个检查防止显示错误时间。
- **失败冷却 30 秒**：NTP 请求失败后，30 秒内不再重试，避免在弱网环境下每帧都发起注定失败的请求，浪费 CPU 和电量。
- **静态变量缓存**：`timeBuffer` 等用 `static` 保留跨调用状态，即使返回 `nullptr`，下次联网成功后还能恢复显示。

### 四、南桥 I2C 驱动——原子总线保护（southbridge.c）

PicoCalc 的键盘、电池、背光由板载协处理器管理，主控通过 I2C 访问。由于 RP2040 双核都可能访问南桥，作者用 `atomic_bool` 保护总线：

```cpp
volatile atomic_bool sb_i2c_in_use = false;  // 原子标志：I2C 总线是否占用

static size_t sb_write(const uint8_t *src, size_t len)
{
    int result = i2c_write_timeout_us(SB_I2C, SB_ADDR, src, len, false, SB_I2C_TIMEOUT_US * len);
    if (result == PICO_ERROR_GENERIC || result == PICO_ERROR_TIMEOUT) {
        return 0;                            // 写错误返回 0
    }
    return result;
}

// 读键盘 FIFO
uint16_t sb_read_keyboard()
{
    uint8_t buffer[2];
    atomic_store(&sb_i2c_in_use, true);      // 占用总线（原子操作）
    buffer[0] = SB_REG_FIF;                  // 寄存器命令：查询是否有键
    if (sb_write(buffer, 1) != 1) {
        atomic_store(&sb_i2c_in_use, false);
        return 0;
    }
    if (sb_read(buffer, 2) != 2) {
        atomic_store(&sb_i2c_in_use, false);
        return 0;
    }
    atomic_store(&sb_i2c_in_use, false);     // 释放总线
    return buffer[0] << 8 | buffer[1];       // 拼接 16 位按键码
}
```

**逐段解析：**

- **原子总线锁**：`atomic_store` / `atomic_load` 是 C11 原子操作，保证双核环境下标志位的读写不可分割。虽然这不是真正的互斥锁（无法阻塞等待），但对于「尝试占用，失败则跳过」的场景足够——另一个核占用时 `sb_available()` 返回 false，调用方可选择稍后重试。
- **超时 I2C**：`i2c_write_timeout_us(..., SB_I2C_TIMEOUT_US * len)` 按字节数线性放大超时，长数据传输给更多时间，避免误判超时。`PICO_ERROR_GENERIC` / `PICO_ERROR_TIMEOUT` 两种错误都按失败处理。
- **寄存器协议**：先写寄存器地址（1 字节命令），再读/写数据（2 字节）。键盘返回 16 位按键码（`buffer[0] << 8 | buffer[1]`），高字节是键值，低字节是状态。

背光写入则用 `SB_WRITE` 标志位区分读/写操作：

```cpp
uint8_t sb_write_lcd_backlight(uint8_t brightness)
{
    uint8_t buffer[2];
    atomic_store(&sb_i2c_in_use, true);
    buffer[0] = SB_REG_BKL | SB_WRITE;       // 0x05 | 0x80 = 0x85，写操作
    buffer[1] = brightness;
    if (sb_write(buffer, 2) != 2) {
        atomic_store(&sb_i2c_in_use, false);
        return 0;
    }
    if (sb_read(buffer, 2) != 2) {           // 读回确认值
        atomic_store(&sb_i2c_in_use, false);
        return 0;
    }
    atomic_store(&sb_i2c_in_use, false);
    return buffer[1];                        // 返回实际生效的亮度
}
```

**逐段解析：**

- `SB_REG_BKL | SB_WRITE` = `0x05 | 0x80` = `0x85`：南桥协议用最高位区分读写，低 7 位是寄存器地址。这是嵌入式 I2C 外设常见的「寄存器 + R/W 位」编码。
- 写入后**回读确认**：发送亮度值后仍读取 2 字节，返回实际生效值。这能检测写入是否被南桥接受，提高可靠性。

### 五、PSRAM QSPI 驱动——PIO + DMA 扩展内存（psram_qspi.c）

RP2040 只有 264KB SRAM，跑 GUI 和游戏吃力。PicoCalc 板载 PSRAM，Picoware 用 PIO 状态机实现 QSPI 协议来扩展内存。初始化采用「先 SPI 后 QPI」两阶段策略：

```cpp
psram_qspi_inst_t psram_qspi_init(PIO pio, int sm, float clkdiv)
{
    psram_qspi_inst_t qspi;
    qspi.pio = pio;
    qspi.pin_cs = PSRAM_QSPI_PIN_CS;
    qspi.pin_sck = PSRAM_QSPI_PIN_SCK;
    qspi.pin_sio0 = PSRAM_QSPI_PIN_SIO0;

    // 1. 申请 PIO 状态机
    if (sm == -1) {
        qspi.sm = pio_claim_unused_sm(pio, true);
    } else {
        qspi.sm = sm;
        pio_sm_claim(pio, sm);
    }

    // 2. 配置 GPIO 驱动强度与压摆率（高速信号完整性）
    gpio_set_drive_strength(qspi.pin_cs, GPIO_DRIVE_STRENGTH_8MA);
    gpio_set_drive_strength(qspi.pin_sck, GPIO_DRIVE_STRENGTH_8MA);
    gpio_set_slew_rate(qspi.pin_cs, GPIO_SLEW_RATE_FAST);
    gpio_set_slew_rate(qspi.pin_sck, GPIO_SLEW_RATE_FAST);
    for (uint i = 0; i < 4; i++) {
        gpio_set_drive_strength(qspi.pin_sio0 + i, GPIO_DRIVE_STRENGTH_8MA);
        gpio_set_slew_rate(qspi.pin_sio0 + i, GPIO_SLEW_RATE_FAST);
        gpio_set_input_hysteresis_enabled(qspi.pin_sio0 + i, false);  // 关闭迟滞提速
    }

    // 3. 加载 PIO 程序并初始化状态机
    qspi.offset = pio_add_program(pio, &qspi_psram_rw_program);
    qspi_psram_rw_program_init(pio, qspi.sm, qspi.offset, 8, clkdiv,
                               qspi.pin_cs, qspi.pin_sio0);

    // 4. 配置 DMA 通道（写通道 + 读通道）
    qspi.write_dma_chan = dma_claim_unused_channel(true);
    qspi.write_dma_cfg = dma_channel_get_default_config(qspi.write_dma_chan);
    channel_config_set_transfer_data_size(&qspi.write_dma_cfg, DMA_SIZE_8);
    channel_config_set_read_increment(&qspi.write_dma_cfg, true);   // 源地址自增（缓冲区）
    channel_config_set_write_increment(&qspi.write_dma_cfg, false); // 目标固定（PIO TX FIFO）
    channel_config_set_dreq(&qspi.write_dma_cfg, pio_get_dreq(pio, qspi.sm, true));  // PIO 就绪触发
    dma_channel_set_write_addr(qspi.write_dma_chan, &pio->txf[qspi.sm], false);

    // 5. SPI 模式复位 PSRAM，再切换到 QPI 模式
    psram_send_spi_command(&qspi, 0x66);  // Reset Enable
    busy_wait_us(50);
    psram_send_spi_command(&qspi, 0x99);  // Reset
    busy_wait_us(100);
    psram_send_spi_command(&qspi, 0x35);  // Enter QPI
    busy_wait_us(100);
    return qspi;
}
```

**逐段解析：**

- **GPIO 信号完整性**：QSPI 高速通信时，8mA 驱动强度 + 快压摆率 + 关闭输入迟滞，能减少上升/下降沿时间，保证 4 位数据线在高速下眼图合格。这是硬件层面的精心调优。
- **DMA 双通道**：写通道「源地址自增、目标固定」——从内存缓冲逐字节搬到 PIO TX FIFO；读通道反之。`pio_get_dreq(pio, sm, true)` 设置数据请求信号，PIO 准备好接收/发送时才触发 DMA 传输，实现**无缝流速匹配**，CPU 完全不参与搬运。
- **SPI→QPI 两阶段**：PSRAM 上电默认是 SPI 模式（单数据线），必须先用 SPI 命令发 `0x35` 切到 QPI 模式（4 数据线）。`psram_send_spi_command()` 临时禁用 PIO、用 GPIO bit-bang 发单字节命令，发完再恢复 PIO 控制——这种「降级到 bit-bang 完成初始化」是 PIO 外设的通用技巧。

QPI 模式下的命令发送走 PIO + DMA，效率远高于 bit-bang：

```cpp
static void psram_send_qpi_command(psram_qspi_inst_t *qspi, uint8_t cmd)
{
    // QSPI PIO 协议：[写 nibble 数, 读 nibble 数, 命令字节]
    uint8_t qpi_cmd[] = {
        2,    // 2 个 nibble = 1 字节待写
        0,    // 0 个 nibble 待读
        cmd}; // 命令字节
    qspi_write_dma_blocking(qspi, qpi_cmd, sizeof(qpi_cmd));
}
```

**逐段解析：**

- PIO 程序用「nibble 计数」协议：4 数据线一次传 4 位（1 nibble），1 字节需 2 个 nibble。前两字节告诉 PIO 状态机本次要写/读多少 nibble，第三字节是实际命令。这种协议让单个 PIO 程序能灵活处理不同长度的 QSPI 事务。

PSRAM 自检函数验证读写正确性：

```cpp
int psram_qspi_test(psram_qspi_inst_t *qspi)
{
    uint8_t test_val = 0xA5;
    uint32_t test_addr = 0x1000;
    psram_qspi_write8(qspi, test_addr, test_val);
    busy_wait_us(10);
    uint8_t read_val = psram_qspi_read8(qspi, test_addr);
    if (read_val != test_val) return -1;          // 8 位测试失败

    uint32_t test_val32 = 0xDEADBEEF;
    psram_qspi_write32(qspi, test_addr + 4, test_val32);
    busy_wait_us(10);
    uint32_t read_val32 = psram_qspi_read32(qspi, test_addr + 4);
    if (read_val32 != test_val32) return -2;      // 32 位测试失败
    return 0;                                      // 全部通过
}
```

`0xA5`（二进制 `10100101`）和 `0xDEADBEEF` 是经典的测试模式——交替位和混合位能检测数据线粘联、字节序错误等问题。

### 六、多核安全——自旋锁保护

PSRAM 驱动还支持双核安全访问，通过 RP2040 硬件自旋锁实现：

```cpp
#if defined(PSRAM_QSPI_SPINLOCK)
    int spin_id = spin_lock_claim_unused(true);
    qspi.spinlock = spin_lock_init(spin_id);
#endif
```

RP2040 有 32 个硬件自旋锁，`spin_lock_claim_unused` 申请一个空闲锁。与 `atomic_bool` 不同，自旋锁是**真正的互斥**——获取不到的核会自旋等待，适合必须独占访问的场景（如 PSRAM 读写不能交错）。这种分层并发控制（南桥用原子标志「尝试占用」，PSRAM 用自旋锁「强制独占」）体现了对不同临界区特性的精准把握。

cd src/SDK/Picoware
mkdir build && cd build

make -j4

cd src/MicroPython

2. 设备枚举为 U 盘，将 `picoware.uf2` 拖入即可。
3. 设备自动重启进入 Picoware 桌面。

## 项目亮点与适用场景

### 技术亮点

- **裸金属迷你 OS**：在无 RTOS 的 RP2040 上实现了视图栈导航、桌面环境、应用商店，架构清晰可复用。
- **PIO QSPI PSRAM**：用 PIO 状态机 + DMA 实现 4 倍速 PSRAM 访问，SPI→QPI 两阶段初始化 + 信号完整性调优，是 RP2040 PIO 的高级应用范本。

- **双形态固件**：C++ SDK 版求性能、MicroPython 版求扩展性，同一套设备两种开发体验。
- **南桥抽象**：把键盘/电池/背光/电源统一抽象为 I2C 寄存器协议，主控代码与硬件解耦。
