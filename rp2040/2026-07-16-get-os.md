## 一、项目简介

GeT_OS 是一个为 Raspberry Pi Pico（RP2040）和 RP2350 微控制器从零手写的宏内核操作系统。与使用 Pico SDK 的项目不同，GeT_OS 完全不依赖官方 SDK，核心代码以 ARM 汇编编写，辅以 C 语言实现高层逻辑。它实现了动态内存分配、ELF 动态加载器、自定义 libc、抢占式多任务调度、POSIX 风格系统调用、虚拟文件系统、TCP/IP 网络栈和 VGA 显示驱动。

这不是一个玩具项目——它有完整的进程管理、文件系统抽象、动态链接支持，甚至能加载和运行编译为 ELF 的用户空间程序。

## 四、核心代码深度分析

### 4.1 进程调度器与上下文切换（proc.c）

GeT_OS 的进程调度器是整个内核最复杂的部分。它管理进程生命周期、执行上下文切换、处理信号。

进程创建：

```c
static struct Process *create_blank_process(
    void (*process_entry_ptr)(void),
    void *static_base,
    char *const argv[])
{
    static pid_t pid = PID_INIT;
    if (pid >= scheduler.max_processes) {
        __asm__("bkpt #0");  // 超过最大进程数，触发断点
    }

    // 分配进程栈页（2KB）
    void *process_page = kmalloc(DEFAULT_PROCESS_STACK_SIZE);
    void *kstack = process_page + DEFAULT_PROCESS_STACK_SIZE - sizeof(size_t);
    void *pstack_begin = process_page + DEFAULT_PROCESS_SP_OFFSET - sizeof(size_t);

    // 初始化 argv 参数到栈上
    size_t offset = process_stack_init_argv(pstack_begin, argv);
    void *pstack = pstack_begin - offset;

    // 构建初始栈帧（模拟中断返回帧）
    create_process_stack_frame(&pstack, &exit,
                               process_entry_ptr,
                               EXC_RETURN_THREAD_PSP_CODE);

    // 设置 argc 和 argv 指针
    *(size_t *)(pstack_begin - offset - 28) = (size_t *)(pstack_begin - offset + sizeof(size_t));
    *(size_t *)(pstack_begin - offset - 32) = (offset - sizeof(size_t)) / sizeof(size_t);

    // r9 = 静态基址寄存器（用于 RWPI 位置无关代码）
    *(size_t *)(pstack + 28) = (uintptr_t)static_base;

    struct Process process = {
        .pid = pid,
        .pstate = NEW,
        .kernel_mode = false,
        .stack_page_ptr = process_page,
        .pstack = pstack,
        .kstack = kstack,
        .allocated_memory = DEFAULT_PROCESS_STACK_SIZE,
        .parent = nullptr,
        .children_count = 0,
        .signal_mask = 0,
        .pending_signals = nullptr,
        .signal_handled = false,
    };

    init_default_sighandlers(&process);
    scheduler.processes[pid] = process;
    scheduler.processes_count += 1;
    return &scheduler.processes[pid++];
}
```

**逐段解析：**

- `kmalloc(DEFAULT_PROCESS_STACK_SIZE)` —— 每个进程分配 2KB 栈空间。栈页内同时包含内核栈和用户栈，通过不同的偏移量分隔。
- `process_stack_init_argv()` —— 将 argv 数组逆序压入栈中，模拟 C 运行时初始化。argc 存放在 argv 指针之前的 4 字节。
- `create_process_stack_frame()` —— 构建一个伪造的异常返回栈帧。ARM Cortex-M 的异常返回机制会自动从栈中恢复寄存器，通过构建正确的栈帧，第一次上下文切换时就像从中断返回一样开始执行新进程。
- `EXC_RETURN_THREAD_PSP_CODE` —— 异常返回值，指示返回到线程模式并使用 PSP（进程栈指针）。
- `r9 = static_base` —— 这是 RWPI（Read-Write Position Independence）的关键。ARM 的 `-fropi -frwpi` 编译选项使代码使用 r9 寄存器作为静态数据基址，实现位置无关的读写数据访问。

上下文切换是操作系统的心脏：

```c
__attribute__((optimize("omit-frame-pointer")))
void context_switch(void) {
    register struct Process *process = scheduler_get_next_process();
    scheduler.current_process = process;

    // 信号处理优先级：已处理信号 > 待处理信号 > 内核模式 > 用户模式
    if (process->signal_handled) {
        goto switch_to_userspace;
    }

    int pending_signal = get_pending_signal();
    if (pending_signal == SIGCHLD && process->pstate == WAITING_FOR_CHILD_EXIT) {
        process->pstate = READY;
        goto switch_to_kernelspace;
    }
    else if (pending_signal >= 0 && handle_pending_signal(pending_signal)) {
        goto switch_to_userspace;
    }

    if (process->kernel_mode) {
        goto switch_to_kernelspace;
    }

switch_to_userspace:
    process->pstate = RUNNING;
    // 设置栈限制寄存器（防止栈溢出）
    __asm__("msr psplim, %0" : : "r"(process->stack_page_ptr));
    __asm__("msr msplim, %0" : : "r"(process->stack_page_ptr + msplim_offset));
    // 恢复栈指针
    __asm__("msr psp, %0" : : "r"(process->pstack));
    __asm__("msr msp, %0" : : "r"(process->kstack));
    // 从 PSP 恢复寄存器并返回用户空间
    RECALL_USERSPACE_STATE

switch_to_kernelspace:
    __asm__("msr psplim, %0" : : "r"(process->stack_page_ptr));
    __asm__("msr msplim, %0" : : "r"(process->stack_page_ptr + msplim_offset));
    __asm__("msr psp, %0" : : "r"(process->pstack));
    __asm__("msr msp, %0" : : "r"(process->kstack));
    // 从 MSP 恢复寄存器并返回内核空间
    RECALL_KERNELSPACE_STATE
}
```

**关键分析：**

- `__attribute__((optimize("omit-frame-pointer")))` —— 禁用帧指针优化。上下文切换函数不能使用帧指针，因为它直接操作栈指针，编译器生成的栈帧会破坏寄存器恢复逻辑。
- `msplim` / `psplim` —— ARM Cortex-M 的栈限制寄存器。设置后，栈指针超出限制会触发 HardFault，这是内核最基础的内存保护机制。
- `RECALL_USERSPACE_STATE` 宏展开为内联汇编，从 PSP 指向的栈帧中恢复 r2-r11 寄存器、LR 和 CONTROL 寄存器，然后通过 `bx lr` 返回。`lr` 中存储的是 `EXC_RETURN` 值，CPU 据此判断返回用户模式还是内核模式。
- 调度优先级链：信号处理 > 子进程等待 > 内核模式 > 用户模式。这确保了信号能及时处理，同时不会饿死正常调度。

调度算法：

```c
static struct Process *scheduler_get_next_process() {
    static size_t current_index = PID_INIT;

    if (scheduler.processes_count == 0)
        return nullptr;
    else if (scheduler.processes_count == 1)
        return scheduler.current_process;

    int max_depth_counter = 0;
    do {
        current_index += 1;
        max_depth_counter += 1;

        if (current_index == scheduler.max_processes)
            current_index = PID_INIT;

        // 防止死锁：如果遍历两倍最大进程数仍未找到 READY 进程
        if (max_depth_counter == 2 * scheduler.max_processes)
            return &scheduler.process_idle;

        // 有待处理信号的进程优先调度
        const struct Process *current_process = &scheduler.processes[current_index];
        if (current_process->pending_signals)
            return &scheduler.processes[current_index];

    } while (scheduler.processes[current_index].pstate != READY);

    return &scheduler.processes[current_index];
}
```

- 轮转调度（Round Robin）：从当前索引开始遍历进程数组，找到第一个 READY 状态的进程。
- 信号优先：有待处理信号的进程会被立即调度，确保信号处理的及时性。
- 死锁保护：如果遍历两倍最大进程数仍未找到可运行进程，切换到 idle 进程（空循环），防止调度器自旋锁死。

### 4.2 内核内存分配器（memory.c）

GeT_OS 实现了一个改进的链表内存分配器：

```c
struct Chunk {
    void *ptr;           // 用户数据指针
    size_t size;         // 分配大小（已对齐）
    struct Chunk *next_node;  // 下一个 Chunk
};

static struct {
    struct Chunk *head;
    size_t allocated_size;
    unsigned int allocated_chunks;
    void *heap_start;
} Allocator = {nullptr, 0, 0, heap_start_ptr};
```

**内存布局：**

```
堆内存布局：
┌──────────┬──────────┬──────────┬─────────────────┬──────────┐
│ Chunk 0  │ User Data│ Chunk 1  │ Free Space      │ Chunk 2  │
│ (header) │          │ (header) │                 │ (header) │
└──────────┴──────────┴──────────┴─────────────────┴──────────┘
     │         │         │                            │
     │◄─size──►│         │                            │
     │                   │◄─────── next_node ────────►│
     └───── next_node ──►│
```

分配函数：

```c
void *kmalloc(size_t size) {
    const size_t aligned_size = align_size(size);
    struct Chunk chunk = {nullptr, aligned_size, nullptr};

    if (Allocator.head == nullptr) {
        // 第一个分配：从堆起始地址开始
        chunk.ptr = Allocator.heap_start + sizeof(struct Chunk);
        chunk.next_node = nullptr;
        Allocator.head = (struct Chunk *)Allocator.heap_start;
    }
    else {
        // 查找空闲间隙
        struct Chunk *previous_chunk = find_free_chunk_predecessor(aligned_size);
        chunk.next_node = previous_chunk->next_node;
        previous_chunk->next_node = align_ptr(previous_chunk->ptr + previous_chunk->size);
        chunk.ptr = (void *)previous_chunk->next_node + sizeof(struct Chunk);
    }

    // 检查是否超出堆边界
    const size_t heap_size_after_alloc = get_current_heap_size()
        + chunk.size + sizeof(struct Chunk);
    if (heap_size_after_alloc >= (size_t)(heap_end_ptr - heap_start_ptr))
        return nullptr;

    Allocator.allocated_size += chunk.size;
    Allocator.allocated_chunks += 1;

    // 写入 Chunk 头部
    *(struct Chunk *)(chunk.ptr - sizeof(struct Chunk)) = chunk;
    return chunk.ptr;
}
```

**逐段解析：**

- `align_size()` —— 将请求大小对齐到 4 字节边界，最小 8 字节。对齐保证内存访问效率和结构体布局正确。
- 每个 Chunk 头部紧跟在用户数据之前，通过 `chunk.ptr - sizeof(struct Chunk)` 可以找到头部。这是一种常见的内联元数据设计。
- `find_free_chunk_predecessor()` —— 遍历链表查找足够大的空闲间隙。如果找不到，在链表末尾追加。这是首次适应（First-Fit）策略的变体，优先利用已释放的空间。
- 堆边界检查防止内存溢出：`heap_end_ptr` 和 `heap_start_ptr` 是链接脚本定义的符号。

释放函数：

```c
void kfree(void *ptr) {
    if (ptr == nullptr) return;

    struct Chunk *temp = Allocator.head;

    // 特殊情况：释放的是头节点
    if (temp->ptr == ptr) {
        Allocator.head = temp->next_node;
        Allocator.allocated_chunks -= 1;
        Allocator.allocated_size -= temp->size;
        return;
    }

    // 一般情况：找到目标节点的前驱
    while (temp->next_node && temp->next_node->ptr != ptr)
        temp = temp->next_node;

    if (temp->next_node->ptr == ptr) {
        struct Chunk *found_chunk = temp->next_node;
        const size_t found_chunk_size = found_chunk->size;
        struct Chunk *next = found_chunk->next_node;
        // 前驱跳过被释放的节点
        temp->next_node = next;
        Allocator.allocated_chunks -= 1;
        Allocator.allocated_size -= found_chunk_size;
    }
}
```

- 释放操作是 O(N) 的链表删除。从头部释放最快（O(1)），从尾部释放最慢（O(N)）。
- 释放的内存不会立即归还给系统，而是留在链表中供后续 `find_free_chunk_predecessor()` 重用。
- 注意：当前实现没有合并相邻空闲块（coalescing），这可能导致外部碎片。作者在注释中标注了 `TODO: extend existing block if there is free space`。

用户空间内存分配通过系统调用委托给内核：

```c
void *sys_malloc(size_t size) {
    struct Process *current_process = scheduler_get_current_process();

    const size_t size_before = Allocator.allocated_size;
    void *ptr = kmalloc(size);
    if (!ptr) return nullptr;

    const size_t size_after = Allocator.allocated_size;
    // 记录进程的内存使用量
    current_process->allocated_memory += size_after - size_before;
    return ptr;
}
```

- `sys_malloc()` 是用户空间 `malloc()` 的系统调用入口。它调用内核的 `kmalloc()`，然后更新当前进程的内存统计。
- 通过比较分配前后的 `Allocator.allocated_size` 差值来计算实际分配大小（包含对齐和 Chunk 头部开销），确保进程内存统计准确。

### 4.3 ELF 动态加载器（loader.c）

GeT_OS 的 ELF 加载器是其最令人印象深刻的部分。它能解析 ELF32 格式、加载程序段到 RAM、解析动态符号、执行重定位。

ELF 验证：

```c
static bool validate_elf(const void *fbytes) {
    const Elf32_EHdr *elf_header = fbytes;
    constexpr uint8_t magic[8] = {0x7f, 0x45, 0x4c, 0x46, 0x01, 0x01, 0x01, 0x00};

    // 检查 ELF 魔数
    if (memcmp(elf_header->e_ident, magic, 8) != 0)
        return false;

    // 只支持 EXEC 和 DYN 类型
    if (elf_header->e_type != E_TYPE_EXEC && elf_header->e_type != E_TYPE_DYN)
        return false;

    // 只支持 ARM 架构
    if (elf_header->e_machine != E_MACHINE_ARM)
        return false;

    if (elf_header->e_version != 1)
        return false;

    return true;
}
```

- 魔数 `0x7f 'E' 'L' 'F'` 是所有 ELF 文件的标准标识。
- `E_TYPE_DYN`（共享库/PIE）支持意味着 GeT_OS 可以加载位置无关的可执行文件，这在没有 MMU 的 MCU 上尤为重要。
- `E_MACHINE_ARM`（0x28）确保只加载 ARM 架构的二进制文件。

ELF 段加载：

```c
static struct ProcessPage *load_pages_to_ram(const void *fbytes) {
    const Elf32_EHdr *elf_header = fbytes;
    const Elf32_PHdr *phdr_table = fbytes + elf_header->e_phoff;
    unsigned int bufsz = 1024;
    unsigned char *buffer = kmalloc(bufsz);
    memset(buffer, 0, bufsz);

    for (size_t i = 0; i < elf_header->e_phnum; ++i) {
        const Elf32_PHdr *phdr = &phdr_table[i];

        // 只加载 PT_LOAD 类型的段
        if (phdr->p_type != PT_LOAD)
            continue;

        size_t index = phdr->p_paddr;

        // 动态扩展缓冲区
        if (index + phdr->p_memsz >= bufsz) {
            bufsz += index + phdr->p_memsz;
            buffer = krealloc(buffer, bufsz);
        }

        // 将段数据从 ELF 文件拷贝到 RAM 缓冲区
        memcpy(buffer + index, fbytes + phdr->p_offset, phdr->p_filesz);
    }

    struct ProcessPage *ppage = kmalloc(sizeof(*ppage));
    ppage->page_ptr = buffer;
    ppage->page_size = bufsz;
    return ppage;
}
```

**解析：**

- `PT_LOAD` 段是 ELF 中需要加载到内存的可加载段。一个 ELF 可执行文件通常有 2-3 个 PT_LOAD 段（.text、.rodata、.data+.bss）。
- `p_paddr` 作为加载地址——在无 MMU 的 MCU 上，物理地址就是运行地址。
- `p_filesz` 是 ELF 文件中段数据的实际大小，`p_memsz` 是运行时需要的内存大小（`.bss` 部分在文件中不占空间但需要零初始化内存）。这里只拷贝 `p_filesz` 字节，剩余部分由 `memset(buffer, 0, bufsz)` 的零初始化覆盖。
- 缓冲区动态扩展：初始分配 1KB，遇到更大的段时通过 `krealloc()` 扩展。

动态库符号解析：

```c
static uint32_t get_dyn(const char *symbol_name,
                        const struct Dynlib *dynlib) {
    const unsigned long symbol_hash = elf_hash(symbol_name);

    // 使用 ELF 哈希表进行符号查找
    const Word *hash = dynlib->sections.hash;
    Word nbucket = hash[0];
    Word nchain = hash[1];
    const Word *bucket = &hash[2];
    const Word *chain = &hash[2 + nbucket];

    // 通过哈希表定位符号
    Word index = bucket[symbol_hash % nbucket];
    while (index != STN_UNDEF) {
        const Elf32_Sym *symbol = &dynlib->sections.dynsym[index];
        const char *name = &dynlib->sections.dynstr[symbol->st_name];

        if (strcmp(name, symbol_name) == 0) {
            const Address offset = symbol->st_value;
            return offset;
        }
        // 哈希冲突链
        index = chain[index];
    }
    return 0x00;
}
```

**解析：**

- `elf_hash()` 是 ELF 标准定义的符号哈希函数（SysV Hash），用于快速符号查找。
- ELF 哈希表由 `nbucket`（桶数）、`nchain`（链数）、`bucket[]`（桶数组）和 `chain[]`（冲突链数组）组成。
- 查找过程：计算符号哈希 → 取模得到桶索引 → 从桶中获取第一个符号索引 → 沿冲突链遍历直到找到匹配的符号名或遇到 `STN_UNDEF`。
- 这种设计将符号查找从 O(N) 线性搜索优化到平均 O(1) 的哈希查找，对于有数百个导出符号的 libc 来说性能提升显著。

ELF 哈希函数：

```c
static unsigned long elf_hash(const unsigned char *name) {
    unsigned long h = 0, g;
    while (*name) {
        h = (h << 4) + *name++;
        if (g = h & 0xf0000000)
            h ^= g >> 24;
        h &= ~g;
    }
    return h;
}
```

- 这是 SysV ELF 规范定义的标准哈希函数。每个字符通过 `(h << 4) + char` 累加，然后通过高位掩码进行混合，确保哈希分布均匀。
- 所有 ELF 工具链（GCC、LD）都使用相同的算法，保证链接时和运行时哈希一致。

ELF 节区解析：

```c
static Elf32_Sections parse_sections(const void *fbytes) {
    Elf32_Sections sections = {0};
    const Elf32_EHdr *elf_header = fbytes;
    const Elf32_SHdr *shdr_table = fbytes + elf_header->e_shoff;
    const Elf32_SHdr *shstrtab = &shdr_table[elf_header->e_shstrndx];

    for (size_t i = 0; i < elf_header->e_shnum; ++i) {
        const Elf32_SHdr *shdr = &shdr_table[i];
        const char *section_name =
            (char *)(fbytes + shstrtab->sh_offset + shdr->sh_name);

        // 按名称匹配节区
        if (strcmp(section_name, ".text") == 0) {
            sections.text = (uint8_t *)section_offset;
            sections.text_len = shdr->sh_size;
        }
        else if (strcmp(section_name, ".dynsym") == 0) {
            sections.dynsym = (Elf32_Sym *)(fbytes + shdr->sh_offset);
            sections.dynsym_len = shdr->sh_size / sizeof(Elf32_Sym);
        }
        else if (strcmp(section_name, ".dynamic") == 0) {
            sections.dynamic = fbytes + shdr->sh_offset;
            sections.dynamic_len = shdr->sh_size;
        }
        // ... 其他节区
    }
    return sections;
}
```

- `e_shstrndx` 是 ELF 头中的节名字符串表索引。通过它可以找到存储所有节名的字符串表。
- 加载器解析所有关键节区：`.text`（代码）、`.data`（数据）、`.rodata`（只读数据）、`.bss`（未初始化数据）、`.dynsym`（动态符号表）、`.dynstr`（动态字符串表）、`.got`/`.got.plt`（全局偏移表）、`.rel.plt`/`.rel.dyn`（重定位表）、`.dynamic`（动态段）、`.hash`（符号哈希表）。
- `parse_sections()` 返回一个包含所有节区指针和大小的结构体，后续的重定位和符号解析都基于这些信息。

### 4.4 进程创建与 ELF 加载（proc.c + loader.c）

从 ELF 文件创建进程：

```c
pid_t sys_spawn_process(
    int fd,
    const spawn_file_actions_t *file_actions,
    const spawnattr_t *attrp,
    char *const argv[],
    char *const envp[])
{
    struct Process *current = scheduler.current_process;

    // 读取 ELF 文件到内存
    const size_t executable_size = current->files.fdtable[fd]->f_inode->i_size;
    char *executable_buffer = kmalloc(executable_size);
    sys_read(fd, executable_buffer, executable_size);

    // 解析并加载 ELF
    struct ProcessPage *ppage = load_exec(executable_buffer);
    kfree(executable_buffer);

    if (!ppage || !ppage->page_ptr)
        return -ENOEXEC;

    // 获取入口地址
    void *_start_address = (uint8_t *)ppage->page_ptr + ppage->_start_offset;

    // 创建进程（构建栈帧）
    struct Process *process = create_blank_process(
        _start_address, ppage->static_base, argv);

    process->ppage = ppage;
    process->allocated_memory += ppage->page_size;

    // 继承父进程的文件描述符表
    struct File **fdtable = kmalloc(
        sizeof(struct File *) * MAX_OPEN_FILE_DESCRIPTORS);
    for (size_t i = 0; i < MAX_OPEN_FILE_DESCRIPTORS; ++i)
        process->files.fdtable[i] = current->files.fdtable[i];

    // 设置进程关系
    process->parent = current;
    process->pgid = current->pgid;
    process->root = current->root;
    process->pwd = current->pwd;

    process->pstate = READY;
    return process->pid;
}
```

**解析：**

- 完整的进程创建流程：读取 ELF → 解析段 → 分配内存 → 构建栈帧 → 继承文件描述符 → 设置进程关系 → 标记为 READY。
- 文件描述符继承：子进程获得父进程 fd 表的副本，类似 Unix 的 `fork()` 语义（但不复制内存空间）。
- `ppage->static_base` 是动态库的静态数据基址，用于 RWPI 位置无关代码。如果 ELF 是静态链接的，`static_base` 为 NULL。

### 4.5 中断与系统调用入口（ctx.S + isr.S）

上下文切换的汇编实现：

```asm
// RECALL_STATE_FROM 宏：从栈中恢复寄存器状态
.macro RECALL_STATE_FROM sp
    mrs    r0, \sp          @ 加载栈指针到 r0
    ldmia  r0!, {r2-r11}    @ 从栈中恢复 r2-r11
    mov    lr, r2           @ 恢复链接寄存器
    msr    CONTROL, r3      @ 恢复 CONTROL 寄存器（特权级/栈选择）
    isb                     @ 指令同步屏障
    msr    \sp, r0          @ 更新栈指针
    bx     lr               @ 返回（EXC_RETURN 决定用户/内核模式）
.endm
```

- `mrs r0, \sp` —— 读取 PSP 或 MSP 到 r0。PSP（Process Stack Pointer）用于用户模式，MSP（Main Stack Pointer）用于内核模式。
- `ldmia r0!, {r2-r11}` —— 从栈中批量加载 10 个寄存器。r2 存放返回地址（LR），r3 存放 CONTROL 寄存器值。
- `msr CONTROL, r3` —— CONTROL 寄存器控制栈选择（MSP/PSP）和特权级。恢复它确保进程在正确的模式下运行。
- `isb` —— 指令同步屏障，确保 CONTROL 寄存器修改生效后再继续执行。
- `bx lr` —— LR 中存储的是 EXC_RETURN 值（如 `0xFFFFFFFD`），CPU 据此执行异常返回，自动恢复 r0-r3、r12、PC、xPSR。

### 4.6 自定义 libc（kstdlib/libc.c）

GeT_OS 实现了自己的 C 标准库，供内核和用户空间共用：

```c
// 用户空间通过 SVC 指令调用内核服务
// libc 中的 malloc() 实现：
void *malloc(size_t size) {
    void *ptr;
    __asm__ volatile(
        "mov r0, %1\n"
        "svc %2\n"          // 触发 MALLOC_SVC 系统调用
        "mov %0, r0\n"
        : "=r"(ptr)
        : "r"(size), "i"(MALLOC_SVC)
        : "r0"
    );
    return ptr;
}
```

- 用户空间的 `malloc()` 通过 `svc`（SuperVisor Call）指令陷入内核模式。ARM Cortex-M 的 `svc` 指令触发 SVC 异常，内核在异常处理程序中根据系统调用号分发到对应的处理函数。
- `svc %2` 中的立即数是系统调用号（如 `MALLOC_SVC = 40`），内核的 `syscall_table.c` 中维护着号到函数的映射。
- 参数通过 r0-r3 寄存器传递，返回值放在 r0。这与 ARM AAPCS（ARM Architecture Procedure Call Standard）一致。

### 技术亮点

1. **完全无 SDK 依赖**：从启动汇编到驱动程序全部从零编写，不依赖 Pico SDK。这要求作者对 ARM Cortex-M 架构、RP2040 硬件寄存器有深入理解。

2. **ELF 动态加载器**：在无 MMU 的 MCU 上实现完整的 ELF 加载、动态链接和符号解析。支持 GOT/PLT 重定位、动态库加载和位置无关代码（RWPI）。

5. **用户空间与内核分离**：内核和用户空间有独立的栈和代码空间。用户程序通过 SVC 指令陷入内核，实现了基本的特权级分离。
