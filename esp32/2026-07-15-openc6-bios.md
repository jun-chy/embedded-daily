**项目描述：** OpenC6 BIOS 是一个面向 ESP32-C6（8MB Flash）的开源 RISC-V 固件平台。它实现了 BIOS/Payload（宿主/载荷）双体架构，配备自定义系统调用接口（ABI）、独立的 LP-Core 协处理器管理引擎（Out-of-Band Management），以及防变砖的 A/B OTA 升级系统。其核心理念是：把 PC/服务器风格的"BIOS + 操作系统载荷"分层架构移植到微控制器之上，让硬件初始化与应用逻辑彻底解耦。

## 二、项目简介与核心特性

传统嵌入式固件多为"单体应用"（monolithic）——初始化代码、业务逻辑、网络栈全部塞进一个二进制。OpenC6 反其道而行：BIOS 本身只负责硬件初始化、提供带外管理能力，并暴露一套标准化的系统调用表（ABI）；真正的"应用"以独立 **Payload**（裸机载荷）的形式存在，可热插拔、可网络下载、可直接在 RAM 或 Flash 上就地执行（XIP）。

### 核心特性一览

| 特性 | 说明 |
| :--- | :--- |
| **LP-Core 管理引擎（ME）** | 独立 RISC-V 协处理器，实现带外监控：电源按键、硬件看门狗、紧急热关机，即使主系统崩溃仍可工作 |
| **SchedUtil 动态调频** | 基于 FreeRTOS 空闲周期实时计算 CPU 负载，自适应在 80/120/160 MHz 间切换频率 |
| **网络启动（PXE）** | 通过 Wi-Fi 动态拉取并执行裸机 Payload，无需反复烧写 Flash |
| **防变砖 A/B OTA** | BIOS 自身的网络升级受硬件回滚机制保护，新固件崩溃可自动回退 |
| **复古 Web 配置界面** | AP 模式下的 Web 服务器，提供经典蓝屏 BIOS 界面，配置超频、BOD、热限等 |
| **标准化 ABI** | Payload 无需 ESP-IDF 即可编译（`-nostdlib -fPIC`），体积仅 2–10 KB，复用 BIOS 提供的 Wi-Fi、数学、加密引擎 |
| **日志结构循环文件系统** | 自研分块式文件系统，具备动态磨损均衡与动态 RAM 索引，支持大于单扇区的文件 |

## 五、核心代码深度分析

### 5.1 main.c —— 应用入口

```c
#include "freertos/FreeRTOS.h"
#include "freertos/task.h"
#include "bios_core.h"

void app_main(void)
{
    // Background stress test removed. BIOS is now completely clean and stable!
    bios_core_start();
}
```

**逐段解析：**

- `#include "freertos/FreeRTOS.h"` 与 `task.h`：引入 ESP-IDF 的 FreeRTOS 内核头文件，因为 ESP32-C6 的应用入口运行在 FreeRTOS 任务上下文中。
- `#include "bios_core.h"`：引入 BIOS 核心模块接口，声明 `bios_core_start()`。
- `app_main(void)`：ESP-IDF 规定的应用入口函数，由 ESP-IDF 启动代码在系统初始化完成后以独立任务调用。
- 注释 `Background stress test removed...`：表明早期版本曾在此处放置压力测试代码，现已移除以保证 BIOS 干净稳定——这呼应了"宿主只做初始化、不做业务"的设计哲学。
- `bios_core_start();`：唯一的实质调用，把控制权完全交给 BIOS 核心。从此处开始进入庞大的初始化与启动状态机。

入口的极简是有意为之：所有复杂度被封装进 `bios_core_start()`，保持入口可读、可测试。

### 5.2 bios_init.c —— ABI 构建与系统调用表

#### 5.2.1 ABI 包装器函数

```c
// ABI 系统调用包装器
static void bios_print(const char *str) {
    printf("%s", str);
}

static void bios_sha256(const uint8_t *input, uint32_t len, uint8_t *output) {
    psa_crypto_init();
    size_t hash_len;
    psa_hash_compute(PSA_ALG_SHA_256, input, len, output, 32, &hash_len);
}

static int32_t bios_wifi_connect(const char* ssid, const char* pass) {
    nvram_set_wifi_sta_config(ssid, pass);
    return (wifi_mgmt_start_sta() == ESP_OK) ? 0 : -1;
}

static uint32_t bios_get_free_ram(void) {
    return heap_caps_get_free_size(MALLOC_CAP_INTERNAL | MALLOC_CAP_8BIT);
}
```

**逐段解析：**

- `bios_print`：将 Payload 的字符串输出请求转发到标准 `printf`，即 BIOS 当前控制台。这是 Payload 唯一的文本输出通道。
- `bios_sha256`：先调用 `psa_crypto_init()` 初始化 PSA 加密子系统，再用 `psa_hash_compute()` 以硬件加速方式计算 SHA-256。输出固定 32 字节。`hash_len` 接收实际写入长度。注意它复用 ESP32-C6 的硬件密码加速器，对无标准库的 Payload 意义重大。
- `bios_wifi_connect`：先把 SSID/密码写入 NVRAM 持久化（`nvram_set_wifi_sta_config`），再调用 `wifi_mgmt_start_sta()` 启动 STA 模式。返回 `0` 表示成功、`-1` 表示失败，符合 POSIX 风格的"0 为成功"约定。
- `bios_get_free_ram`：用 `heap_caps_get_free_size` 并指定 `MALLOC_CAP_INTERNAL | MALLOC_CAP_8BIT`，仅统计内部 DRAM 中可用的字节对齐堆内存，避免把外部 PSRAM 误算进 Payload 可执行内存。

#### 5.2.2 整数数学加速

```c
// 整数数学加速
static uint32_t bios_math_isqrt(uint32_t x) {
    uint32_t res = 0;
    uint32_t bit = 1UL << 30;
    while (bit > x) bit >>= 2;
    while (bit != 0) {
        if (x >= res + bit) {
            x -= res + bit;
            res = (res >> 1) + bit;
        } else {
            res >>= 1;
        }
        bit >>= 2;
    }
    return res;
}
```

**逐行解析：**

这是经典的"逐位"（digit-by-digit）整数平方根算法，专为无 FPU 或想避免 `libm` 的场景设计。

- `res = 0`：累积结果。
- `bit = 1UL << 30`：从最高可能的"两位组"（bit-pair）开始。平方根每产生一位二进制，对应被开方数的两位，故用 `>> 2` 步进。
- `while (bit > x) bit >>= 2;`：把 `bit` 降到不超过 `x` 的最高两位组，避免无谓迭代。
- 主循环：若 `x >= res + bit`，说明该位可为 1，于是 `x -= res + bit`，并以 `(res >> 1) + bit` 更新结果；否则该位为 0，仅 `res >>= 1`。
- `bit >>= 2`：每次处理两位。
- 返回 `res` 即整数平方根（向下取整）。

该函数让 Payload 在不链接浮点库的前提下获得高性能开方。

#### 5.2.3 ABI 跳转表定义

```c
// ABI 跳转表定义
static const openc6_abi_t bios_abi = {
    .magic = OPENC6_ABI_MAGIC,
    .version = OPENC6_ABI_VERSION,
    .sys_reset = esp_restart,
    .set_led_color = led_mgmt_set_color,
    .delay_ms = bios_delay_ms,
    .malloc = bios_malloc,
    .free = free,
    .print = bios_print,
    .get_random = esp_random,
    .sha256 = bios_sha256,
    .math_isqrt = bios_math_isqrt,
    .math_sin_deg = bios_math_sin_deg,
    .math_cos_deg = bios_math_cos_deg,
    .wifi_connect = bios_wifi_connect,
    .wifi_start_ap = bios_wifi_start_ap,
    .wifi_is_connected = bios_wifi_is_connected,
    .get_free_ram = bios_get_free_ram,
    .get_total_ram = bios_get_total_ram,
    .get_total_flash = bios_get_total_flash,
    .fs_write_file = abi_fs_write_file,
    .fs_read_file = abi_fs_read_file,
    .fs_delete = abi_fs_delete
};
```

**逐段解析：**

- 该结构体是整个"宿主—载荷"契约的物理载体。`static const` 表示它在编译期完全确定、存放于只读数据段，运行时不可被篡改。
- `magic` / `version`：完整性校验字段。Payload 在入口处必须校验 `magic == OPENC6_ABI_MAGIC` 且 `version == OPENC6_ABI_VERSION`，否则拒绝执行，防止版本不匹配导致调用错位函数指针而崩溃。
- 系统类：`sys_reset` 直接绑到 `esp_restart`；`delay_ms` 转发到 `vTaskDelay` 以让出 CPU（避免触发看门狗）；`malloc/free` 复用宿主堆，使 Payload 共享同一内存分配器。
- 加密/随机：`get_random = esp_random` 使用硬件 TRNG；`sha256` 走 PSA 硬件加速。
- 数学：三个定点函数，专为无 FPU 系统优化。
- Wi-Fi：三个网络接口，把复杂的 Wi-Fi 栈封装为 Payload 可直接调用的同步 API。
- 遥测：`get_free_ram/get_total_ram/get_total_flash` 让 Payload 感知可用资源。
- 文件系统：`fs_write_file/fs_read_file/fs_delete` 把自研文件系统暴露给 Payload，实现持久化。

关键设计点：Payload 通过函数指针调用这些能力，而非静态链接——这意味着 ABI 可以在不重新编译 Payload 的前提下，由 BIOS 升级带来新能力（只要版本号兼容）。

### 5.3 bios_init.c —— bios_core_start() 启动流程

```c
void bios_core_start(void)
{
    esp_reset_reason_t rst_reason = esp_reset_reason();
    bool is_cold_boot = (rst_reason != ESP_RST_DEEPSLEEP);
    
    // 获取管理引擎启动原因
    uint32_t me_reason = management_engine_get_boot_reason();
    management_engine_clear_boot_reason();

    // 初始化各子系统
    nvram_init();
    led_mgmt_init();
    wifi_mgmt_init();
    hal_flash_init();
    fs_init();

    // 初始化管理引擎
    management_engine_init(is_me_enabled, is_cold_boot);
    if (is_me_enabled) {
        ulp_me_hp_is_awake = 1;
        xTaskCreate(me_watchdog_pet_task, "me_wdt", 2048, NULL, 5, NULL);
    }
    ...
}
```

**逐段解析：**

- `esp_reset_reason()`：读取本次复位的原因（上电、看门狗、深度睡眠唤醒等）。
- `is_cold_boot = (rst_reason != ESP_RST_DEEPSLEEP)`：只要不是从深度睡眠唤醒，就视为冷启动。冷启动意味着 LP RAM 中是垃圾数据，需要重新部署 LP-Core 固件；热唤醒则 LP-Core 仍在运行。
- `management_engine_get_boot_reason()`：从 LP 共享 RAM 读取 ME 写入的启动原因（如按键短按、3s、5s、看门狗超时、热关机）。注意源码中该函数会做有效性校验——冷启动时 LP RAM 是随机值，必须过滤掉"垃圾原因"。
- `management_engine_clear_boot_reason()`：读取后立即清零，防止下次启动重复消费。
- 子系统初始化顺序：`nvram_init` → `led_mgmt_init` → `wifi_mgmt_init` → `hal_flash_init` → `fs_init`。先配置存储（NVRAM），再点灯（便于诊断），再网络，最后 Flash HAL 与文件系统。这个顺序保证后续 Clear CMOS、OTA、PXE 都有所需依赖。
- `management_engine_init(is_me_enabled, is_cold_boot)`：若 ME 启用且为冷启动，则把 LP-Core 固件二进制加载到 LP RAM 并启动协处理器。
- `ulp_me_hp_is_awake = 1`：通过共享变量通知 LP-Core"主核已醒来"，于是 LP-Core 开始对主核执行软件看门狗监控。
- `xTaskCreate(me_watchdog_pet_task, ...)`：创建喂狗任务，HP 核每 2 秒递增 `ulp_me_wdt_counter`；LP-Core 若 15 秒内看不到计数变化，则判定主核卡死并触发硬件复位。

#### 启动原因分析矩阵与状态机路由（节选）

```c
    // 启动原因分析矩阵
    if (me_reason == ME_BOOT_REASON_THERMAL) {
        bios_enter_s5_state();  // 热关机
    }
    else if (me_reason == ME_BOOT_REASON_FORCE_RESET) {
        bios_enter_s5_state();  // 强制关机
    }
    else if (me_reason == ME_BOOT_REASON_WDT) {
        direct_flash_boot = true;  // 看门狗恢复启动
    }
    else if (me_reason == ME_BOOT_REASON_SETUP) {
        enter_setup = true;  // 进入BIOS设置
    }
    else if (me_reason == ME_BOOT_REASON_NORMAL) {
        direct_flash_boot = true;  // 正常启动
    }
```

**逐段解析：** 这是一个优先级分发的决策矩阵。

- `ME_BOOT_REASON_THERMAL`：温度达到紧急阈值，直接进入 S5 软关机，保护硬件。
- `ME_BOOT_REASON_FORCE_RESET`：用户长按 5s，意图关机而非重启，进入 S5。
- `ME_BOOT_REASON_WDT`：主核上次卡死被看门狗复位，本次走恢复启动（直接从 Flash 启动 Payload，跳过交互菜单以免再次卡在交互环节）。
- `ME_BOOT_REASON_SETUP`：用户长按 3s，进入 Web 配置界面。
- `ME_BOOT_REASON_NORMAL`：短按，正常启动。

源码中真实实现还包含冷启动时的 `dc_loss_action`（掉电恢复策略：上电自启或保持关机）以及 BOOT 键长按 1s 触发的交互菜单。随后进入 `execute_boot` 标签的状态机路由：A) 串口载入到 SRAM；B) 串口载入到 Flash（UNIX Shell）；C) PXE 无线网络启动；D) XIP 本地启动；E) BIOS 设置界面（AP Web 配置器）。

#### XIP 启动核心代码

```c
    // XIP启动核心代码：
    if (direct_flash_boot) {
        int16_t file_id = fs_find_id("payload.bin", boot_dir_id);
        if (file_id >= 0) {
            void *payload_ram_ptr = heap_caps_malloc(file_size, MALLOC_CAP_EXEC | MALLOC_CAP_INTERNAL);
            if (payload_ram_ptr != NULL) {
                fs_read_file(file_id, payload_ram_ptr, 0, file_size);
                // 跳转到Payload，传入ABI
                typedef void (*payload_entry_t)(const openc6_abi_t *) __attribute__((noreturn));
                payload_entry_t launch_payload = (payload_entry_t)payload_ram_ptr;
                asm volatile ("fence.i");
                launch_payload(&bios_abi);
            } else {
                // RAM满，XIP回退
                esp_partition_mmap(xip_part, 0, file_size, ESP_PARTITION_MMAP_INST, &mapped_ptr, &mmap_handle);
                launch_payload(&bios_abi);
            }
        }
    }
```

**逐行解析：**

- `fs_find_id("payload.bin", boot_dir_id)`：在文件系统的 `/downloaded/` 目录下查找 `payload.bin`，返回文件 ID（找不到为负）。
- `heap_caps_malloc(file_size, MALLOC_CAP_EXEC | MALLOC_CAP_INTERNAL)`：这是关键——分配的内存必须同时满足"可执行"（`MALLOC_CAP_EXEC`，映射到指令总线）和"内部"（`MALLOC_CAP_INTERNAL`）。ESP32-C6 的内部 SRAM 可配置为可执行区域，这样才能把代码拷进去并跳转执行。
- `fs_read_file(...)`：把整个 Payload 从文件系统读入可执行 RAM。
- `typedef void (*payload_entry_t)(const openc6_abi_t *)`：定义函数指针类型，签名与 Payload 入口 `payload_main(const openc6_abi_t *abi)` 完全一致；`__attribute__((noreturn))` 告诉编译器该函数不会返回，避免生成无用的返回清理代码。
- `asm volatile ("fence.i")`：RISC-V 的指令缓存屏障指令。刚拷贝进 RAM 的代码可能还在旧的 I-Cache 里，`fence.i` 强制刷新指令缓存，确保 CPU 取到的是新写入的指令而非陈旧数据。这是"代码即数据"场景下必须的手续。
- `launch_payload(&bios_abi)`：把 ABI 跳转表的地址作为第一个参数（`a0` 寄存器）传给 Payload，然后跳转。控制权从此移交给 Payload。
- `else` 分支（RAM 不足回退）：当可执行 RAM 装不下时，回退到真正的 XIP——用 `esp_partition_mmap` 把 `payload_xip` 分区以指令映射方式映射到地址空间，直接在 Flash 上就地执行，无需拷贝到 RAM。源码中真实流程是先把文件从 `openc6_fs` 分块拷贝写入 `payload_xip` 分区，再 mmap 执行。

这种"RAM 优先、XIP 兜底"的双策略，兼顾了运行速度（RAM 更快）与容量限制（XIP 不占 RAM）。

### 5.4 openc6_abi.h —— ABI 系统调用接口定义

```c
#define OPENC6_ABI_MAGIC   0x43364249  // "C6BI"
#define OPENC6_ABI_VERSION 1

typedef struct {
    uint32_t magic;
    uint32_t version;

    // 系统API
    void (*sys_reset)(void) __attribute__((noreturn));
    void (*set_led_color)(uint8_t r, uint8_t g, uint8_t b);
    void (*delay_ms)(uint32_t ms);
    void* (*malloc)(uint32_t size);
    void (*free)(void* ptr);
    void (*print)(const char *str);
    uint32_t (*get_random)(void);
    void (*sha256)(const uint8_t *input, uint32_t len, uint8_t *output);

    // 整数数学函数（无FPU系统优化）

    int32_t (*math_cos_deg)(int32_t angle_deg);

    // Wi-Fi 网络接口
    int32_t (*wifi_connect)(const char* ssid, const char* pass);
    int32_t (*wifi_start_ap)(const char* ssid, const char* pass);
    int32_t (*wifi_is_connected)(void);

    // 系统遥测
    uint32_t (*get_free_ram)(void);
    uint32_t (*get_total_ram)(void);
    uint32_t (*get_total_flash)(void);

    // 文件系统接口
    void (*fs_write_file)(const char *name, const uint8_t *data, uint32_t len, uint32_t dir_sector, uint8_t force);
    int32_t (*fs_read_file)(const char *name, uint8_t *dest, uint32_t offset, uint32_t len, uint32_t dir_sector);
    void (*fs_delete)(const char *name, uint32_t dir_sector);
} openc6_abi_t;
```

**逐段解析：**

- `OPENC6_ABI_MAGIC = 0x43364249`：ASCII 即 `"C6BI"`（C6 BIOS Interface）。这是一个完整性标记，Payload 入口必须验证它，确保拿到的指针确实是 OpenC6 ABI 而非随机内存。
- `OPENC6_ABI_VERSION = 1`：版本号。当 ABI 增删字段时递增，Payload 据此判断兼容性。这是"稳定契约 + 演进能力"的平衡手段。
- 结构体本质是一张**函数指针表**。`magic` 与 `version` 之后全部是函数指针，按"系统—数学—网络—遥测—文件系统"分组。
- `sys_reset` 标注 `noreturn`：调用即复位，调用方无需考虑返回后状态。
- 数学函数的注释"输出 x10000 定点"揭示了无 FPU 优化策略：`math_sin_deg(90)` 返回 `10000` 表示 `1.0000`，`math_sin_deg(30)` 返回 `5000` 表示 `0.5000`。定点运算避免浮点库依赖，且在整数流水线上更快。
- `fs_read_file` 返回 `int32_t`（实际读取字节数或 -1），`fs_write_file` 返回 void——这是有意简化：写操作通过文件名定位，失败由调用方后续读取校验。
- 整张表就是 Payload 与 BIOS 之间的"系统调用号 + 实现"。与传统 OS 的 `syscall` 编号不同，这里直接传函数指针，省去查表开销，且对 `-nostdlib` 的裸机代码极其友好。

### 5.5 management_engine.c —— HP Core 管理引擎接口

```c
void management_engine_init(bool is_enabled, bool is_cold_boot) {
    if (!is_enabled) {
        ESP_LOGW(TAG, "Management Engine is DISABLED. System running in Legacy Mode.");
        return;
    }

    if (!is_cold_boot) {
        // 热唤醒：LP Core 已在运行，跳过重新部署
        ESP_LOGI(TAG, "ME already running (wakeup path). Skipping LP-Core re-deploy.");
        me_active = true;
        return;
    }

    ESP_LOGI(TAG, "Deploying OpenC6 Management Engine to LP-Core...");

    // 加载LP Core固件二进制
    esp_err_t err = ulp_lp_core_load_binary(me_core_bin_start,
                                            (me_core_bin_end - me_core_bin_start));
    if (err != ESP_OK) {
        ESP_LOGE(TAG, "Failed to load ME binary: %s", esp_err_to_name(err));
        return;
    }

    // 初始化LP RAM参数
    ulp_me_is_enabled  = 1;
    ulp_me_boot_reason = ME_BOOT_REASON_NONE;

    ulp_lp_core_cfg_t cfg = { .wakeup_source = ULP_LP_CORE_WAKEUP_SOURCE_HP_CPU };
    ESP_ERROR_CHECK(ulp_lp_core_run(&cfg));

    me_active = true;
    ESP_LOGI(TAG, "ME is Active. Hardware WDT & Button logic delegated to LP-Core.");
}

// HP Core每2秒喂狗一次，LP Core有15秒超时
void management_engine_pet_watchdog(void) {
    if (!me_active) return;
    ulp_me_wdt_counter++;
}
```

**逐段解析：**

- `is_enabled` 为假时：仅打印警告并返回，系统进入"传统模式"（Legacy Mode），按键与看门狗改由 HP 核用 GPIO 唤醒与 ext1 中断实现（见 `bios_enter_s5_state` 的 Legacy 分支）。
- `!is_cold_boot` 热唤醒分支：从深度睡眠醒来时 LP-Core 仍在运行、LP RAM 仍有效，无需重新加载二进制，直接置 `me_active = true` 复用。这避免了热唤醒时无谓的固件重部署开销。
- `ulp_lp_core_load_binary(me_core_bin_start, size)`：把链接器符号 `_binary_me_core_bin_start` 到 `_binary_me_core_bin_end` 之间的 LP-Core 固件二进制加载到 LP RAM。这两个符号由 ESP-IDF 的二进制嵌入机制生成。
- `ulp_me_is_enabled = 1` / `ulp_me_boot_reason = ME_BOOT_REASON_NONE`：初始化共享变量。注意源码注释强调必须在加载二进制之后才写这些值，避免被加载过程覆盖。
- `ulp_lp_core_cfg_t cfg = { .wakeup_source = ULP_LP_CORE_WAKEUP_SOURCE_HP_CPU }`：配置 LP-Core 由 HP 核唤醒启动。
- `ESP_ERROR_CHECK(ulp_lp_core_run(&cfg))`：启动 LP-Core，失败则断言 panic。
- `management_engine_pet_watchdog`：HP 核每 2 秒（由 `me_watchdog_pet_task` 的 `vTaskDelay(pdMS_TO_TICKS(2000))` 决定）调用一次，递增 `ulp_me_wdt_counter`。LP-Core 侧若 15 秒（`ME_WDT_TIMEOUT_MS`）内看不到计数变化，则判定主核卡死并硬件复位。

这就是"带外管理"的核心：即使 HP 核的 FreeRTOS 完全死锁，LP-Core 仍能独立检测并复位系统。

### 5.6 lp_core_main.c —— LP Core 管理引擎固件

#### 5.6.1 寄存器与共享内存

```c
// LP WDT寄存器定义
#define LP_WDT_BASE         0x600B1C00UL
#define LP_WDT_CONFIG0      (*(volatile uint32_t *)(LP_WDT_BASE + 0x00))
#define LP_WDT_CONFIG1      (*(volatile uint32_t *)(LP_WDT_BASE + 0x04))
#define LP_WDT_FEED         (*(volatile uint32_t *)(LP_WDT_BASE + 0x14))
#define LP_WDT_WPROTECT     (*(volatile uint32_t *)(LP_WDT_BASE + 0x18))
#define LP_WDT_WRITE_KEY    0x50D83AA1UL
#define ME_WDT_TIMEOUT_MS   15000

// IPC共享内存变量
volatile uint32_t me_is_enabled  = 1;
volatile uint32_t me_hp_is_awake = 0;
volatile uint32_t me_boot_reason = ME_BOOT_REASON_NONE;
volatile uint32_t me_wdt_counter = 0;
volatile uint32_t me_cpu_load       = 0;
volatile uint32_t me_target_freq    = 80;
volatile uint32_t me_temperature    = 0;
volatile uint32_t me_throttle_temp  = 55;
volatile uint32_t me_emergency_temp = 75;
```

**逐段解析：**

- LP-Core 直接操作底层寄存器：`LP_WDT_BASE = 0x600B1C00` 是 ESP32-C6 的 LP 看门狗（RTC WDT）寄存器基址。`CONFIG0/CONFIG1/FEED/WPROTECT` 分别是配置、超时计数、喂狗、写保护寄存器。
- `LP_WDT_WRITE_KEY = 0x50D83AA1`：写保护密钥，必须先写入 `WPROTECT` 才能修改看门狗配置，防止误写。
- `ME_WDT_TIMEOUT_MS = 15000`：主核 15 秒不喂狗即复位。
- 共享变量全部 `volatile`：因为它们被双核并发访问，必须禁用编译器缓存优化，确保每次读写都直达内存。这些变量位于 LP RAM 的共享段，是 HP 与 LP 之间的唯一 IPC 通道。
- `me_throttle_temp/me_emergency_temp` 默认 55/75℃，但运行时由 `power_governor_start` 从 NVRAM 读取后广播覆盖（动态配置）。

#### 5.6.2 硬件复位传播

```c
// 硬件复位传播
static void me_hw_reset(uint32_t reason) {
    me_boot_reason = reason;
    ulp_lp_core_delay_us(1000);

    LP_WDT_WPROTECT = LP_WDT_WRITE_KEY;
    LP_WDT_FEED = (1UL << 31);
    LP_WDT_CONFIG1 = LP_WDT_RESET_TICKS;

    uint32_t cfg = LP_WDT_CONFIG0;
    cfg &= ~LP_WDT_STG0_MASK;
    cfg |= LP_WDT_STG0_RTC_RST;
    cfg |= LP_WDT_EN;
    LP_WDT_CONFIG0 = cfg;

    LP_WDT_WPROTECT = 0;
    while (1) { __asm__ volatile("nop"); }
}
```

**逐行解析：**

- 先把复位原因写入 `me_boot_reason`，让 HP 核复位后能读到"我是被谁复位的"。
- `ulp_lp_core_delay_us(1000)`：等 1ms，确保共享变量写入对 HP 核可见（内存一致性）。
- `LP_WDT_WPROTECT = LP_WDT_WRITE_KEY`：解除写保护。
- `LP_WDT_FEED = (1UL << 31)`：喂一次狗（重置计数）。
- `LP_WDT_CONFIG1 = LP_WDT_RESET_TICKS`：设置超时计数值（100 ticks，极短）。
- 修改 `CONFIG0`：清除 stage0 的动作掩码，设为 `LP_WDT_STG0_RTC_RST`（触发 RTC 复位），并置 `LP_WDT_EN` 使能。这样看门狗几乎立即超时并复位整个芯片。
- `LP_WDT_WPROTECT = 0`：重新加锁。
- `while (1) { nop }`：死等复位到来。函数不会返回。

这一机制让 LP-Core 拥有对整机的"硬复位权"，是实现带外看门狗的关键。

#### 5.6.3 主循环：按键、看门狗、热保护

```c
int main(void) {
    // 初始化电源按钮GPIO
    ulp_lp_core_gpio_init(PIN_BTN_GND);
    ulp_lp_core_gpio_output_enable(PIN_BTN_GND);
    ulp_lp_core_gpio_set_level(PIN_BTN_GND, 0);

    ulp_lp_core_gpio_init(PIN_BTN_SENSE);
    ulp_lp_core_gpio_input_enable(PIN_BTN_SENSE);
    ulp_lp_core_gpio_pullup_enable(PIN_BTN_SENSE);

    uint32_t pwr_press_timer  = 0;
    uint32_t wdt_last_counter = 0;
    uint32_t wdt_stale_ms     = 0;
    uint32_t sched_loop_timer = 0;
    uint32_t me_uptime_ms     = 0;

    while (1) {
        if (me_is_enabled == 0) {
            ulp_lp_core_delay_us(100000);
            continue;
        }

        me_uptime_ms += 10;

        // 1. 智能电源按钮管理
        if (ulp_lp_core_gpio_get_level(PIN_BTN_SENSE) == 0) {
            if (pwr_press_timer < 10000) pwr_press_timer += 10;
            if (me_hp_is_awake == 1 && pwr_press_timer >= 5000) {
                me_hw_reset(ME_BOOT_REASON_FORCE_RESET);  // 5秒硬重置
            }
        } else {
            if (pwr_press_timer > 0) {
                if (me_hp_is_awake == 0) {
                    if (pwr_press_timer >= 3000) {
                        me_boot_reason = ME_BOOT_REASON_SETUP;  // 3秒进设置
                        ulp_lp_core_wakeup_main_processor();
                    } else if (pwr_press_timer >= 50) {
                        me_boot_reason = ME_BOOT_REASON_NORMAL;  // 短按正常启动
                        ulp_lp_core_wakeup_main_processor();
                    }
                }
                pwr_press_timer = 0;
            }
        }

        // 2. 主处理器软件看门狗解码器
        if (me_hp_is_awake == 1) {
            if (me_wdt_counter != wdt_last_counter) {
                wdt_last_counter = me_wdt_counter;
                wdt_stale_ms = 0;
            } else {
                wdt_stale_ms += 10;
                if (wdt_stale_ms >= ME_WDT_TIMEOUT_MS) {
                    me_hw_reset(ME_BOOT_REASON_WDT);  // 15秒看门狗超时
                }
            }
        }

        // 3. SchedUtil 频率缩放与热保护
        sched_loop_timer += 10;
        if (sched_loop_timer >= 100) {
            sched_loop_timer = 0;

            // 紧急热关机
            if (me_uptime_ms > 3000 && me_temperature >= me_emergency_temp && me_hp_is_awake == 1) {
                me_hw_reset(ME_BOOT_REASON_THERMAL);
            }

            // 动态频率调节
            uint32_t target = 80;
            if (me_temperature >= me_throttle_temp) {
                target = 80;  // 热降频
            } else {
                if (me_cpu_load > 75) target = 160;
                else if (me_cpu_load > 30) target = 120;
                else target = 80;
            }
            if (target > me_max_freq) target = me_max_freq;
            me_target_freq = target;
        }

        ulp_lp_core_delay_us(10000);
    }
    return 0;
}
```

**逐段解析：**

- GPIO 初始化：`PIN_BTN_GND`（GPIO3）设为输出低电平，充当"软件地"；`PIN_BTN_SENSE`（GPIO4）设为输入并启用内部上拉。按键按下时 Sense 被拉低。
- `me_is_enabled == 0` 时空转 100ms，节能且不干扰。
- `me_uptime_ms += 10`：每轮迭代约 10ms（由末尾 `delay_us(10000)` 决定），用作时间基准。
- **按键管理**：Sense 为低（按下）时累加 `pwr_press_timer`（上限 10000ms）。若主核已醒且持续 ≥5000ms，立即 `me_hw_reset(FORCE_RESET)`——这是"长按 5s 强制复位/关机"。松开时（Sense 回高）：若主核在睡（`me_hp_is_awake == 0`），则按持续时长判定——≥3000ms 写 `SETUP` 唤醒进设置，≥50ms 写 `NORMAL` 唤醒正常启动。注意只有主核在睡眠时才由按键唤醒它；主核已醒时的短按不触发启动（避免误唤醒）。
- **看门狗解码器**：主核醒时，比较 `me_wdt_counter` 与本地缓存 `wdt_last_counter`。若变化则喂狗成功、清零计时；若 15 秒无变化则 `me_hw_reset(WDT)`。主核在睡时则不监控（`else` 分支同步基准、不计数）。
- **SchedUtil 与热保护**（每 100ms 执行一次）：先做紧急热关机判断——启动超过 3 秒后若温度 ≥ 紧急阈值且主核醒，则 `me_hw_reset(THERMAL)`。`me_uptime_ms > 3000` 是为避免上电瞬间传感器读数不稳误触发。随后计算目标频率：温度达 throttle 阈值强制 80MHz 降频；否则按负载分档（>75→160, >30→120, 否则 80），再用 `me_max_freq`（来自 NVRAM 的用户上限）封顶。结果写入 `me_target_freq` 供 HP 核的 governor 读取应用。
- 末尾 `delay_us(10000)`：10ms 节拍，平衡响应灵敏度与功耗。

这份 LP-Core 固件把"按键去抖 + 时长判定 + 软件看门狗 + 热保护 + 频率策略"全部下沉到协处理器，是整个平台"带外管理"能力的灵魂。

### 5.7 power_governor.c —— SchedUtil 动态调频任务

```c
static void openc6_schedutil_task(void *arg) {
    uint32_t last_idle_time = 0;
    uint32_t last_total_time = 0;
    uint32_t current_applied_freq = 0;

    cpu_freq_t max_bios_freq;
    nvram_get_cpu_freq(&max_bios_freq);

    while (1) {
        TaskStatus_t idle_stats;
        vTaskGetInfo(xTaskGetIdleTaskHandle(), &idle_stats, pdFALSE, eInvalid);

        uint32_t now = esp_timer_get_time();
        uint32_t idle_now = idle_stats.ulRunTimeCounter;

        if (last_total_time > 0) {
            uint32_t delta_total = now - last_total_time;
            uint32_t delta_idle = idle_now - last_idle_time;

            // 计算CPU负载 0-100%
            uint32_t load = 0;
            if (delta_total > delta_idle) {
                load = 100 - ((delta_idle * 100) / delta_total);
            }

            uint32_t target_freq = 80;

            if (is_me_enabled) {
                // 管理引擎模式：委托给LP协处理器
                ulp_me_cpu_load = load;
                target_freq = ulp_me_target_freq;
            } else {
                // 传统模式：本地计算
                if (load > 75) target_freq = 160;
                else if (load > 30) target_freq = 120;
                else target = 80;
                if (target_freq > (uint32_t)max_bios_freq) {
                    target_freq = (uint32_t)max_bios_freq;
                }
            }

            // 仅在频率变化时应用
            if (target_freq != current_applied_freq) {
                esp_pm_config_t pm_cfg = {
                    .max_freq_mhz = target_freq,
                    .min_freq_mhz = target_freq,
                    .light_sleep_enable = false
                };
                esp_pm_configure(&pm_cfg);
                current_applied_freq = target_freq;
                ESP_LOGI(TAG, "Load: %lu%% | Temp: %lu C | New Freq: %lu MHz", load, ulp_me_temperature, target_freq);
            }
        }

        last_total_time = now;
        last_idle_time = idle_now;
        vTaskDelay(pdMS_TO_TICKS(100));  // 100ms重计算
    }
}
```

**逐段解析：**

- 任务维护两个基准：`last_total_time`（上次采样的微秒时间戳）与 `last_idle_time`（上次空闲任务的运行时计数）。
- `vTaskGetInfo(xTaskGetIdleTaskHandle(), ...)`：获取 FreeRTOS 空闲任务的统计信息。`ulRunTimeCounter` 是该任务累计占用的"运行时 tick"（需开启 `CONFIG_FREERTOS_GENERATE_RUN_TIME_STATS`，sdkconfig.defaults 中已启用）。
- `now = esp_timer_get_time()`：当前微秒时间。
- 负载计算：`delta_total` 为这段时间总时长，`delta_idle` 为空闲任务占用的运行时。`load = 100 - (delta_idle*100/delta_total)`——空闲占比越低，负载越高。这是类 Linux `cpufreq` 的 ondemand 思路。
- **双模式**：ME 启用时，把负载写入 `ulp_me_cpu_load` 供 LP-Core 决策，并直接采纳 LP-Core 算出的 `ulp_me_target_freq`（已含热降频与上限封顶）；ME 禁用时本地按负载分档，再用 NVRAM 的 `max_bios_freq` 封顶。
- **去抖应用**：仅当 `target_freq != current_applied_freq` 时才调用 `esp_pm_configure`，避免频繁切换 PMU 造成抖动开销。`max == min` 表示锁定频率、不禁 light sleep。
- `vTaskDelay(pdMS_TO_TICKS(100))`：100ms 重算一次。

源码中还有 `openc6_temperature_task`：每 2 秒读取片内温度传感器（带量程回退）写入 `ulp_me_temperature`，以及 `power_governor_start` 把 NVRAM 的频率上限与热限广播到 LP 共享内存。三者协同构成完整的"感知—决策—执行"调频链路。

### 5.8 openc6_fs.c —— 日志结构循环文件系统

#### 5.8.1 RAM 索引与结构

```c
#define MAX_FILES_LIMIT 1024
#define FS_MAGIC_SKIP 0xEEEE

// RAM索引节点
typedef struct {
    uint16_t id;
    uint16_t parent_id;
    uint8_t  type;
    char     name[18];
    uint32_t size;
} RamNode;

static RamNode *g_nodes = NULL;
static uint32_t g_head = 0;  // 写入头
static uint32_t g_tail = 0;  // 回收尾
```

**逐段解析：**

- `MAX_FILES_LIMIT 1024`：最多 1024 个文件/目录。
- `FS_MAGIC_SKIP 0xEEEE`：扇区内"跳过填充"记录的魔数。日志结构 FS 中，当一条记录写不下当前扇区剩余空间时，用 skip 记录填充剩余字节并跳到下一扇区。
- `RamNode`：RAM 中的索引节点，记录 id、父目录 id、类型（文件/目录）、名称、大小。文件内容仍在 Flash，RAM 只存元数据，加速查找。
- `g_head`：循环缓冲区的写入头（append-only）。`g_tail`：回收尾（垃圾回收推进位置）。两者构成一个环形，体现"循环"特性。

#### 5.8.2 写文件（分块写入）

```c
// 写文件（分块写入）
int16_t fs_write_file(const char *name, const uint8_t *data, uint32_t len, uint16_t parent_id) {
    int16_t id = fs_find_id(name, parent_id);
    if (id >= 0) {
        fs_delete(id);  // 已存在则先删除
    } else {
        id = g_next_id++;
    }

    uint32_t offset = 0;
    uint16_t c_idx = 0;
    uint32_t max_payload = SECTOR_SIZE - sizeof(RecordHeader);

    while (offset < len || len == 0) {
        uint32_t chunk_len = len - offset;
        if (chunk_len > max_payload) chunk_len = max_payload;

        uint32_t padded_len = (sizeof(RecordHeader) + chunk_len + 3) & ~3;
        
        check_gc(padded_len);  // 垃圾回收检查

        RecordHeader hdr = {0};
        hdr.magic = FS_MAGIC;
        hdr.total_len = padded_len;
        hdr.data_len = chunk_len;
        hdr.id = id;
        hdr.parent_id = parent_id;
        hdr.state = 1;
        hdr.type = TYPE_FILE;
        hdr.chunk_idx = c_idx;
        strncpy(hdr.name, name, 17);

        flash_write(g_head, (uint8_t*)&hdr, sizeof(hdr));
        if (chunk_len > 0) flash_write(g_head + sizeof(hdr), data + offset, chunk_len);

        g_head = (g_head + padded_len) % hal_flash_get_size();
        offset += chunk_len;
        c_idx++;
        if (len == 0) break;
    }

    update_ram_index(id, parent_id, TYPE_FILE, name, len);
    return id;
}
```

**逐行解析：**

- 先查同名文件：存在则 `fs_delete(id)` 标记旧记录为失效（日志结构 FS 不原地改写，只追加新版本），不存在则分配新 id。
- `max_payload = SECTOR_SIZE - sizeof(RecordHeader)`：单分块最大有效数据（扣除头）。
- 主循环把数据切成多个 chunk：每个 chunk 不超过 `max_payload`，从而支持大于单扇区的文件（这是"分块式"核心）。
- `padded_len = (sizeof(RecordHeader) + chunk_len + 3) & ~3`：4 字节对齐填充，符合 Flash 写入对齐要求。
- `check_gc(padded_len)`：写入前检查剩余空间是否足够（含 3 个扇区余量），不足则触发垃圾回收。
- 构造 `RecordHeader`：`magic` 标识合法记录；`total_len` 对齐后总长；`data_len` 实际数据长；`id/parent_id` 标识归属；`state=1` 表示存活；`chunk_idx` 为该文件的第几块（重组时按序拼接）；`name` 内嵌文件名。
- `flash_write` 先写头、再写数据。
- `g_head = (g_head + padded_len) % size`：环形推进写入头。
- `offset += chunk_len; c_idx++`：推进数据偏移与块序号。`len == 0` 时写一个空块后 break（支持空文件）。
- `update_ram_index`：更新 RAM 索引，记录该文件的总大小。
- 返回文件 id。

这种"只追加、不覆写"的设计天然支持磨损均衡——每个扇区被均匀写入，且断电时旧版本仍可恢复（日志回放）。

#### 5.8.3 垃圾回收

```c
// 垃圾回收
static void gc_step(void) {
    uint32_t sector_start = (g_tail / SECTOR_SIZE) * SECTOR_SIZE;
    uint32_t curr = sector_start;

    while (curr < sector_start + SECTOR_SIZE) {
        // 遍历扇区中的记录
        RecordHeader hdr;
        flash_read(curr, (uint8_t*)&hdr, sizeof(hdr));

        if (hdr.state == 1 && find_idx_by_id(hdr.id) >= 0) {
            // 存活记录：搬迁到头部
            uint8_t *buf = malloc(hdr.data_len);
            flash_read(curr + sizeof(RecordHeader), buf, hdr.data_len);
            append_chunk_raw(&hdr, buf);  // 追加到新位置
            free(buf);
        }
        curr += tlen;
    }
    flash_erase_sector(sector_start / SECTOR_SIZE);  // 擦除旧扇区
    g_tail = (sector_start + SECTOR_SIZE) % hal_flash_get_size();
}
```

**逐段解析：**

- 从 `g_tail` 所在扇区开始回收。回收以整扇区为单位（Flash 只能按扇区擦除）。
- 遍历扇区内每条记录：若 `state == 1` 且其 id 仍在 RAM 索引中（即仍存活），则把数据读出、`append_chunk_raw` 重新追加到写入头部（搬移到新位置）。
- 全部存活记录搬完后，`flash_erase_sector` 擦除整个旧扇区。
- `g_tail` 推进一个扇区，环形继续。

这就是经典的"复制式垃圾回收"在 Flash 文件系统上的应用：把存活对象搬走、整片擦除死亡对象，既回收空间又实现磨损均衡。`check_gc` 保证写入前总有足够余量，避免回收途中无空间可用。

### 5.9 payload_example.c —— 裸机 Payload 示例

```c
#include "openc6_abi.h"

// 自定义printf（-nostdlib模式）
static void print_uint32(const openc6_abi_t *abi, uint32_t val) {
    char buf[16];
    int i = 14;
    buf[15] = '\0';
    if (val == 0) {
        buf[i--] = '0';
    } else {
        while (val > 0 && i >= 0) {
            buf[i--] = '0' + (val % 10);
            val /= 10;
        }
    }
    abi->print(&buf[i + 1]);
}

// Payload入口点
void __attribute__((section(".text.entry"), noreturn)) payload_main(const openc6_abi_t *abi) {
    // ABI 验证
    if (!abi || abi->magic != OPENC6_ABI_MAGIC || abi->version != OPENC6_ABI_VERSION) {
        while (1) { __asm__ volatile("nop"); }
    }

    // 测试1：内存监控
    uint32_t flash_kb = abi->get_total_flash() / 1024;
    uint32_t ram_free_kb = abi->get_free_ram() / 1024;
    abi->print(" -> Total Flash: "); print_uint32(abi, flash_kb); abi->print(" KB\n");

    // 测试2：动态内存分配
    uint8_t *test_buf = (uint8_t*)abi->malloc(1024);
    if (test_buf) {
        test_buf[0] = 0xAA;
        abi->free(test_buf);
    }

    // 测试3：硬件加密与TRNG
    uint32_t r1 = abi->get_random();
    uint8_t hash[32];
    abi->sha256((const uint8_t*)"OpenC6", 6, hash);

    // 测试4：数学加速
    uint32_t sq_val = abi->math_isqrt(144);
    uint32_t sin_val = (uint32_t)abi->math_sin_deg(90);

    // 测试5：Wi-Fi连接
    int wifi_res = abi->wifi_connect("Test", "12345678");
    if (wifi_res == 0 && abi->wifi_is_connected()) {
        abi->set_led_color(0, 255, 0);  // 绿色
    }

    // 测试6：LED控制
    for (int i = 0; i < 3; i++) {
        abi->set_led_color(255, 0, 0); abi->delay_ms(333);
        abi->set_led_color(0, 255, 0); abi->delay_ms(333);
        abi->set_led_color(0, 0, 255); abi->delay_ms(333);
    }

    abi->delay_ms(3000);
    abi->sys_reset();  // 重启回BIOS
}
```

**逐段解析：**

- `print_uint32`：因为 Payload 以 `-nostdlib` 编译，没有 `printf`，所以手写一个十进制转字符串的微型实现。从缓冲区末尾向前填字符，最后用 `abi->print` 输出。这体现了"裸机"的极致精简。
- 入口 `payload_main`：
  - `__attribute__((section(".text.entry")))`：把入口放到专门的 `.text.entry` 段，链接脚本保证该段位于二进制最前端，使 BIOS 跳转到的首地址正好是入口。
  - `noreturn`：声明不返回，与 BIOS 侧的函数指针类型匹配。
  - 参数 `abi`：BIOS 跳转时通过 `a0` 寄存器传入的 ABI 表指针。
- **ABI 验证**：检查 `abi` 非空、`magic` 与 `version` 匹配。失败则死循环——绝不带着错误的 ABI 继续调用函数指针（否则必崩）。这是防御性编程的典范。
- **测试1 内存监控**：调用 `get_total_flash/get_free_ram`，演示遥测 API。注意全部输出通过 `abi->print`。
- **测试2 动态内存**：`abi->malloc(1024)` 复用宿主堆，写入 `0xAA` 后 `abi->free`——验证宿主堆对 Payload 可用。
- **测试3 加密与 TRNG**：`get_random` 取硬件随机数，`sha256` 对 `"OpenC6"` 求哈希——验证硬件密码引擎可用。
- **测试4 数学**：`math_isqrt(144)` 应得 12，`math_sin_deg(90)` 应得 10000（定点）——验证无 FPU 数学加速。
- **测试5 Wi-Fi**：`wifi_connect` 连接热点，成功且 `wifi_is_connected` 为真时点亮绿色 LED——验证网络栈可用。
- **测试6 LED**：循环三次红绿蓝切换，演示 `set_led_color` 与 `delay_ms`。
- 末尾 `delay_ms(3000)` 后 `sys_reset()`：把控制权交还 BIOS，重启循环。

这个示例同时是一份"ABI 能力清单"与自检程序：它逐一验证了 Payload 可用的所有宿主能力。

### 5.10 nvram_schema.h —— NVRAM 配置架构定义

```c
#define BIOS_NVS_NAMESPACE "bios_cfg"

// 基本配置
typedef enum { DC_LOSS_POWER_OFF = 0, DC_LOSS_POWER_ON = 1 } dc_loss_action_t;
typedef enum { POST_LED_DISABLED = 0, POST_LED_ENABLED = 1 } post_led_mode_t;
typedef enum { AURA_DISABLED = 0, AURA_RAINBOW = 1 } aura_mode_t;

// AI Tweaker: CPU配置
typedef enum {
    CPU_FREQ_80MHZ  = 80,
    CPU_FREQ_160MHZ = 160,
} cpu_freq_t;

typedef enum {
    GOV_PERFORMANCE = 0,  // 持续最高频率
    GOV_DYNAMIC     = 1,  // 动态调频（DVS）
} cpu_governor_t;

// 降压保护
typedef enum {
    BOD_STRICT   = 0,  // 2.8V触发复位
    BOD_RELAXED  = 1,  // 2.5V触发复位
    BOD_DISABLED = 2,  // 禁用
} bod_level_t;

// 热限制
#define VAL_TEMP_THROTTLE_DEFAULT  55
#define VAL_TEMP_EMERGENCY_DEFAULT 75
```

**逐段解析：**

- `BIOS_NVS_NAMESPACE "bios_cfg"`：所有配置存于 NVS 的 `bios_cfg` 命名空间，对应 PC 的 CMOS。
- `dc_loss_action_t`：掉电恢复策略——上电后保持关机或自动开机，对应 PC BIOS 的 "AC Power Recovery"。
- `post_led_mode_t` / `aura_mode_t`：POST LED 开关与 Aura Sync 彩虹模式。
- `cpu_freq_t`：CPU 频率上限 80/160MHz，对应 PC 的外频/倍频设定。
- `cpu_governor_t`：性能模式（持续高频）与动态模式（DVS 调频），对应 Intel SpeedStep / AMD Cool'n'Quiet。
- `bod_level_t`：欠压保护等级——严格(2.8V)/宽松(2.5V)/禁用，对应 PC 的 Brownout 保护。
- `VAL_TEMP_THROTTLE/EMERGENCY_DEFAULT`：默认热降频 55℃、紧急关机 75℃，对应 PC 的 Thermal Throttling。

整套枚举刻意模仿 PC BIOS 的 "AI Tweaker" 超频菜单术语，让熟悉 PC 超频的开发者能无缝上手。

## 六、API 使用指南

OpenC6 的"API"即 `openc6_abi_t` 跳转表。Payload 开发遵循以下流程：

1. **包含头文件**：`#include "openc6_abi.h"`，仅此一个依赖。
2. **实现入口**：`void payload_main(const openc6_abi_t *abi)`，并标注 `__attribute__((section(".text.entry"), noreturn))`。
3. **校验 ABI**：入口首行检查 `abi->magic == OPENC6_ABI_MAGIC` 与 `abi->version == OPENC6_ABI_VERSION`。
4. **调用能力**：通过 `abi->函数名(...)` 调用系统、数学、Wi-Fi、遥测、文件系统接口。
5. **交还控制**：结束时调用 `abi->sys_reset()` 回到 BIOS。

### ABI 接口分类速查

| 分类 | 函数 | 说明 |
| :--- | :--- | :--- |
| 系统 | `sys_reset` / `delay_ms` / `print` | 复位、延时让出 CPU、文本输出 |
| 内存 | `malloc` / `free` | 共享宿主堆 |
| LED | `set_led_color(r,g,b)` | 控制 POST RGB |
| 加密 | `get_random` / `sha256` | 硬件 TRNG 与 SHA-256 |
| 数学 | `math_isqrt` / `math_sin_deg` / `math_cos_deg` | 定点整数数学加速 |
| 网络 | `wifi_connect` / `wifi_start_ap` / `wifi_is_connected` | Wi-Fi STA/AP 与状态 |
| 遥测 | `get_free_ram` / `get_total_ram` / `get_total_flash` | 资源监控 |
| 文件 | `fs_write_file` / `fs_read_file` / `fs_delete` | 持久化读写删 |

### 调用约定

ABI 表指针通过 RISC-V `a0` 寄存器传入；所有函数返回值遵循"0 为成功、负数为失败"的 POSIX 风格（除明确标注返回字节数或布尔值的接口）。Payload 编译时使用 `-nostdlib -fPIC`，体积可压至 2–10 KB。

## 八、项目亮点与适用场景

### 亮点

1. **真正的带外管理**：LP-Core 协处理器独立监控主核，即使主系统死锁仍可复位——这是嵌入式领域罕见的"服务器级 BMC"思路下沉。
2. **ABI 解耦**：Payload 与 BIOS 通过函数指针表契约解耦，Payload 无需 ESP-IDF 即可编译，体积极小，可热替换。
3. **防变砖 OTA**：A/B 分区 + 硬件回滚 + LP-Core 看门狗三重保障，让无线升级真正敢用。
4. **自研日志结构 FS**：分块写入、复制式 GC、磨损均衡，专为 Flash 寿命优化，且支持大于单扇区文件。
5. **PC 化体验**：复古 Web BIOS、AI Tweaker 超频术语、Clear CMOS、POST 诊断码，把 PC 主板的整套交互范式搬上 MCU。
6. **SchedUtil 调频**：基于空闲周期的实时负载感知调频，兼顾性能与功耗。

### 适用场景

- **固件热更新与 A/B 灰度**：需要安全 OTA 且防变砖的量产设备。
- **多载荷切换**：同一硬件需运行多个独立程序（如诊断、生产、出厂测试）并通过网络/串口动态分发。
- **带外监控**：对主控卡死零容忍的工业/物联网网关。
- **教学与研究**：学习 RISC-V 双核 IPC、日志结构文件系统、ABI 设计、动态调频的绝佳实例。
- **极简 Payload 生态**：希望用 2–10 KB 裸机程序复用宿主 Wi-Fi/加密/文件系统能力的场景。

---

## 九、总结

OpenC6 BIOS 并非又一块"ESP32 模板工程"，而是一次把 PC/服务器架构哲学系统性地移植到 RISC-V 微控制器的尝试。它用 **ABI 跳转表** 解耦宿主与载荷、用 **LP-Core 管理引擎** 实现真正的带外监控、用 **A/B OTA + 看门狗回滚** 解决无线升级的信任问题、用 **日志结构循环文件系统** 兼顾持久化与 Flash 寿命、用 **SchedUtil** 做到负载自适应调频。每一处设计都对应着一个真实的工程痛点，且都有真实可读的源码支撑。

从 `app_main` 的一行调用，到 LP-Core 的寄存器级看门狗复位，再到 Payload 的 `-nostdlib` 裸机入口，整条链路自洽而精巧。对于嵌入式开发者而言，它既是一份可用的生产级固件骨架，也是一本关于"如何在 MCU 上构建操作系统式抽象"的活教材。随着 ESP32-P4 移植、zRAM 压缩内存、PXE/串口 SHA-256 校验等 Roadmap 推进，这个平台值得持续关注。
