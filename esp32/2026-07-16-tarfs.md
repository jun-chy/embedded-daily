## 一、项目简介

TARFS 是一个专为嵌入式系统设计的轻量级只读文件系统。它的核心思想极为巧妙：**不发明新的文件系统格式，而是直接在标准 TAR 归档之上提供 POSIX 兼容的文件系统接口**。TAR 归档由 GNU `tar` 工具生成，存储在内存映射的 Flash 中，文件可以通过 `mmap()` 实现真正的零拷贝访问。

固件中包含的大量资源文件——HTML 页面、CSS、JavaScript、图片、证书、字体、字典、查找表、ML 模型——在构建后从不修改。使用可写文件系统存储这些数据不仅浪费，还增加了复杂度。TARFS 正是针对这一痛点设计的解决方案。

### 核心特性一览

| 特性 | 说明 |
|------|------|
| 零拷贝 mmap() | 文件直接通过内存映射访问，不消耗额外 RAM |
| 极低内存开销 | 每个文件/目录仅需 24 字节 RAM，1000 个文件约占 24KB |
| 无动态内存分配 | 运行时零 malloc，仅在 mount/unmount 时分配两次 |
| POSIX 兼容 | 支持 open/read/lseek/mmap/readdir/readlink 等标准接口 |
| CRC64 完整性校验 | 每个文件可独立验证，校验值存储在 TAR 头未使用字段中 |
| 容错能力 | 无中心超级块，损坏部分记录不影响其他文件访问 |
| 二分查找索引 | 基于文件名哈希的二分查找，65535 个文件最多 16 次比较 |
| 锁-free 数据路径 | 除 mount/unmount 外不使用同步原语 |
| 多平台支持 | ESP32/STM32/RP2040/SAMD21/nRF52/LPC17xx 等 |

## 四、核心代码深度分析

### 4.1 文件系统挂载流程（fs.c）

挂载是 TARFS 最复杂的操作，涉及内存映射、TAR 解析、Inode 构建和 VFS 注册。

```c
int tarfs_mount(const char *label, const char *mountpoint,
                const char *link_rebase, const char *path_rebase) {
  int slot;
  size_t size;
  void const *map;
  void *os_handle;

  // 第1步：调用平台层将 Flash 分区映射到地址空间
  log("loading OS-specific resource '%s'..\r\n", label);
  if (NULL != (map = tarfs_os_map_tarfile(label, &os_handle, &size))) {
    log("resource is available at %p, %lu bytes. mounting from memory..\r\n",
        map, size);

    // 第2步：从已映射的内存构建文件系统
    slot = tarfs_mount_memory(map, size, mountpoint,
                              link_rebase, path_rebase);
    if (slot >= 0) {
      log("success, filesystem slot %d was assigned\r\n", slot);
      struct tarfs_fs *fs = tarfs_getfs(slot);
      // 保存 OS 句柄，供 unmount 时释放映射
      fs->fs_handle = (uintptr_t)os_handle;
      return slot;
    }
    // 挂载失败，释放映射
    tarfs_os_unmap_tarfile(os_handle, map, size);
  }

  if (errno == 0)
    errno = EIO;
  logerr("tarfile map failed: errno=%d, label=%s\r\n", errno, label);
  return -1;
}
```

**逐段解析：**

- `tarfs_os_map_tarfile()` 是平台相关函数。在 ESP32 上，它调用 `esp_partition_find()` 查找指定 label 的数据分区，然后用 `esp_partition_mmap()` 将整个分区映射到 CPU 地址空间，返回可直接指针访问的虚拟地址。
- `tarfs_mount_memory()` 是核心挂载逻辑，它接收已映射的内存指针，解析 TAR 归档，构建 Inode 索引。
- `fs_handle` 保存了 OS 层的映射句柄，当文件系统卸载时，`commit_unmount()` 会用它调用 `tarfs_os_unmap_tarfile()` 释放映射。

接下来看 `tarfs_mount_memory()` 的核心逻辑：

```c
int tarfs_mount_memory(const void *map, size_t size,
                       const char *mountpoint,
                       const char *link_rebase,
                       const char *path_rebase) {
  int slot = -1;
  struct tarfs_fs *fs = NULL;
  char base_dir[128] = { '/' };

  // 自动检测 TAR 归档的根目录名
  if (path_rebase) {
    strncpy(&base_dir[1], path_rebase, sizeof(base_dir)-1);
  } else {
    if (false == tar_rootdir(map, size, &base_dir[1], sizeof(base_dir)-1))
      base_dir[0] = '\0';
  }

  // ... mountpoint 验证 ...

  // 分配 FS 描述符（含 mountpoint 字符串的柔性数组）
  if (NULL == (fs = tarfs_calloc(1, sizeof(struct tarfs_fs) + len + 1))) {
    errno = ENOMEM;
    goto unmap_and_return_error;
  }

  // 在 s_tarfs[] 数组中找到空闲槽位
  tarfs_lock();
  if (0 > (slot = findfs(NULL))) {
    tarfs_unlock();
    tarfs_os_free(fs);
    errno = EBUSY;
    goto unmap_and_return_error;
  }
  s_tarfs[slot] = fs;
  s_numfs++;
  tarfs_unlock();  // 尽快释放锁

  // 初始化引用计数为 1
  initref(&fs->fs_ref);

  // 存储映射参数
  fs->fs_vaddr  = map;
  fs->fs_handle = 0;
  fs->fs_size   = size;
  memcpy(fs->fs_mountpoint, mountpoint, len);

  // 核心：解析 TAR 归档，构建 Inode 索引
  if (inode_mount(fs, map, size, link_rebase, &base_dir[1]) < 0) {
    if (fs->fs_ino == NULL) {
      // 没有找到任何有效 Inode，文件系统不可用
      tarfs_lock();
      s_tarfs[slot] = NULL;
      s_numfs--;
      tarfs_unlock();
      goto unmap_and_return_error;
    }
    // 部分记录损坏，但仍有有效 Inode，降级运行
    logerr("WARN: running in degraded mode\r\n");
  }

  // 在 VFS 中注册文件系统
  if (tarfs_os_register_fs(mountpoint, (void *)(intptr_t)slot) == false) {
    logerr("WARN: can not register POSIX handlers\r\n");
  }

  return slot;
}
```

**关键设计点：**

- `tarfs_calloc(1, sizeof(struct tarfs_fs) + len + 1)` —— FS 描述符末尾包含 mountpoint 字符串的柔性数组成员，一次分配避免内存碎片。
- 锁的持有时间极短：仅在操作 `s_tarfs[]` 数组时加锁，分配和 Inode 构建在锁外完成。
- **降级模式**是 TARFS 的容错设计：即使部分 TAR 记录损坏，只要有一个有效 Inode，文件系统仍能挂载并访问正常文件。

### 4.2 Inode 二分查找（inode.c）

TARFS 的核心查找机制是基于 FNV-1a 哈希的二分搜索：

```c
int inode_lookup(struct tarfs_inode const * const *index,
                 size_t num_inodes, const char *path) {
  uint32_t hash;
  size_t left, right;

  if (index == NULL || path == NULL || num_inodes < 1)
    return -EINVAL;

  left = 0;
  right = num_inodes;

  // 计算路径的 FNV-1a 哈希
  hash = hash32(HASH32_IV, (uint8_t const *)path, strlen(path));

  // 标准二分查找
  while (left < right) {
    size_t mid = left + ((right - left) >> 1);
    uint32_t mid_hash = index[mid]->in_hash;

    if (hash < mid_hash)
      right = mid;
    else if (hash > mid_hash)
      left = mid + 1;
    else {
      // 哈希碰撞处理：向左找到第一个相同哈希的 Inode
      size_t first = mid;
      while (first > 0 && index[first - 1]->in_hash == mid_hash)
        first--;

      // 在碰撞链中逐个比较路径字符串
      for (int i = first; i < num_inodes && index[i]->in_hash == hash; i++) {
        if (inode_pathcmp(index[i], path)) {
          return i;  // 找到匹配
        }
      }
      // 哈希碰撞但路径不匹配
      break;
    }
  }
  return -ENOENT;
}
```

**逐行解析：**

- `hash32(HASH32_IV, ...)` —— 使用 FNV-1a 算法计算路径的 32 位哈希。FNV-1a 具有良好的分布性，碰撞率低。
- 二分查找的时间复杂度为 O(log N)，65535 个文件最多 16 次比较即可定位。
- 哈希碰撞处理：当找到相同哈希时，先向左回溯到碰撞链起点，然后从左到右逐个比较实际路径字符串。这确保了即使哈希碰撞也能正确找到目标文件。
- `inode_pathcmp()` 是最终验证步骤，比较 Inode 中的路径与查找路径。它处理 TAR 字符串可能以 `\r`、`\n` 或 `\0` 结尾的特殊情况。

### 4.3 FNV-1a 优化哈希实现（hash.c）

TARFS 对标准 FNV-1a 做了内存访问优化：

```c
uint32_t hash32(uint32_t prev_hash, const uint8_t *data, size_t len) {
  const uint32_t *wdata;    // 32位对齐的指针
  size_t nwords;            // 32位块的个数
  uint32_t hash;
  uintptr_t addr;

  if (len == 0)
    return prev_hash;
  hash = prev_hash;
  addr = (uintptr_t)data;

  // 对齐处理：逐字节处理未对齐部分
  if (addr & 1) {
    HASH32_PSTEP(hash, data);  // hash ^= *data++; hash *= 16777619;
    HASH32_RETURN_IF_DONE;
  }
  if (addr & 2) {
    HASH32_PSTEP(hash, data);
    HASH32_RETURN_IF_DONE;
    HASH32_PSTEP(hash, data);
    HASH32_RETURN_IF_DONE;
  }

  // 快速路径：4字节一次读取，减少内存访问次数
  wdata = (const uint32_t *)data;
  nwords = len >> 2;
  len &= 3;  // 剩余尾部字节数 [0..3]

  while (nwords > 0) {
    uint32_t v = *wdata++;
#if CONFIG_TARFS_BIG_ENDIAN
    HASH32_CSTEP(hash, v >> 24);
    HASH32_CSTEP(hash, v >> 16);
    HASH32_CSTEP(hash, v >> 8);
    HASH32_CSTEP(hash, v);
#else
    // 小端序：直接按字节顺序处理
    HASH32_CSTEP(hash, v);
    HASH32_CSTEP(hash, v >> 8);
    HASH32_CSTEP(hash, v >> 16);
    HASH32_CSTEP(hash, v >> 24);
#endif
    nwords--;
  }

  // 处理尾部 1-3 字节
  data = (const uint8_t *)wdata;
  if (len & 2) {
    HASH32_PSTEP(hash, data);
    HASH32_PSTEP(hash, data);
  }
  if (len & 1) {
    HASH32_PSTEP(hash, data);
  }
  return hash;
}
```

**优化分析：**

- 标准 FNV-1a 逐字节处理，每次读取 1 字节。TARFS 的优化版本先对齐地址，然后以 4 字节为单位读取，将内存访问次数减少到 1/4。
- `HASH32_PSTEP` 宏展开为 `hash ^= *data++; hash *= 16777619;`，这是 FNV-1a 的核心操作。
- `HASH32_CSTEP` 类似但不递增指针，用于处理已读入寄存器的 32 位值。
- 支持增量哈希：可以分多次调用 `hash32()` 并传入上次的返回值，这在处理 PAX 头部等变长数据时很有用。

CRC64 完整性校验算法：

```c
uint64_t hash64(uint64_t prev_crc, void const *buffer0, size_t buf_len) {
    uint64_t crc = prev_crc;
    uint8_t const *buffer = buffer0;
    while (buf_len--) {
        crc ^= (uint64_t)*buffer++ << 56;
        for (int i = 0; i < 8; i++) {
            if (crc & 0x8000000000000000ULL)
                crc = (crc << 1) ^ 0x42F0E1EBA9EA3693ULL;
            else
                crc <<= 1;
        }
    }
    return crc;
}
```

- 这是 CRC-64/ECMA-182 的无查表实现。多项式 `0x42F0E1EBA9EA3693` 是 ECMA-182 标准多项式。
- 无查表版本不消耗额外 RAM（查表版本需要 256×8=2KB），适合内存受限的 MCU。速度较慢但只在 `tarfs_fsck()` 校验时调用，不影响运行时性能。

### 4.4 TAR 头部解析与校验（tar.c）

```c
uint32_t tar_hdrsum(tarhdr_t const *hdr) {
    uint8_t const *p = (uint8_t const *)hdr;
    uint32_t calc = 0;
    for (size_t i = 0; i < sizeof(tarhdr_t); i++)
      if (i >= 148 && i < (148 + 8))  // checksum 字段本身按空格处理
        calc += ' ';
      else
        calc += p[i];
    return calc;
}

bool tar_badhdr(tarhdr_t const *hdr) {
    uint32_t calc, expc;

    // 检查 entry type 是否支持
    switch (hdr->type) {
      case TART_AFILE:    // GNU 长文件名
      case TART_PAX:      // PAX 扩展头
      case TART_PAX_G:    // GNU PAX
      case TART_FILE:     // 普通文件
      case TART_HARDLINK: // 硬链接
      case TART_SYMLINK:  // 符号链接
      case TART_CHRDEV:   // 字符设备
      case TART_BLKDEV:   // 块设备
      case TART_DIR:      // 目录
      case TART_FIFO:     // FIFO
      case TART_CONT:     // 连续文件
        break;
      default:
        return true;  // 未知类型，跳过
    };

    // 验证保留字节为零（确保 prefix 字段 NUL 终止）
    if (hdr->zero != 0)
      return true;

    // 校验头部校验和
    calc = tar_hdrsum(hdr);
    expc = tar_octal(hdr->checksum, sizeof(hdr->checksum));
    if (expc != calc)
      return true;
    return false;
}
```

**解析：**

- `tar_hdrsum()` 计算 TAR 头部校验和。TAR 标准规定：计算校验和时，checksum 字段本身（字节 148-155）按 8 个空格处理。这是 TAR 格式的经典设计。
- `tar_badhdr()` 执行三重验证：类型合法性、保留字节、校验和。任何一项失败都视为坏块。
- TARFS 支持丰富的 TAR 类型，包括 GNU 长文件名扩展（`TART_AFILE`）和 PAX 扩展头（`TART_PAX`），这保证了对现代 `tar` 工具生成的归档的兼容性。

八进制数解析：

```c
uint32_t tar_octal(const char *p, size_t max_len) {
  uint32_t value = 0;
  while(max_len > 0 && *p == ' ') {  // 跳过前导空格
    p++;
    max_len--;
  }
  while (max_len > 0 && *p >= '0' && *p <= '7') {  // 移位累加
    value = (value << 3) | (*p - '0');
    p++;
    max_len--;
  }
  return value;
}
```

- TAR 格式中所有数值字段（文件大小、修改时间、校验和等）都以八进制 ASCII 字符串存储。`tar_octal()` 用移位加法（`value = (value << 3) | digit`）高效解析，比 `strtol()` 更轻量。

### 4.5 文件打开与读取（file.c）

```c
int tarf_open(void* ctx, const char *path0, int flags, int mode) {
  const char *path = path0;
  struct tarfs_fs *fs = NULL;
  int const fs_idx = (int)(intptr_t)ctx;

  // 只读文件系统，拒绝写操作
  if (((flags & O_ACCMODE) == O_WRONLY) || (flags & O_TRUNC)) {
      errno = EROFS;
      return -1;
  }

  // 获取 FS 引用（增加引用计数，确保 FS 在操作期间不被卸载）
  if (NULL == (fs = tarfs_getfs_addref(fs_idx))) {
    errno = EIO;
    return -1;
  }

  // O_DIRECTORY 路径修正：自动添加尾部斜杠
  if (flags & O_DIRECTORY) {
    int i = strlen(path);
    if (i > 0 && path[i - 1] != '/') {
      char *p = tar_strdup1(path, NULL);  // 双 NUL 终止
      if (p != NULL) {
        p[i] = '/';
        path = (const char *)p;
      }
    }
  }

  // 二分查找 Inode
  int idx = inode_lookup(fs->fs_ino, fs->fs_nino, path);
  if (idx < 0) {
    errno = ENOENT;
    goto unref_and_exit;
  }

  struct tarfs_inode const *inode = fs->fs_ino[idx];

  // 获取文件类型和大小
  type = inode_getinfo(fs->fs_ino, idx, &size, &mtime);

  // 类型检查：O_DIRECTORY 要求目标是目录
  if ((flags & O_DIRECTORY) != 0) {
    if (type != TART_DIR) {
      errno = ENOTDIR;
      goto unref_and_exit;
    }
    size = 0;  // 目录的 read() 必须失败
  } else {
    if (type == TART_DIR) {
      errno = EISDIR;
      goto unref_and_exit;
    }
  }

  // 原子分配文件描述符
  if ((fd = allocfd(fs)) >= 0) {
    struct tarfs_fp *fp = &fs->fs_fd[fd];
    // 直接指向 TAR 归档中的文件数据（跳过 512 字节头部）
    fp->fp_vaddr = inode->in_dvaddr + sizeof(struct tarhdr);
    fp->fp_pos   = 0;
    fp->fp_size  = size;
    fp->fp_idx   = idx;
    return fd;
  }

  errno = EMFILE;
unref_and_exit:
  tarfs_unref(fs);
  return -1;
}
```

**关键设计：**

- `tarfs_getfs_addref()` 增加文件系统引用计数，防止在 open 和 close 之间文件系统被卸载。这是 TARFS 线程安全策略的核心。
- `fp->fp_vaddr = inode->in_dvaddr + sizeof(struct tarhdr)` —— 文件指针直接指向 Flash 中 TAR 数据区域，跳过 512 字节的 TAR 头部。这是零拷贝设计的基础。
- `tar_strdup1()` 返回双 NUL 终止的字符串，允许在不重新分配的情况下追加一个字符（如 `/`），这在目录路径修正中很有用。

文件描述符的原子分配使用 CAS（Compare-And-Swap）操作：

```c
static int allocfd(struct tarfs_fs *fs) {
  int index;
  uint32_t free_mask;
  uint32_t used_indices, new_value;

  do {
    used_indices = atomic_load_explicit(&fs->fs_usedfd, memory_order_relaxed);
    free_mask = ~used_indices & s_valid_mask;
    if (free_mask == 0)
      return -1;  // 无可用 fd
    index = __builtin_ctz(free_mask);  // 找到第一个空闲位
    new_value = used_indices | (1u << index);
  } while (!atomic_compare_exchange_weak_explicit(
              &fs->fs_usedfd, &used_indices, new_value,
              memory_order_relaxed, memory_order_relaxed));
  return index;
}
```

- 使用 32 位位图管理文件描述符，`__builtin_ctz()`（Count Trailing Zeros）指令在单周期内找到第一个空闲位。
- CAS 循环确保多线程环境下 fd 分配的原子性。`memory_order_relaxed` 足够，因为 fd 的内容在 `open()` 返回前已完全初始化，发布通过函数返回值完成。

读取操作：

```c
ssize_t tarf_read(void* ctx, int fd, void *dst, size_t size) {
  PROLOGUE(ssize_t);  // 验证 fs 和 fd
  struct tarfs_fp *fp = &fs->fs_fd[fd];

  // 计算源地址：文件基址 + 当前位置
  void const *src = (void const *)(fp->fp_vaddr + fp->fp_pos);

  // 限制读取范围，不超出文件末尾
  if (fp->fp_size - fp->fp_pos < size)
    size = fp->fp_size - fp->fp_pos;

  // 拷贝数据（使用架构优化 memcpy）
  if (size > 0) {
    fp->fp_pos += size;
    tarfs_os_memcpy(dst, src, size);
  }
  return size;
}
```

- `PROLOGUE` 宏验证文件系统上下文和文件描述符的有效性，所有文件操作函数都以它开头。
- 读取操作本质是一次 `memcpy`，源地址直接指向 Flash 中的 TAR 数据。ESP32-S3 上可使用汇编优化的 `memcpy` 实现（通过 `CONFIG_TARFS_HAVE_OPTIMIZED_MEMCPY` 启用）。
- 测试数据：ESP32-S3 240MHz、Flash 80MHz 四线模式下，1KB 读取的吞吐量约为 26 MiB/s。

### 4.6 ESP32 平台适配层（os_esp32.c）

```c
void const *tarfs_os_map_tarfile(const char *label, void **os_handle_out,
                                  size_t *size_out) {
  esp_partition_iterator_t i;
  esp_partition_mmap_handle_t handle;
  const esp_partition_t *part;
  size_t len;
  void const *map;

  // 通过 label 查找 Flash 分区
  i = esp_partition_find(ESP_PARTITION_TYPE_DATA,
                         ESP_PARTITION_SUBTYPE_ANY, label);
  if (i == NULL) {
    errno = ENOENT;
    return false;
  }
  part = esp_partition_get(i);
  esp_partition_iterator_release(i);

  // 将整个分区映射到 CPU 地址空间
  if (ESP_OK == esp_partition_mmap(part, 0, part->size,
                                    ESP_PARTITION_MMAP_DATA,
                                    &map, &handle)) {
    *os_handle_out = (void *)handle;
    if (size_out)
      *size_out = part->size;
    return map;  // 返回可直接指针访问的虚拟地址
  }

  errno = ENOMEM;
  return false;
}
```

**解析：**

- `esp_partition_find()` 遍历分区表，查找指定 label 的数据分区。TARFS 使用自定义子类型 `0xF0`（`ESP_PARTITION_SUBTYPE_DATA_TARFS`）标识 TARFS 分区。
- `esp_partition_mmap()` 是 ESP-IDF 的内存映射 API，它将 Flash 区域映射到 CPU 地址空间，返回的指针可以像访问 RAM 一样直接读取 Flash 数据。
- `ESP_PARTITION_MMAP_DATA` 表示映射为数据区域（不可执行），这在 ESP32-S3 上通过 cache 映射实现。

VFS 注册：

```c
static const esp_vfs_fs_ops_t s_tarfs_fs = {
    .write_p = &tarf_write,
    .lseek_p = &tarf_lseek,
    .read_p  = &tarf_read,
    .pread_p = &tarf_pread,
    .open_p  = &tarf_open,

    .fstat_p = &tarf_fstat,

    .fsync_p = &tarf_sync,
    .dir     = &s_tarfs_dir,
};

  return ESP_OK == esp_vfs_register_fs(prefix,
                                      &s_tarfs_fs,
                                       ESP_VFS_FLAG_CONTEXT_PTR |
                                       ESP_VFS_FLAG_STATIC |
                                       ESP_VFS_FLAG_READONLY_FS,
                                       context);

```

- TARFS 使用 ESP-IDF 的 VFS 注册 API 将自己注册到系统 VFS 层。注册后，标准 POSIX 函数（`open()`、`read()`、`mmap()` 等）会自动路由到 TARFS 的处理函数。
- `ESP_VFS_FLAG_CONTEXT_PTR` —— VFS 在调用处理函数时传入 context 指针（即 FS 槽位索引），这使得多个 TARFS 实例可以同时挂载。
- `ESP_VFS_FLAG_READONLY_FS` —— 告知 VFS 这是只读文件系统，写操作会被 VFS 层直接拒绝。

### 4.7 引用计数与安全卸载（fs.c + refc.c）

TARFS 使用引用计数确保卸载安全：

```c
static void commit_unmount(void *ctx) {
  struct tarfs_fs *fs = (struct tarfs_fs *)ctx;
  if (fs == NULL) return;

  // 1. 从 VFS 注销（防止新的 open() 调用）
  if (!tarfs_os_unregister_fs(fs->fs_mountpoint)) {
    logerr("failed to unregister '%s', memory leaked!\r\n",
           fs->fs_mountpoint);
    return;  // 注销失败，保留数据防止崩溃
  }

  // 2. 清除 s_tarfs[] 槽位
  tarfs_lock();
  for (int i = 0; i < TARFS_MAX_FS; i++)
    if (s_tarfs[i] == fs) {
      s_numfs--;
      s_tarfs[i] = NULL;
      break;
    }
  tarfs_unlock();

  // 3. 释放 Inode 索引
  if (fs->fs_ino != NULL && fs->fs_nino > 0) {
    inode_unmount(fs, (const void *)fs->fs_vaddr, fs->fs_size);
    fs->fs_ino = NULL;
  }

  // 4. 解除 Flash 映射
  if (fs->fs_vaddr != NULL && fs->fs_handle != 0) {
    tarfs_os_unmap_tarfile((void *)fs->fs_handle,
                           (void *)fs->fs_vaddr, fs->fs_size);
    fs->fs_vaddr = NULL;
  }

  // 5. 释放 FS 描述符
  tarfs_os_free(fs);
}
```

- `commit_unmount()` 是引用计数归零时的析构函数。它只在最后一个使用者释放引用时被调用。
- 卸载流程严格有序：先从 VFS 注销（阻止新请求）→ 清除全局槽位 → 释放 Inode → 解除映射 → 释放描述符。
- 如果 VFS 注销失败，函数选择内存泄漏而非继续卸载，因为继续释放可能导致正在进行的 VFS 回调访问已释放的内存。

### 4.8 Inode 内存布局与分配（inode.c）

```c
struct tarfs_inode **inode_alloc(size_t count) {
    char *ptr;
    struct tarfs_inode *nodes, **index;
    size_t index_size, nodes_size;

    index_size = count * sizeof(struct tarfs_inode *);
    nodes_size = count * sizeof(struct tarfs_inode);

    // 单次分配：索引数组 + Inode 数组连续存放
    if (NULL == (ptr = tarfs_calloc(1, index_size + nodes_size))) {
      errno = ENOMEM;
      return NULL;
    }

    index = (struct tarfs_inode **)ptr;
    nodes = (struct tarfs_inode *)(ptr + index_size);

    // 初始化：index[i] 指向 nodes[i]
    for (size_t i = 0; i < count; i++)
      index[i] = &nodes[i];

    return index;
}
```

**内存布局图：**

```
低地址 ──────────────────────────────────────► 高地址

┌──────────────────────────────────────────────────────────┐
│  index[0]  │  index[1]  │ ... │  index[N-1]  │           │  指针数组
├────────────┴────────────┴─────┴──────────────┤           │
│  inode[0]  │  inode[1]  │ ... │  inode[N-1]  │           │  Inode 数组
└────────────┴────────────┴─────┴──────────────┘           │
                                                            │
  index[i] ──────────────────────────────────► &inode[i]   │
```

- 索引数组和 Inode 数组在一次 `calloc` 中分配，保证内存连续性，减少堆碎片。
- 排序时只交换索引指针，不移动 Inode 结构体本身。这是经典的"排序指针数组"技巧。
- 每个 `tarfs_inode` 为 24 字节，包含哈希值、路径指针、数据虚拟地址和链表指针。

### 4.9 Inode 排序与目录遍历（inode.c）

TARFS 对 Inode 进行两种排序：哈希排序（用于查找）和字典序排序（用于 readdir）。

```c
// 堆排序：按哈希值排序（用于二分查找）
void inode_sort(struct tarfs_inode **iarr, size_t count) {
    if (count < 2) return;
    // 建堆
    for (size_t i = count / 2; i-- > 0;)
        inode_siftdown(v, i, count);
    // 堆排序
    for (size_t end = count; end > 1; end--) {
        inode_exchange(&v[0], &v[end - 1]);
        inode_siftdown(v, 0, end - 1);
    }
}

// 归并排序：按字典序排序（用于 readdir 目录遍历）
static struct tarfs_inode *merge_sort(struct tarfs_inode *head) {
    if (!head || !head->in_next)
        return head;
    // 快慢指针找中点
    struct tarfs_inode *slow = head;
    struct tarfs_inode *fast = head->in_next;
    while (fast && fast->in_next) {
        slow = slow->in_next;
        fast = fast->in_next->in_next;
    }
    struct tarfs_inode *right = slow->in_next;
    slow->in_next = NULL;
    // 递归排序
    struct tarfs_inode *left = merge_sort(head);
    right = merge_sort(right);
    return merge(left, right);
}
```

- **堆排序**用于哈希排序，时间复杂度 O(N log N)，原地排序不需要额外空间。选择堆排序而非快排是因为 TAR 记录可能已部分有序，堆排序的最坏情况复杂度更有保证。
- **归并排序**用于字典序排序，通过 `in_next` 指针构建单链表，时间复杂度 O(N log N)。归并排序是稳定排序，保证相同文件名的相对顺序不变。
- 两种排序共用 `in_next` 字段：在 `inode_resolve()` 完成后，该字段被清零，然后由 `inode_alphasort()` 重新构建为字典序链表。

```cpp
  // === 读取文件 ===
  int fd = open("/My_FS/list/example.c", O_RDONLY);
  if (fd >= 0) {
    char buf[128];
    ssize_t n;
    while ((n = read(fd, buf, sizeof(buf))) > 0)
      fwrite(buf, 1, n, stdout);

  // === 目录列表 ===
  DIR *dir = opendir("/My_FS/list");
  if (dir) {
    struct dirent *de;
    while ((de = readdir(dir)) != NULL)
      printf("%s\n", de->d_name);
    closedir(dir);
  }

  // === mmap 零拷贝访问 ===
  fd = open("/My_FS/list/example.c", O_RDONLY);
  if (fd >= 0) {
    struct stat st;
    if (fstat(fd, &st) == 0) {
      void *ptr = mmap(NULL, st.st_size, PROT_READ, MAP_SHARED, fd, 0);
      if (ptr != MAP_FAILED) {
        // ptr 直接指向 Flash 中的文件数据，零拷贝
        printf("mmap() succeeded, %ld bytes\r\n", st.st_size);
        munmap(ptr, st.st_size);
      }
    }
    close(fd);
  }
}
```

## 七、项目亮点与适用场景

### 技术亮点

1. **极致的零拷贝设计**：`mmap()` 返回的指针直接指向 Flash 中的 TAR 数据，文件内容无需拷贝到 RAM。这使得访问任意大小文件都不会增加内存消耗。

2. **O(log N) 查找性能**：基于 FNV-1a 哈希和二分查找的 Inode 索引，即使在数千个文件的场景下也能快速定位。

3. **优雅的容错策略**：无中心超级块的设计意味着单个 TAR 记录损坏不会导致整个文件系统不可用。损坏记录在挂载时被跳过，有效记录仍可正常访问。

4. **运行时零分配**：除 mount/unmount 外，整个数据路径无动态内存分配。文件描述符使用位图分配，读取操作仅涉及指针运算和 memcpy。

5. **POSIX 兼容性**：支持 mmap、readlink、fdopendir、符号链接、硬链接等高级 POSIX 特性，使得 Linux 移植软件能直接运行。

### 适用场景

| 场景 | 说明 |

| UI 资源管理 | LVGL 字体、图片、主题文件打包为 TAR 归档 |
| 证书存储 | TLS 证书链和密钥文件 |

## 八、总结

TARFS 是一个设计精巧的嵌入式文件系统，它在"够用就好"和"功能完整"之间找到了完美平衡点。通过复用标准 TAR 格式，它避免了设计新文件系统的复杂性和兼容性问题；通过零拷贝 mmap 和哈希索引，它在资源受限的 MCU 上实现了接近理论极限的性能。

项目创建于 2026 年 7 月 4 日，虽然年轻，但代码质量高、文档完善、设计思路清晰。对于需要在 ESP32 等 MCU 上管理大量只读资源文件的开发者来说，TARFS 是一个值得关注的解决方案。它的平台抽象层设计（仅需一个 `os_*.c` 文件）也使其能轻松移植到 STM32、RP2040 等其他平台。
