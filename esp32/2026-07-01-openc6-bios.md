## 项目简介与核心特性一览

OpenC6-BIOS 是一款为 ESP32-C6（RISC-V 架构，8MB Flash）设计的开源 BIOS 固件平台。它颠覆了传统嵌入式固件的单一应用模式，引入了类似 PC 主板 BIOS 的系统架构：BIOS/Payload 分离设计、自定义系统调用接口（ABI）、独立低功耗协处理器管理（LP-Core Management Engine）、以及防变砖的 A/B OTA 更新系统。

| 特性 | 说明 |
|------|------|
| BIOS/Payload 架构 | 类似 PC BIOS，BIOS 加载并执行独立 Payload |
| 自定义 ABI | 安全跳转表，Payload 无需静态地址映射 |
| A/B OTA | 防变砖双分区 OTA 更新系统 |
| PXE 网络启动 | Wi-Fi 网络下载并部署 Payload |
| 串口启动 | UART 加载到 RAM 或 Flash |
| NVRAM | 类似 CMOS 的配置存储（NVS） |
| Management Engine | LP-Core 独立协处理器，带看门狗 |
| 电源管理 | S5 软关机、DVFS 调频、热保护 |
| Web UI | 配置门户 AP 模式 |
| 自定义文件系统 | 基于扇区的轻量级 Flash 文件系统 |

## 核心代码深度分析

### 1. ABI 系统调用接口 — 安全跳转表设计

openc6-bios 的核心创新在于其 ABI（Application Binary Interface）设计。BIOS 在运行时构建一个函数指针表，传递给独立编译的 Payload，实现解耦。

```c
#pragma once
#include <stdint.h>

// 完整性标记 "C6BI" (C6 BIOS Interface)，验证调用上下文
#define OPENC6_ABI_MAGIC   0x43364249
#define OPENC6_ABI_VERSION 1

typedef struct {
    uint32_t magic;    // 必须等于 OPENC6_ABI_MAGIC
    uint32_t version;  // 必须等于 OPENC6_ABI_VERSION

    // ─── 系统API封装 ───
    void (*sys_reset)(void) __attribute__((noreturn));
    void (*set_led_color)(uint8_t r, uint8_t g, uint8_t b);
    void (*delay_ms)(uint32_t ms);
    void* (*malloc)(uint32_t size);
    void (*free)(void* ptr);
    void (*print)(const char *str);
    uint32_t (*get_random)(void);
    void (*sha256)(const uint8_t *input, uint32_t len, uint8_t *output);

    // ─── 数学函数（无 FPU 系统优化）───
    uint32_t (*math_isqrt)(uint32_t x);
    int32_t (*math_sin_deg)(int32_t angle_deg);
    int32_t (*math_cos_deg)(int32_t angle_deg);

    // ─── 无线网络接口 ───
    int32_t (*wifi_connect)(const char* ssid, const char* pass);
    int32_t (*wifi_start_ap)(const char* ssid, const char* pass);
    int32_t (*wifi_is_connected)(void);

    // ─── 遥测与堆监控 ───
    uint32_t (*get_free_ram)(void);
    uint32_t (*get_total_ram)(void);
    uint32_t (*get_total_flash)(void);

    // ─── 高速文件系统接口 ───
    void (*fs_write_file)(const char *name, const uint8_t *data,
                          uint32_t len, uint32_t dir_sector, uint8_t force);
    int32_t (*fs_read_file)(const char *name, uint8_t *dest,
                           uint32_t offset, uint32_t len, uint32_t dir_sector);
    void (*fs_delete)(const char *name, uint32_t dir_sector);
} openc6_abi_t;
```

**逐行解析：**

- `#define OPENC6_ABI_MAGIC 0x43364249` — 魔数 "C6BI" 的 ASCII 编码。Payload 入口点首先检查此值，确保 ABI 表来自合法的 OpenC6 BIOS 而非内存垃圾。这类似于 UEFI 的签名验证机制。
- `#define OPENC6_ABI_VERSION 1` — ABI 版本号。当 BIOS 升级添加新接口时递增版本号，Payload 通过版本检查避免调用不存在的函数指针（会导致 HardFault）。
- 整个结构体是一个**函数指针跳转表**。每个字段都是 BIOS 提供的函数地址。Payload 通过 `abi->print("hello")` 调用，等价于间接函数调用 `((void(*)(const char*))abi->print)("hello")`。
- `void (*sys_reset)(void) __attribute__((noreturn));` — `noreturn` 属性告知编译器此函数不会返回，允许编译器优化掉 reset 调用后的代码。
- 数学函数使用**定点格式**（`int32_t` 返回值 ×10000），因为 ESP32-C6 的 RISC-V 内核无硬件 FPU。BIOS 在内部使用 `double` 计算，通过 ABI 提供给 Payload 整数结果，避免了 Payload 需要链接浮点库的开销。
- 文件系统接口中 `dir_sector` 参数使用 `uint32_t` 而非指针，确保 ABI 在不同编译器/架构间兼容（Payload 可能用不同的工具链编译）。

### 2. ABI 构建与函数绑定

在 `bios_init.c` 中，BIOS 启动时构建 ABI 表并绑定所有函数实现：

```c
static void bios_sha256(const uint8_t *input, uint32_t len, uint8_t *output) {
    psa_crypto_init();
    size_t hash_len;
    psa_hash_compute(PSA_ALG_SHA_256, input, len, output, 32, &hash_len);
}

static uint32_t bios_get_free_ram(void) {
    return heap_caps_get_free_size(MALLOC_CAP_INTERNAL | MALLOC_CAP_8BIT);
}

static uint32_t bios_get_total_flash(void) {
    uint32_t flash_size = 0;
    if (esp_flash_get_size(NULL, &flash_size) == ESP_OK) {
        return flash_size;
    }
    return 0;
}

static uint32_t bios_math_isqrt(uint32_t x) {
    uint32_t res = 0;
    uint32_t bit = 1UL << 30;
    while (bit > x) bit >>= 2;
    while (bit != 0) {
        if (x >= res + bit) {
            x -= res + bit;
            res = (res >> 1) + bit;
        } else {
            res >>= 1;
        }
        bit >>= 2;
    }
    return res;
}

static const openc6_abi_t bios_abi = {
    .magic = OPENC6_ABI_MAGIC,
    .version = OPENC6_ABI_VERSION,
    .sys_reset = esp_restart,
    .set_led_color = led_mgmt_set_color,
    .delay_ms = bios_delay_ms,
    .malloc = bios_malloc,
    .free = free,
    .print = bios_print,
    .get_random = esp_random,
    .sha256 = bios_sha256,
    .math_isqrt = bios_math_isqrt,
    .math_sin_deg = bios_math_sin_deg,
    .math_cos_deg = bios_math_cos_deg,
    .wifi_connect = bios_wifi_connect,
    .wifi_start_ap = bios_wifi_start_ap,
    .wifi_is_connected = bios_wifi_is_connected,
    .get_free_ram = bios_get_free_ram,
    .get_total_ram = bios_get_total_ram,
    .get_total_flash = bios_get_total_flash,
    .fs_write_file = abi_fs_write_file,
    .fs_read_file = abi_fs_read_file,
    .fs_delete = abi_fs_delete
};
```

**逐行解析：**

- `static const openc6_abi_t bios_abi = { ... };` — 使用 C99 指定初始化器构建常量结构体。`const` 关键字使结构体存储在 Flash 中（而非 RAM），节省宝贵的内部 SRAM。
- `.sha256 = bios_sha256` — SHA-256 函数绑定到 PSA Crypto API。ESP32-C6 的硬件加密加速器在后台完成计算，对 Payload 透明。
- `.sys_reset = esp_restart` — 直接映射到 ESP-IDF 的系统重启函数。
- `bios_math_isqrt` — 手写的整数平方根算法（牛顿迭代变体）。使用位移操作 `1UL << 30` 从最高位开始试探，每次比较后右移 2 位。这种算法在无硬件乘除法器的内核上比浮点 `sqrt()` 快数倍。
- `bios_get_free_ram` — 使用 `heap_caps_get_free_size(MALLOC_CAP_INTERNAL | MALLOC_CAP_8BIT)` 精确获取内部 SRAM 剩余空间。`MALLOC_CAP_INTERNAL` 排除外部 PSRAM，`MALLOC_CAP_8BIT` 确保返回的内存支持字节级访问。

### 3. PXE 网络启动 — Payload 下载与部署

```c
bool pxe_boot_execute(const char* url) {
    ESP_LOGI(TAG, "Starting Network Boot from: %s", url);

    esp_http_client_config_t config = {
        .url = url,
        .timeout_ms = 5000,
        .keep_alive_enable = false,
    };
    esp_http_client_handle_t client = esp_http_client_init(&config);
    if (!client) {
        ESP_LOGE(TAG, "Failed to init HTTP client");
        return false;
    }
    esp_err_t err = esp_http_client_open(client, 0);
    if (err != ESP_OK) {
        ESP_LOGE(TAG, "Failed to open HTTP connection: %s", esp_err_to_name(err));
        esp_http_client_cleanup(client);
        return false;
    }
    int content_length = esp_http_client_fetch_headers(client);
    if (content_length <= 0) {
        ESP_LOGE(TAG, "Failed to get Content-Length or file is empty!");
        esp_http_client_cleanup(client);
        return false;
    }
    // 限制最大 Payload 大小，保护 SRAM 边界
    if (content_length > 150 * 1024) {
        ESP_LOGE(TAG, "Payload size %d exceeds safe RAM limits (150KB)!", content_length);
        esp_http_client_cleanup(client);
        return false;
    }
    uint8_t *h_buf = malloc(content_length);
    if (!h_buf) {
        ESP_LOGE(TAG, "Failed to allocate memory buffer of %d bytes", content_length);
        esp_http_client_cleanup(client);
        return false;
    }
    // 流式下载
    int total_read = 0;
    char buffer[1024];
    while (total_read < content_length) {
        int read_len = esp_http_client_read(client, buffer, sizeof(buffer));
        if (read_len < 0) {
            ESP_LOGE(TAG, "Error reading data or connection closed prematurely");
            free(h_buf);
            esp_http_client_cleanup(client);
            return false;
        }
        if (read_len == 0) break;
        memcpy(h_buf + total_read, buffer, read_len);
        total_read += read_len;
    }
```

**逐行解析：**

- `esp_http_client_config_t config` — 配置 HTTP 客户端。`timeout_ms = 5000` 设置 5 秒超时，`keep_alive_enable = false` 关闭保活以节省连接资源。
- `esp_http_client_open(client, 0)` — 第二个参数 0 表示使用 GET 请求且不发送请求体。
- `content_length > 150 * 1024` — 安全边界检查。ESP32-C6 的内部 SRAM 约 500KB，预留系统开销后，限制单次下载不超过 150KB，防止 OOM 崩溃。
- `char buffer[1024]` — 使用 1KB 栈缓冲区逐块下载。这是嵌入式内存管理的典型模式：用小缓冲区处理大数据流，避免一次性分配大内存。
- `memcpy(h_buf + total_read, buffer, read_len)` — 将每块数据追加到堆缓冲区。下载完成后整个 Payload 被写入 Flash 文件系统，再从 Flash 原地执行（XIP）。

### 4. A/B OTA 系统固件自更新

```c
bool pxe_bios_ota_execute(const char* url) {
    ESP_LOGW(TAG, "Starting Network BIOS OTA from: %s", url);

    const esp_partition_t *update_part = esp_ota_get_next_update_partition(NULL);
    if (update_part == NULL) {
        ESP_LOGE(TAG, "Passive OTA partition not found!");
        esp_http_client_cleanup(client);
        return false;
    }
    ESP_LOGI(TAG, "Writing BIOS OTA to partition: %s at offset 0x%08lX",
             update_part->label, update_part->address);
    ESP_LOGI(TAG, "Erasing OTA partition...");

    esp_ota_handle_t update_handle = 0;
    err = esp_ota_begin(update_part, OTA_SIZE_UNKNOWN, &update_handle);
    if (err != ESP_OK) {
        ESP_LOGE(TAG, "esp_ota_begin failed: %s", esp_err_to_name(err));
        return false;
    }
    // 流式写入 Flash，仅消耗 1KB RAM
    char buffer[1024];
    int total_read = 0;
    while (total_read < content_length) {
        int read_len = esp_http_client_read(client, buffer, sizeof(buffer));
        if (read_len < 0) {
            esp_ota_abort(update_handle);
            esp_http_client_cleanup(client);
            return false;
        }
        if (read_len == 0) break;
        err = esp_ota_write(update_handle, buffer, read_len);
        if (err != ESP_OK) {
            esp_ota_abort(update_handle);
            esp_http_client_cleanup(client);
            return false;
        }
        total_read += read_len;
    }
    esp_http_client_cleanup(client);

    if (total_read == content_length) {
        err = esp_ota_end(update_handle);
        if (err != ESP_OK) return false;
        err = esp_ota_set_boot_partition(update_part);
        if (err != ESP_OK) return false;
        ESP_LOGI(TAG, "BIOS OTA Flashed successfully! Total: %d bytes", total_read);
        return true;
    }
    esp_ota_abort(update_handle);
    return false;
}
```

**逐行解析：**

- `esp_ota_get_next_update_partition(NULL)` — 获取被动（非活跃）OTA 分区。ESP-IDF 的 A/B 分区方案：Flash 被分为 Factory + OTA_0 + OTA_1。当前运行在 OTA_0 时，更新写入 OTA_1，反之亦然。
- `esp_ota_begin(update_part, OTA_SIZE_UNKNOWN, &update_handle)` — `OTA_SIZE_UNKNOWN` 表示总大小未知，ESP-IDF 会按需擦除分区。
- `char buffer[1024]` — 关键优化：与 PXE Payload 下载不同，OTA 流式写入**不分配大堆缓冲区**。每读 1KB 就直接写入 Flash，仅需 1KB 栈空间。这使得即使 8MB 的 BIOS 也能在 500KB SRAM 的设备上更新。
- `esp_ota_write(update_handle, buffer, read_len)` — 将数据块写入被动分区。ESP-IDF 内部处理 Flash 擦除和写入对齐。
- `esp_ota_set_boot_partition(update_part)` — 设置下次启动从更新后的分区引导。当前运行不受影响，直到重启。
- 这种 A/B 方案确保了**防变砖**：如果新固件有问题，可以通过 Boot Manager 回退到旧分区。

### 5. BIOS 启动状态机

```c
void bios_core_start(void) {
    esp_reset_reason_t rst_reason = esp_reset_reason();
    bool is_cold_boot = (rst_reason != ESP_RST_DEEPSLEEP);
    uint32_t legacy_hold_ms = 0;

    // ... 唤醒原因检测与按钮长按计时 ...

    nvram_init();
    led_mgmt_init();
    wifi_mgmt_init();
    hal_flash_init();
    fs_init();

    // 读取 Management Engine 状态
    me_state_t me_state;
    nvram_get_me_state(&me_state);
    is_me_enabled = (me_state == ME_ENABLED);
    management_engine_init(is_me_enabled, is_cold_boot);
    if (is_me_enabled) {
        ulp_me_hp_is_awake = 1;
        xTaskCreate(me_watchdog_pet_task, "me_wdt", 2048, NULL, 5, NULL);
    }

    // ─── 网络 BIOS 更新检查 ───
    bios_update_state_t ota_state;
    nvram_get_bios_update_state(&ota_state);
    if (ota_state == BIOS_UPDATE_PENDING) {
        ESP_LOGW(TAG, ">>> NETWORK BIOS UPDATE PENDING <<<");
        nvram_set_bios_update_state(BIOS_UPDATE_NONE);
        // ... 执行 OTA 更新 ...
    }

    // ─── CMOS 清除检测 ───
    if (gpio_get_level((gpio_num_t)PIN_CLEAR_NVRAM) == 0) {
        ESP_LOGE(TAG, "Jumper on PIN %d detected! Clearing NVRAM...", PIN_CLEAR_NVRAM);
        nvram_load_defaults();
        esp_restart();
    }

    // ─── POST 诊断 ───
    ESP_LOGI(TAG, "Executing POST...");
    led_mgmt_blink_post(COLOR_POST_OK, 3);

    // ─── OTA 回滚验证 ───
    esp_ota_img_states_t ota_img_state;
    if (esp_ota_get_state_partition(esp_ota_get_running_partition(), &ota_img_state) == ESP_OK) {
        if (ota_img_state == ESP_OTA_IMG_PENDING_VERIFY) {
            ESP_LOGI(TAG, "POST OK! Confirming stable boot to Rollback Manager...");
            esp_ota_mark_app_valid_cancel_rollback();
        }
    }

    // ─── 启动原因分析矩阵 ───
    if (me_reason == ME_BOOT_REASON_THERMAL) {
        ESP_LOGE(TAG, "THERMAL EMERGENCY! Powering off.");
        bios_enter_s5_state();
    }
    else if (me_reason == ME_BOOT_REASON_FORCE_RESET) {
        bios_enter_s5_state();
    }
    else if (me_reason == ME_BOOT_REASON_WDT) {
        direct_flash_boot = true;  // 看门狗复位，恢复启动
    }
    else if (me_reason == ME_BOOT_REASON_SETUP || legacy_hold_ms >= 3000) {
        enter_setup = true;  // 进入 BIOS 设置
    }
    else if (me_reason == ME_BOOT_REASON_NORMAL || legacy_hold_ms > 0) {
        direct_flash_boot = true;
    }
    // ... 状态机路由执行 ...
}
```

**逐行解析：**

- `esp_reset_reason()` — 读取复位原因寄存器。区分冷启动（power-on）和深度睡眠唤醒（deep sleep），这对于电源管理状态机至关重要。
- `management_engine_init(is_me_enabled, is_cold_boot)` — 初始化 LP-Core 管理引擎。如果 ME 启用，HP-Core 进入睡眠后由 LP-Core 监控电源按钮和温度。
- `xTaskCreate(me_watchdog_pet_task, ...)` — 创建看门狗喂食任务。每 2 秒调用 `management_engine_pet_watchdog()`。如果 HP-Core 卡死，LP-Core 的看门狗超时后会强制复位——这类似于 PC 主板的 Intel ME 看门狗机制。
- `if (ota_state == BIOS_UPDATE_PENDING)` — 检查 NVRAM 中的 OTA 待更新标志。Web UI 设置此标志后，下次重启时自动执行 OTA。更新前先清除标志，防止更新失败后陷入循环。
- `esp_ota_mark_app_valid_cancel_rollback()` — ESP-IDF 的回滚保护机制。新固件首次启动后标记为"待验证"（PENDING_VERIFY），只有 POST 成功后调用此函数标记为"有效"。如果新固件启动后崩溃，ESP-IDF 会在数次尝试后自动回滚到旧分区。
- `ME_BOOT_REASON_THERMAL` — LP-Core 在 HP-Core 睡眠期间检测到温度超过阈值后唤醒 HP-Core，并设置此原因。BIOS 收到后直接关机保护。
- `legacy_hold_ms >= 3000` — 传统模式下（ME 禁用），通过检测电源按钮长按时长决定操作：短按正常启动，长按 3 秒进入 BIOS 设置。

### 6. Payload 示例 — ABI 实战调用

```c
/* tools/example/payload.c */
#include "openc6_abi.h"

// 自定义微型 printf（因为使用 -nostdlib 编译）
static void print_uint32(const openc6_abi_t *abi, uint32_t val) {
    char buf[16];
    int i = 14;
    buf[15] = '\0';
    if (val == 0) {
        buf[i--] = '0';
    } else {
        while (val > 0 && i >= 0) {
            buf[i--] = '0' + (val % 10);
            val /= 10;
        }
    }
    abi->print(&buf[i + 1]);
}

void __attribute__((section(".text.entry"), noreturn))
payload_main(const openc6_abi_t *abi) {
    // 0. ABI 验证
    if (!abi || abi->magic != OPENC6_ABI_MAGIC || abi->version != OPENC6_ABI_VERSION) {
        while (1) { __asm__ volatile("nop"); }
    }

    // ─── 测试1: 内存监控 ───
    uint32_t flash_kb = abi->get_total_flash() / 1024;
    uint32_t ram_tot_kb = abi->get_total_ram() / 1024;
    uint32_t ram_free_kb = abi->get_free_ram() / 1024;

    // ─── 测试2: 动态内存分配 ───
    uint8_t *test_buf = (uint8_t*)abi->malloc(1024);
    if (test_buf) {
        test_buf[0] = 0xAA;
        abi->free(test_buf);
    }

    // ─── 测试3: 硬件 TRNG & SHA-256 ───
    uint32_t r1 = abi->get_random();
    const char* secret = "OpenC6";
    uint8_t hash[32];
    abi->sha256((const uint8_t*)secret, 6, hash);

    // ─── 测试4: 数学加速 ───
    uint32_t sq_val = abi->math_isqrt(144);       // 返回 12
    uint32_t sin_val = (uint32_t)abi->math_sin_deg(90); // 返回 10000 (=1.0×10000)

    // ─── 测试5: Wi-Fi 连接 ───
    int wifi_res = abi->wifi_connect("Test", "12345678");
    if (wifi_res == 0) {
        abi->delay_ms(2000);
        if (abi->wifi_is_connected()) {
            abi->set_led_color(0, 255, 0); // 绿色 — 有网络
        } else {
            abi->set_led_color(255, 165, 0); // 橙色
        }
    } else {
        abi->set_led_color(255, 0, 0); // 红色
    }

    // ─── 完成后重启回 BIOS ───
    abi->delay_ms(3000);
    abi->sys_reset();
}
```

**逐行解析：**

- `__attribute__((section(".text.entry")))` — 将函数放入 `.text.entry` 段。自定义链接脚本（`tools/payload.ld`）将此段放置在固定 Flash 地址，BIOS 的 Boot Manager 跳转到此地址开始执行 Payload。
- `__attribute__((noreturn))` — 标记函数永不返回。Payload 执行完毕后调用 `abi->sys_reset()` 重启，不会返回到 BIOS。
- `while (1) { __asm__ volatile("nop"); }` — ABI 验证失败时的安全陷阱。使用内联汇编 `nop` 避免编译器优化掉空循环，保持 CPU 在低功耗空转状态而非执行随机内存。
- `print_uint32` — 由于 Payload 使用 `-nostdlib` 编译（不链接标准库），必须自己实现数字转字符串。这个函数用除法逐位提取十进制数字。
- `abi->sha256((const uint8_t*)secret, 6, hash)` — 通过 ABI 调用 BIOS 的硬件加速 SHA-256。Payload 自身不包含加密库代码，大幅减小体积。对 "OpenC6"（6 字节）计算哈希，结果写入 32 字节缓冲区。
- `abi->math_sin_deg(90)` — 返回 10000（即 1.0 × 10000）。定点格式避免了浮点运算，Payload 无需链接 `libm`。
- `abi->wifi_connect("Test", "12345678")` — 通过 ABI 连接 Wi-Fi。BIOS 内部处理 Wi-Fi 驱动初始化、WPA2 认证、DHCP 获取 IP 等全部复杂逻辑，对 Payload 透明。
- `abi->sys_reset()` — 执行完毕，触发硬件复位回到 BIOS。这确保 Payload 不会意外修改 BIOS 的内存空间。

### 7. NVRAM 配置存储 — 类 CMOS 设计

```c
esp_err_t nvram_load_defaults(void) {
    ESP_LOGW(TAG, "Restoring factory defaults (Clear CMOS)...");
    nvs_flash_erase();
    nvram_init();

    // 1. 基本系统参数
    nvram_set_dc_loss_action(DC_LOSS_POWER_OFF);
    nvram_set_post_led_mode(POST_LED_ENABLED);

    // 2. Aura Sync RGB 参数
    nvram_set_aura_mode(AURA_RAINBOW);
    nvram_set_aura_brightness(128);

    // 3. AI Tweaker (CPU & 电源核心调优)
    nvram_set_cpu_freq(CPU_FREQ_160MHZ);      // 标准最大频率
    nvram_set_cpu_governor(GOV_PERFORMANCE);  // 高性能模式
    nvram_set_bod_level(BOD_STRICT);          // 严格掉电检测阈值
    nvram_set_me_state(ME_ENABLED);           // 启用管理引擎
    nvram_set_thermal_limits(VAL_TEMP_THROTTLE_DEFAULT,
                             VAL_TEMP_EMERGENCY_DEFAULT);

    // 4. Wi-Fi 凭据与 PXE 默认 URL
    nvram_set_wifi_sta_config("", "");
    nvram_set_pxe_url(VAL_PXE_URL_DEFAULT);

    ESP_LOGI(TAG, "Factory defaults loaded successfully.");
    return ESP_OK;
}
```

**逐行解析：**

- `nvs_flash_erase()` — 擦除整个 NVS 分区，等价于 PC 主板的 "Clear CMOS" 操作。ESP-IDF 的 NVS（Non-Volatile Storage）使用 Flash 的特定分区，支持键值对存储。
- `nvram_set_dc_loss_action(DC_LOSS_POWER_OFF)` — 设置掉电恢复策略。`DC_LOSS_POWER_OFF` 表示掉电后不自动开机（类似 PC 的 AC Loss 状态），`DC_LOSS_BOOT` 则自动恢复启动。
- `nvram_set_cpu_freq(CPU_FREQ_160MHZ)` — 设置 CPU 最大频率。ESP32-C6 的 RISC-V 核心可运行在 80MHz 或 160MHz，BIOS 根据此设置配置 DVFS（动态电压频率调节）。
- `nvram_set_bod_level(BOD_STRICT)` — Brownout Detector（掉电检测器）阈值。严格模式在电压降至 2.7V 时触发复位，宽松模式降至 2.4V。这对电池供电场景至关重要。
- `nvram_set_thermal_limits(throttle, emergency)` — 设置温度阈值。`throttle` 是降频阈值，`emergency` 是强制关机阈值。这些值被传给 LP-Core 的 Management Engine 在深度睡眠期间监控。

### 开发自定义 Payload

```c
// my_payload.c
#include "openc6_abi.h"

void __attribute__((section(".text.entry"), noreturn))
payload_main(const openc6_abi_t *abi) {
    if (abi->magic != OPENC6_ABI_MAGIC) return;

    abi->print("Hello from custom payload!\n");

    // 使用 BIOS 的 Wi-Fi 连接网络
    if (abi->wifi_connect("MySSID", "MyPass") == 0) {
        abi->set_led_color(0, 255, 0);
        // 通过文件系统保存数据
        const char *data = "boot_log=success";
        abi->fs_write_file("log.txt", (const uint8_t*)data, 16, 0, 1);
    }

    abi->delay_ms(5000);
    abi->sys_reset();
}
```

## 项目亮点与适用场景

### 项目亮点

1. **PC 级 BIOS 架构**：将 PC 主板 BIOS 的设计理念（BIOS/Payload 分离、ABI 跳转表、A/B OTA、NVRAM/CMOS、Management Engine）完整移植到 ESP32-C6，是嵌入式固件架构的重大创新。
2. **安全解耦设计**：ABI 跳转表将 BIOS 和 Payload 完全解耦，Payload 使用 `-nostdlib` 编译，通过函数指针调用 BIOS 服务。这种设计使得 Payload 可以使用不同的工具链编译，甚至用 Rust 或 Zig 编写。
3. **防变砖 OTA**：A/B 双分区 + 回滚验证机制确保固件更新不会变砖。新固件首次启动后必须通过 POST 验证才被标记为有效，否则自动回滚。
4. **LP-Core 管理引擎**：利用 ESP32-C6 的双核特性，LP-Core 在深度睡眠时充当管理引擎，监控电源按钮和温度，实现类似 Intel ME 的独立管理能力。

### 适用场景

- IoT 设备远程固件管理系统
- 需要防变砖 OTA 的生产级设备
- 多应用动态加载的嵌入式平台
- 教学用 BIOS 架构参考实现
- 需要低功耗远程管理的传感器节点

## 总结

OpenC6-BIOS 是一款极具创新性的嵌入式固件项目。它将 PC 主板 BIOS 的成熟架构——BIOS/Payload 分离、ABI 系统调用、A/B OTA 防变砖、Management Engine 独立监控——完整地移植到 ESP32-C6 这一低成本 RISC-V 微控制器上。ABI 跳转表设计使得 Payload 开发变得极为简洁，开发者无需关心底层驱动，只需调用 BIOS 提供的 Wi-Fi、文件系统、加密等服务。对于希望学习系统级固件架构设计的嵌入式开发者来说，这个项目提供了教科书级的参考实现。
