### 项目简介

P4KVM 是一个基于乐鑫 ESP32-P4 芯片构建的概念验证（Proof of Concept）级 IP KVM（Keyboard, Video, Mouse over IP）方案。它将一台被控计算机的 HDMI 视频信号通过 TC358743 桥接芯片转换为 MIPI CSI-2 接口输入到 ESP32-P4，再由芯片内部的硬件 JPEG 编码器实时压缩为 MJPEG 流，最终通过 HTTP multipart/x-mixed-replace 协议推送到浏览器；同时，浏览器端的键鼠输入经 WebSocket 传回 ESP32-P4，再由 USB HID（TinyUSB 协议栈）模拟成标准键盘鼠标发送给被控主机。整个方案在单颗 SoC 上完成了"采集-编码-传输-控制"的完整 KVM 链路。

### 核心特性一览

| 特性 | 说明 |
|------|------|
| **视频采集** | TC358743 HDMI-to-MIPI CSI-2 桥接，最高 1920×1080 RGB888 |
| **视频编码** | ESP32-P4 内置硬件 JPEG 编码器，RGB888 输入 |
| **流媒体** | HTTP multipart/x-mixed-replace，浏览器原生支持 |
| **键鼠控制** | WebSocket + USB HID（TinyUSB），支持绝对/相对鼠标、6 键无冲键盘 |
| **网络** | 内置以太网 RMII + DHCP + mDNS（p4kvm.local） |
| **缓冲策略** | PSRAM 三缓冲 + ping-pong DMA，无撕裂 |
| **热插拔** | HDMI 断连自动检测与恢复 |
| **配置持久化** | JPEG 质量等参数存于 NVS |
| **单客户端独占** | WebSocket 会话强制单活跃客户端 |

## 四、核心代码深度分析

本节是全文重点，对每段关键源码进行逐段/逐行解析。

### 4.1 main.c — 应用入口

```c
#include "esp_log.h"
#include "freertos/FreeRTOS.h"
#include "freertos/task.h"

#include "capture.h"
#include "ethernet.h"
#include "http_server.h"
#include "jpeg_frame.h"
#include "usb_hid.h"

static const char *TAG = "main";

void app_main(void)
{
    ESP_LOGI(TAG, "P4KVM boot");

    /* JPEG triple-buffer + NVS quality. */
    jpeg_frame_init();
    jpeg_quality_load_from_nvs();

    /* Start the HDMI→CSI→MJPEG pipeline (Core 0). */
    capture_start();

    /* USB HID keyboard/mouse (TinyUSB). */
    ESP_ERROR_CHECK(usb_hid_init());

    /* Ethernet (DHCP) + mDNS → http://p4kvm.local/ */
    ESP_ERROR_CHECK(ethernet_init());

    /* HTTP server: / /stream /ws /jpeg-quality */
    http_server_start();
}
```

**逐段解析：**

- 头文件部分引入了 ESP-IDF 的日志、FreeRTOS 任务头，以及项目自身的五个模块头文件（capture、ethernet、http_server、jpeg_frame、usb_hid），体现了模块化分层设计。
- `static const char *TAG = "main";` 是 ESP-IDF 日志系统的标准做法，所有 `ESP_LOGI/W/E` 调用都会带上此 tag，方便在串口监视器中过滤。
- `app_main` 是 ESP-IDF 应用的唯一入口，由系统启动后调用。注意它本身不是 FreeRTOS 任务，但 ESP-IDF 会为其创建一个主任务。
- `jpeg_frame_init()` 最先执行：初始化 JPEG 三缓冲结构体（槽位、互斥锁、信号量）。之所以放在最前面，是因为后续所有路径（编码、流传输）都依赖这个共享数据结构。
- `jpeg_quality_load_from_nvs()` 从 NVS（非易失存储）读取上次保存的 JPEG 质量值，实现参数断电不丢失。
- `capture_start()` 启动采集 pipeline，注释明确指出运行在 **Core 0**。ESP32-P4 为双核，将高负载的视频采集绑定到固定核心可避免缓存抖动和任务迁移开销。
- `ESP_ERROR_CHECK(usb_hid_init())` 初始化 TinyUSB USB HID 栈。`ESP_ERROR_CHECK` 宏会在返回非 ESP_OK 时直接 abort，适用于"必须成功"的初始化。
- `ESP_ERROR_CHECK(ethernet_init())` 初始化以太网与 mDNS。注意这里没有显式等待 DHCP 拿到 IP，因为后续 HTTP 服务器启动后，mDNS 已可解析。
- `http_server_start()` 最后启动，此时采集 pipeline 已就绪、网络已通，服务器可立即响应 `/stream` 等请求。

**设计要点：** 初始化顺序经过精心安排——先就绪共享数据结构，再启动数据生产者（采集），然后启动控制通道（USB），接着打通网络，最后启动服务暴露层（HTTP）。这种"自底向上"的依赖顺序避免了任何模块访问未初始化资源。

### 4.2 capture.c — 采集任务启动

```c
static void camera_task(void *arg)
{
    (void)arg;
    capture_ctx_t *ctx = capture_hw_init_start();
    if (!ctx) {
        return;
    }
    capture_mjpeg_run(ctx);
}

void capture_start(void)
{
    const uint32_t cam_stack = 10240;
    xTaskCreatePinnedToCore(camera_task, "cam", cam_stack, NULL, 5, NULL, 0);
}
```

**逐段解析：**

- `camera_task` 是 FreeRTOS 任务函数，参数 `arg` 未使用故用 `(void)arg` 消除警告。
- `capture_hw_init_start()` 完成所有硬件初始化（LDO、I2C、TC358743、CSI、ISP、Bridge）并返回一个 `capture_ctx_t` 上下文指针。如果初始化失败返回 NULL，任务直接退出（生产环境应增加日志和重试）。
- `capture_mjpeg_run(ctx)` 进入 MJPEG 编码主循环——这是一个永不返回的 `while(1)`，持续从 DMA 缓冲取帧、编码、写入三缓冲。
- `capture_start()` 中 `cam_stack = 10240`（10KB）栈空间。视频采集与 JPEG 编码涉及较多局部缓冲和函数调用层级，10KB 是合理配置。
- `xTaskCreatePinnedToCore(..., 5, NULL, 0)` 的参数依次为：任务函数、任务名、栈大小、参数、优先级 5、句柄 NULL、**核心 0**。优先级 5 属中等偏高，确保采集不被低优先级任务饿死；绑定 Core 0 与 main.c 注释一致。

**设计要点：** 将"硬件初始化"与"编码循环"分离到 `capture_hw.c` 和 `capture_mjpeg.c` 两个文件，通过 `capture_ctx_t` 上下文串联，符合单一职责原则。任务绑定固定核心有利于 cache 局部性。

### 4.3 capture_hw.c — CSI/ISP/TC358743 硬件初始化（关键节选）

```c
// LDO 初始化（MIPI CSI 供电）
esp_ldo_channel_config_t ldo_cfg = {
    .chan_id = P4KVM_MIPI_LDO_CHAN_ID,
    .voltage_mv = P4KVM_MIPI_LDO_VOLTAGE_MV,
};
ESP_ERROR_CHECK(esp_ldo_acquire_channel(&ldo_cfg, &ldo));

// TC358743 硬件复位脉冲
tc358743_resetn_pulse();

// I2C 主总线初始化
i2c_master_bus_config_t i2c_bus_cfg = {
    .i2c_port = I2C_NUM_0,
    .sda_io_num = P4KVM_TC358743_I2C_SDA_GPIO,
    .scl_io_num = P4KVM_TC358743_I2C_SCL_GPIO,
    .glitch_ignore_cnt = 7,
    .flags = {.enable_internal_pullup = 1},
};
ESP_ERROR_CHECK(i2c_new_master_bus(&i2c_bus_cfg, &i2c_bus));

// 探测并初始化 TC358743
ESP_ERROR_CHECK(tc358743_probe(i2c_bus, NULL, &s_cap.tc));
ESP_ERROR_CHECK(tc358743_init_streaming(s_cap.tc));

// 帧缓冲分配（PSRAM，对齐到 DMA 缓存行）
s_cap.frame_bytes = (size_t)s_cap.hres * (size_t)s_cap.vres * 3u;  // RGB888 = 3 bytes/pixel
const uint32_t caps = MALLOC_CAP_DMA | MALLOC_CAP_SPIRAM | MALLOC_CAP_8BIT;
uint8_t *blk = heap_caps_aligned_calloc(align, CAPTURE_FB_COUNT, s_cap.frame_bytes, caps);

// CSI 控制器配置
esp_cam_ctlr_csi_config_t csi_cfg = {
    .h_res = s_cap.hres,
    .v_res = s_cap.vres,
    .data_lane_num = 2,
    .lane_bit_rate_mbps = P4KVM_MIPI_LANE_MBPS,
    .queue_items = CAPTURE_FB_COUNT,
};

// ISP 旁路模式（直接传递 RGB888）
esp_isp_processor_cfg_t isp_cfg = {
    .input_data_source = ISP_INPUT_DATA_SOURCE_CSI,
    .flags = {.bypass_isp = true},
};
capture_fill_esp_cam_color_types(&csi_cfg, &isp_cfg);  // RGB888 in/out

// P4 CSI Bridge 寄存器配置
MIPI_CSI_BRIDGE.frame_cfg.hadr_num = hres;
MIPI_CSI_BRIDGE.frame_cfg.vadr_num = vres;
MIPI_CSI_BRIDGE.data_type_cfg.data_type_min = 0x24u;  // RGB888
MIPI_CSI_BRIDGE.data_type_cfg.data_type_max = 0x24u;

// CSI 回调注册
esp_cam_ctlr_evt_cbs_t cbs = {
    .on_get_new_trans = cam_on_get_new,    // 请求新缓冲区
    .on_trans_finished = cam_on_done,       // DMA 传输完成
};
ESP_ERROR_CHECK(esp_cam_ctlr_register_event_callbacks(s_cam, &cbs, &s_cap));

// 启动 HDMI 输出
ESP_ERROR_CHECK(tc358743_enable_hdmi_output(s_cap.tc));
wait_tc358743_pixel_stream(s_cap.tc, 5000);  // 等待 HDMI 信号锁定

// 启动 CSI 控制器
ESP_ERROR_CHECK(esp_cam_ctlr_start(s_cam));
```

**逐段解析：**

1. **LDO 供电**：ESP32-P4 的 MIPI CSI 接口需要独立的 2.5V LDO 供电（通道 3）。`esp_ldo_acquire_channel` 获取并保持该通道，确保 CSI PHY 稳定工作。这是任何 MIPI 外设上电的第一步。

2. **硬件复位**：`tc358743_resetn_pulse()` 产生一个复位脉冲（拉低 RESET 引脚 GPIO 23 再释放），确保 TC358743 从已知状态启动，避免上电时寄存器处于随机态。

3. **I2C 总线**：配置 I2C_NUM_0，SDA=GPIO7、SCL=GPIO8，开启内部上拉，毛刺过滤计数 7。这是与 TC358743 通信的控制总线（配置寄存器、读取 EDID 等）。

4. **TC358743 探测与流式初始化**：`tc358743_probe` 通过 I2C 读取芯片 ID 确认存在；`tc358743_init_streaming` 配置芯片内部的 HDMI 接收、色彩空间转换、MIPI CSI-2 发送等寄存器，使其开始输出 MIPI 数据流。

5. **帧缓冲分配**：
   - `frame_bytes = hres * vres * 3`：1920×1080×3 = 6,220,800 字节（约 5.93MB）/帧。RGB888 每像素 3 字节。
   - `heap_caps_aligned_calloc` 分配对齐到 DMA 缓存行的连续内存，标志要求 `MALLOC_CAP_DMA`（DMA 可访问）+ `MALLOC_CAP_SPIRAM`（PSRAM）+ `MALLOC_CAP_8BIT`。
   - `CAPTURE_FB_COUNT` 帧 ping-pong 缓冲一次性分配，避免运行时动态分配导致碎片。1080p 三帧约 18MB，必须使用 PSRAM。

6. **CSI 控制器配置**：2 条数据 lane、每 lane 972Mbps、队列项数等于帧缓冲数。CSI 控制器是 ESP32-P4 内部的硬件模块，负责接收 MIPI CSI-2 数据并 DMA 到内存。

7. **ISP 旁路**：`bypass_isp = true` 表示跳过 ISP（图像信号处理器）的 demosaic/降噪/白平衡等处理。因为 TC358743 已输出 RGB888（非 Bayer RAW），无需 ISP 处理，直接透传可降低延迟和带宽。`capture_fill_esp_cam_color_types` 统一设置输入输出色彩格式为 RGB888。

8. **P4 CSI Bridge 寄存器**：直接操作 `MIPI_CSI_BRIDGE` 硬件寄存器，设置水平/垂直地址数（像素数），并将数据类型过滤为 `0x24`（MIPI CSI-2 规范中 RGB888 的数据类型码）。这确保只有 RGB888 类型的数据包被接受。

9. **回调注册**：注册两个中断回调——`on_get_new`（CSI 需要新缓冲区时调用，提供 ping-pong 地址）和 `on_trans_done`（一帧 DMA 传输完成时调用，唤醒编码任务）。这是零拷贝 pipeline 的核心。

10. **启动顺序**：先 `tc358743_enable_hdmi_output` 使桥接芯片开始输出，再 `wait_tc358743_pixel_stream` 轮询等待 HDMI 信号稳定锁定（超时 5 秒），最后 `esp_cam_ctlr_start` 启动 CSI 控制器开始接收。这个顺序确保 CSI 启动时已有有效像素流，避免接收到残帧。

**设计要点：** 整个初始化是典型的"从物理层到协议层"自底向上流程：供电→复位→I2C 控制→芯片配置→内存准备→控制器配置→回调注册→启动采集。每步都用 `ESP_ERROR_CHECK` 保障失败即停。

### 4.3b capture_hw.c — CSI DMA 回调（IRAM_ATTR 中断处理）

```c
static bool IRAM_ATTR cam_on_get_new(esp_cam_ctlr_handle_t h, esp_cam_ctlr_trans_t *trans, void *ud)
{
    capture_ctx_t *c = (capture_ctx_t *)ud;
    int i = c->ping_fb_idx % CAPTURE_FB_COUNT;
    trans->buffer = c->fb[i];           // 提供 ping-pong 缓冲区
    trans->buflen = c->frame_bytes;
    c->ping_fb_idx = (c->ping_fb_idx + 1) % CAPTURE_FB_COUNT;
    return false;
}

static bool IRAM_ATTR cam_on_done(esp_cam_ctlr_handle_t h, esp_cam_ctlr_trans_t *trans, void *ud)
{
    capture_ctx_t *c = (capture_ctx_t *)ud;
    if (trans && trans->buffer) {
        c->done_fb = trans->buffer;     // 标记完成的缓冲区
    }
    BaseType_t high_task_woken = pdFALSE;
    if (c->csi_done_sem) {
        xSemaphoreGiveFromISR(c->csi_done_sem, &high_task_woken);  // 唤醒编码任务
    }
    return high_task_woken;
}
```

**逐段解析：**

- `IRAM_ATTR` 宏将函数放入内部 RAM（IRAM），因为它们在中断上下文中执行，不能访问 flash（flash 访问会与 cache 访问冲突导致异常）。这是 ESP32 中断处理的硬性要求。
- **`cam_on_get_new`**：CSI 控制器需要新缓冲区时调用。通过 `ping_fb_idx` 循环索引在 `CAPTURE_FB_COUNT` 个缓冲区间轮转（ping-pong），将地址填入 `trans->buffer`。`return false` 表示不触发任务切换（在 ISR 中尽快返回）。这是零拷贝的关键——DMA 直接写入预分配的 PSRAM 缓冲，无内存拷贝。
- **`cam_on_done`**：一帧 DMA 传输完成时调用。将刚完成的缓冲区指针存入 `done_fb`，然后 `xSemaphoreGiveFromISR` 释放信号量唤醒 MJPEG 编码任务。`high_task_woken` 跟踪是否有更高优先级任务被唤醒，返回该值让 ISR 决定是否触发上下文切换。
- 注意 `done_fb` 是单值而非队列：每次只保留最新完成帧。如果编码任务来不及处理，旧帧会被覆盖（丢帧策略优于积压），这对实时 KVM 是合理的——宁可丢帧也不增加延迟。

**设计要点：** ISR 极简——只做指针赋值和信号量释放，将繁重工作推迟到任务上下文。ping-pong 双缓冲让采集与编码可并行：DMA 写 A 缓冲时编码任务处理 B 缓冲。

### 4.4 capture_hw.c — HDMI 热插拔恢复

```c
esp_err_t capture_hw_hdmi_recover(capture_ctx_t *c)
{
    // 1. 停止 CSI 控制器
    esp_err_t er = esp_cam_ctlr_stop(s_cam);
    
    // 2. 清空 DMA 完成信号量
    capture_drain_csi_done_sem(c->csi_done_sem);
    
    // 3. TC358743 热插拔复位（模拟断开/重连）
    er = tc358743_hdmi_hotplug_reset(c->tc);
    
    // 4. 等待 HDMI 信号重新锁定
    wait_tc358743_pixel_stream(c->tc, 5000);
    
    // 5. 重新配置 CSI TX 路径
    er = tc358743_reapply_csi_path_after_hdmi(c->tc);
    
    // 6. 重新配置 P4 CSI Bridge 寄存器
    capture_configure_p4_csi_bridge(c->hres, c->vres);
    
    // 7. 重置帧缓冲状态
    c->ping_fb_idx = 0;
    c->done_fb = NULL;
    
    // 8. 重新启动 CSI 控制器
    er = esp_cam_ctlr_start(s_cam);
    return er;
}
```

**逐段解析：**

这是一个八步恢复流程，处理 HDMI 线缆被拔出又重新插入的场景：

1. **停止 CSI**：先停控制器，避免在恢复过程中接收到无效数据。
2. **清空信号量**：`capture_drain_csi_done_sem` 排空遗留的 DMA 完成信号量，防止编码任务基于过时帧工作。
3. **热插拔复位**：让 TC358743 重新检测 HDMI 输入，重新读取 EDID 并协商分辨率。
4. **等待信号锁定**：5 秒超时等待像素流稳定。
5. **重配 CSI TX 路径**：HDMI 源可能改变了分辨率/时序，需重新配置桥接芯片的 MIPI 输出。
6. **重配 Bridge 寄存器**：同步更新 ESP32-P4 侧的 CSI Bridge 寄存器以匹配新分辨率。
7. **重置缓冲状态**：`ping_fb_idx` 归零、`done_fb` 清空，回到初始 ping-pong 状态。
8. **重启 CSI**：恢复采集。

**设计要点：** 热插拔是 KVM 场景的刚需（用户会切换主机）。该函数在 MJPEG 编码循环的 CSI 超时分支中被调用（见 4.5），实现了自动故障检测与恢复，无需人工干预。

### 4.5 capture_mjpeg.c — MJPEG 编码循环（节选关键逻辑）

```c
void capture_mjpeg_run(capture_ctx_t *c)
{
    while (1) {
        // 等待 CSI DMA 完成一帧
        if (xSemaphoreTake(c->csi_done_sem, pdMS_TO_TICKS(2000)) != pdTRUE) {
            // 超时：可能 HDMI 断开
            ESP_LOGW(TAG, "CSI timeout - attempting HDMI recovery");
            capture_hw_hdmi_recover(c);
            continue;
        }

        void *fb = c->done_fb;
        if (!fb) continue;

        // 选择编码槽位（三缓冲）
        int slot = jpeg_frame_pick_encode_slot();
        if (slot < 0) {
            continue;  // 所有槽位都被占用，跳过此帧
        }

        // JPEG 硬件编码
        jpeg_encoder_config_t enc_cfg = {
            .width = c->hres,
            .height = c->vres,
            .quality = g_jpeg_frame.jpeg_quality,
            .input_format = JPEG_PIXEL_FORMAT_RGB888,
            .output_format = JPEG_PIXEL_FORMAT_RGB888,
        };
        
        size_t out_len = 0;
        esp_err_t ret = jpeg_encode(&enc_cfg, fb, c->frame_bytes,
                                     g_jpeg_frame.jpeg_buf[slot],
                                     g_jpeg_frame.jpeg_cap, &out_len);
        
        if (ret == ESP_OK && out_len > 0) {
            // 更新帧索引（原子操作）
            xSemaphoreTake(g_jpeg_frame.mutex, portMAX_DELAY);
            g_jpeg_frame.jpeg_len[slot] = out_len;
            // 确保不是当前正在传输的帧
            if (g_jpeg_frame.front_idx != slot || 
                g_jpeg_frame.slot_ref[g_jpeg_frame.front_idx] == 0) {
                g_jpeg_frame.front_idx = slot;
            }
            g_jpeg_frame.frame_seq++;
            xSemaphoreGive(g_jpeg_frame.mutex);
            
            // 通知流媒体客户端有新帧
            jpeg_frame_notify_new_frame();
        }
    }
}
```

**逐段解析：**

- **等待帧就绪**：`xSemaphoreTake(..., pdMS_TO_TICKS(2000))` 阻塞等待 DMA 完成信号量，超时 2 秒。这是热插拔检测的关键——正常情况下每帧间隔约 16ms（60fps），2 秒无帧意味着 HDMI 断连。超时后调用 `capture_hw_hdmi_recover` 恢复并 `continue` 重新等待。
- **取帧指针**：`fb = c->done_fb` 读取 ISR 写入的最新完成帧。若为 NULL（理论上不应发生）则跳过。
- **选槽位**：`jpeg_frame_pick_encode_slot()` 在三缓冲中找一个空闲槽位。返回 -1 表示全部被流传输引用中，此时跳过该帧（丢帧保护），避免编码覆盖正在传输的数据。
- **JPEG 硬件编码配置**：宽高、质量值（从全局 `g_jpeg_frame.jpeg_quality` 读取，可被 HTTP 接口动态修改）、输入输出格式均为 RGB888。ESP32-P4 内置硬件 JPEG 编码器，编码速度远超软件实现，是实时 1080p 的前提。
- **jpeg_encode 调用**：输入是 DMA 帧缓冲 `fb`，输出写入 `g_jpeg_frame.jpeg_buf[slot]`（PSRAM），容量上限 `jpeg_cap`，实际输出长度写入 `out_len`。
- **原子更新帧索引**：进入互斥区后更新 `jpeg_len[slot]`。关键逻辑在 `if` 判断：只有当目标槽位不是当前传输帧，或当前传输帧引用计数为 0（无人读取）时，才将 `front_idx` 指向新槽位。这防止了"正在发送的帧被替换"导致的撕裂。
- `frame_seq++` 是帧序列号，流媒体任务通过比较它判断是否有新帧。
- **通知**：`jpeg_frame_notify_new_frame()` 释放信号量唤醒所有等待的流媒体客户端。

**设计要点：** 编码循环是生产者，流媒体任务是消费者。三缓冲 + 引用计数实现了无锁的"生产者可不阻塞地推进，消费者持有帧时不会被覆盖"。丢帧策略（slot<0 时跳过）保证编码端永不阻塞，这是实时流的关键。

### 4.6 http_server.c — MJPEG 流媒体（multipart/x-mixed-replace）

```c
static void stream_worker_task(void *arg)
{
    httpd_req_t *req = (httpd_req_t *)arg;
    jpeg_frame_stream_enter();
    uint32_t last_seq = g_jpeg_frame.frame_seq;

    while (1) {
        // 等待新帧
        while (g_jpeg_frame.frame_seq == last_seq) {
            if (xSemaphoreTake(g_jpeg_frame.frame_ready_sem, pdMS_TO_TICKS(500)) != pdTRUE) {
                if (stream_peer_disconnected(req)) break;
                continue;
            }
            // 合并突发帧（只发送最新帧）
            while (xSemaphoreTake(g_jpeg_frame.frame_ready_sem, 0) == pdTRUE) {}
        }

        // 获取传输锁和帧槽位
        xSemaphoreTake(g_jpeg_frame.xmit_mutex, portMAX_DELAY);
        
        int slot = g_jpeg_frame.front_idx;
        if (slot >= 0 && g_jpeg_frame.jpeg_buf[slot]) {
            g_jpeg_frame.slot_ref[slot]++;  // 引用计数
            
            // 发送 multipart 帧
            char hdr[96];
            snprintf(hdr, sizeof(hdr),
                "--frame\r\n"
                "Content-Type: image/jpeg\r\n"
                "Content-Length: %zu\r\n\r\n",
                g_jpeg_frame.jpeg_len[slot]);
            
            httpd_resp_send_chunk(req, hdr, strlen(hdr));
            httpd_resp_send_chunk(req, (const char *)g_jpeg_frame.jpeg_buf[slot],
                                  g_jpeg_frame.jpeg_len[slot]);
            httpd_resp_send_chunk(req, "\r\n", 2);
            
            g_jpeg_frame.slot_ref[slot]--;  // 释放引用
        }
        xSemaphoreGive(g_jpeg_frame.xmit_mutex);
        last_seq = g_jpeg_frame.frame_seq;
    }
    
    jpeg_frame_stream_leave();
    httpd_req_async_handler_complete(req);
    vTaskDelete(NULL);
}

// 流媒体端点注册
static esp_err_t stream_get(httpd_req_t *req)
{
    httpd_resp_set_type(req, "multipart/x-mixed-replace; boundary=frame");
    httpd_resp_set_hdr(req, "Cache-Control", "no-cache, no-store, must-revalidate");
    
    // 异步处理（不阻塞 httpd 主线程）
    httpd_req_t *async_req;
    httpd_req_async_handler_begin(req, &async_req);
    
    // 在独立任务中运行流媒体
    xTaskCreate(stream_worker_task, "p4kvm_stream", 12*1024, async_req, 5, NULL);
    return ESP_OK;
}
```

**逐段解析：**

- **`stream_get` 端点**：
  - 设置 Content-Type 为 `multipart/x-mixed-replace; boundary=frame`，这是 MJPEG over HTTP 的标准 MIME 类型，浏览器（如 `<img>` 标签）会持续接收并刷新显示每一帧。
  - `Cache-Control` 禁用缓存，确保实时性。
  - **异步处理**是核心：`httpd_req_async_handler_begin` 将请求转为异步句柄，使 `stream_get` 立即返回 ESP_OK，不占用 httpd 主线程。然后在独立任务 `stream_worker_task` 中持续推送帧。若不异步，一个长连接流会阻塞整个 HTTP 服务器。
  - 任务栈 12KB，优先级 5。

- **`stream_worker_task`**：
  - `jpeg_frame_stream_enter()` 增加流客户端计数，使编码端知道有消费者在等待。
  - `last_seq` 记录上次发送的帧序列号。
  - **等待新帧**：内层 while 比较 `frame_seq`，相等表示无新帧。`xSemaphoreTake(..., 500ms)` 阻塞等待 `frame_ready_sem`。超时则检查客户端是否断开（`stream_peer_disconnected`），断开则退出循环。
  - **合并突发帧**：`while (xSemaphoreTake(..., 0) == pdTRUE) {}` 以非阻塞方式排空信号量。如果编码速度快于网络，会积压多个信号量；这里只保留最新帧，丢弃中间帧，避免延迟累积。
  - **传输锁**：`xmit_mutex` 串行化多个流客户端的发送，保证同一时刻只有一个任务在操作 `front_idx` 和引用计数。
  - **引用计数**：`slot_ref[slot]++` 标记该槽位正在被读取，编码端会跳过引用计数非零的槽位，实现无锁并发。
  - **multipart 帧格式**：`--frame\r\n` 分隔符 + Content-Type + Content-Length + 空行 + JPEG 数据 + `\r\n`。浏览器据此解析每帧。
  - 发送完毕 `slot_ref[slot]--` 释放引用。
  - 循环退出后 `jpeg_frame_stream_leave()` 减少客户端计数，`httpd_req_async_handler_complete` 通知 httpd 异步处理完成，`vTaskDelete(NULL)` 自删除任务。

**设计要点：** 这是典型的"生产者-多消费者"模式。三缓冲让编码器持续产出，多个流客户端各自独立取最新帧。突发帧合并策略确保低延迟。异步任务模型使单个流连接不阻塞其他 HTTP 请求。

### 4.7 http_server.c — WebSocket 键鼠输入协议

```c
static esp_err_t ws_input_handler(httpd_req_t *req)
{
    if (req->method == HTTP_GET) {
        ws_take_session(req);  // WebSocket 握手
        return ESP_OK;
    }

    // 接收 WebSocket 帧
    httpd_ws_frame_t pkt = {0};
    httpd_ws_recv_frame(req, &pkt, 0);
    
    uint8_t buf[32];
    pkt.payload = buf;
    httpd_ws_recv_frame(req, &pkt, sizeof(buf));

    // 单客户端强制：检查是否为当前活跃会话
    int my_fd = httpd_req_to_sockfd(req);
    if (s_ws_fd != my_fd && s_ws_fd >= 0) {
        return ESP_OK;  // 非活跃客户端，忽略
    }

    // 鼠标消息 (type=0x01)
    if (pkt.len >= 8 && buf[0] == 0x01) {
        uint8_t buttons = buf[1];
        int8_t wheel = (int8_t)buf[6];
        bool relative = buf[7] != 0;
        
        if (relative) {
            int16_t dx = (int16_t)((uint16_t)buf[2] | ((uint16_t)buf[3] << 8));
            int16_t dy = (int16_t)((uint16_t)buf[4] | ((uint16_t)buf[5] << 8));
            usb_hid_mouse_rel(buttons, dx, dy, wheel);
        } else {
            uint16_t x = (uint16_t)buf[2] | ((uint16_t)buf[3] << 8);
            uint16_t y = (uint16_t)buf[4] | ((uint16_t)buf[5] << 8);
            usb_hid_mouse(buttons, x, y, wheel);
        }
    }
    // 键盘消息 (type=0x02)
    else if (pkt.len >= 8 && buf[0] == 0x02) {
        usb_hid_keyboard(buf[1], &buf[2]);  // modifier + 6 keycodes
    }

    return ESP_OK;
}
```

**逐段解析：**

- **握手处理**：WebSocket 连接以 HTTP GET 升级请求开始。`ws_take_session` 完成 Sec-WebSocket-Key 校验和 101 Switching Protocols 响应，并记录 socket fd 为活跃会话（`s_ws_fd`）。
- **两步接收帧**：第一次 `httpd_ws_recv_frame(req, &pkt, 0)` 仅获取帧头（长度、类型），不分配 payload；然后用栈缓冲 `buf[32]` 作为 payload，第二次接收读取实际数据。这种两步法避免了对未知长度帧的动态分配。32 字节足以容纳 8 字节的键鼠消息。
- **单客户端独占**：`httpd_req_to_sockfd` 获取当前请求的 socket fd，与全局 `s_ws_fd` 比较。若不一致且已有活跃会话，直接 `return ESP_OK` 忽略。KVM 场景下多人同时控制会造成混乱，强制单客户端是合理设计。
- **鼠标消息解析（0x01）**：
  - `buf[1]` 按键位掩码（bit0 左、bit1 右、bit2 中）。
  - `buf[6]` 滚轮 int8。
  - `buf[7]` 坐标模式标志：0 绝对、1 相对。
  - 相对模式：`dx/dy` 为 int16 小端序（手动拼接高低字节），调用 `usb_hid_mouse_rel`。
  - 绝对模式：`x/y` 为 uint16 小端序，调用 `usb_hid_mouse`。
  - 小端序拼接 `(uint16_t)buf[2] | ((uint16_t)buf[3] << 8)` 是网络字节流到主机字的经典写法。
- **键盘消息解析（0x02）**：`buf[1]` 修饰键位掩码，`buf[2..7]` 最多 6 个 HID Usage Code，调用 `usb_hid_keyboard`。

**设计要点：** 二进制协议而非 JSON，8 字节定长包，解析零开销，适合高频键鼠输入（鼠标移动可达每秒数百次）。单客户端独占通过 socket fd 比较实现，简洁有效。

### 4.8 usb_hid.c — USB HID 鼠标分段发送

```c
static void send_mouse_segments(uint8_t buttons, int32_t dx, int32_t dy, int8_t wheel)
{
    // HID 鼠标报告的 dx/dy 范围是 int8_t (-127~127)
    // 大位移需要分段发送
    while (dx != 0 || dy != 0) {
        int8_t sx = 0, sy = 0;
        if (dx > 0) sx = (dx > 127) ? 127 : (int8_t)dx;
        else if (dx < 0) sx = (dx < -127) ? -127 : (int8_t)dx;
        if (dy > 0) sy = (dy > 127) ? 127 : (int8_t)dy;
        else if (dy < 0) sy = (dy < -127) ? -127 : (int8_t)dy;
        
        tud_hid_mouse_report(HID_ITF_PROTOCOL_MOUSE, buttons, sx, sy, 0, 0);
        wait_report_sent();  // 等待 USB 传输完成
        dx -= sx;
        dy -= sy;
    }
    
    if (wheel != 0) {
        tud_hid_mouse_report(HID_ITF_PROTOCOL_MOUSE, buttons, 0, 0, wheel, 0);
        wait_report_sent();
    }
}

// HID 工作任务（高优先级，避免被 MJPEG 延迟）
static void hid_worker(void *arg)
{
    usb_hid_q_msg_t msg;
    for (;;) {
        xQueueReceive(s_hid_q, &msg, portMAX_DELAY);
        
        if (!tud_mounted()) continue;
        
        if (msg.type == Q_KEY) {
            tud_hid_keyboard_report(HID_ITF_PROTOCOL_KEYBOARD,
                                     msg.u.key.modifier, msg.u.key.keycode);
            wait_report_sent();
        } else {
            // 鼠标消息合并（减少 USB 报告数量）
            usb_hid_q_msg_t acc = msg;
            for (;;) {
                usb_hid_q_msg_t next;
                if (xQueueReceive(s_hid_q, &next, 0) != pdTRUE) {
                    process_mouse_msg(&acc);
                    break;
                }
                if (next.type == Q_KEY) {
                    process_mouse_msg(&acc);
                    tud_hid_keyboard_report(...);
                    break;
                }
                merge_mouse_msgs(&acc, &next);  // 合并连续鼠标消息
            }
        }
    }
}
```

**逐段解析：**

- **分段发送的原因**：USB HID 鼠标报告的位移字段是 int8_t（-127~127）。若浏览器发送的相对位移 `dx` 超过 127（例如快速划动），单次报告无法表达，必须拆成多个 127 的段依次发送。这是 HID 协议的固有限制。
- **分段循环**：每次取 `dx` 的不超过 127 的部分作为 `sx`，调用 `tud_hid_mouse_report`（TinyUSB API）发送，`wait_report_sent` 阻塞等待 USB 传输完成，然后 `dx -= sx` 递减，直到归零。负方向同理（-127 截断）。
- **滚轮单独发送**：因为循环中 `sx/sy` 占位时滚轮传 0，最后单独发一次滚轮报告。
- **`hid_worker` 任务**：
  - `xQueueReceive(s_hid_q, ...)` 阻塞等待消息队列。WebSocket handler 不直接调用 TinyUSB，而是将键鼠消息入队，由专门任务处理，解耦网络与 USB 的时序。
  - `tud_mounted()` 检查 USB 是否已枚举，未挂载则跳过。
  - **键盘消息**：直接发送报告。
  - **鼠标消息合并**：这是优化亮点。高频鼠标移动会产生大量队列消息，逐个发送 USB 报告会浪费带宽并增加延迟。`acc` 累积当前消息，用非阻塞 `xQueueReceive(..., 0)` 尝试取出后续消息，若同为鼠标则 `merge_mouse_msgs` 合并位移（累加 dx/dy），直到遇到键盘消息或队列为空。这样多个移动事件合并为一次（或少数几次）USB 报告。
  - 注释"高优先级，避免被 MJPEG 延迟"说明 HID 任务优先级高于编码任务，保证键鼠响应不卡顿——KVM 场景下输入延迟比帧率更重要。

**设计要点：** 队列 + 专门任务解耦了输入接收与 USB 发送；消息合并显著减少 USB 报告数；分段发送解决了 HID 协议的位移范围限制。这三者共同保障了流畅的键鼠体验。

### 4.9 ethernet.c — 以太网 + mDNS 初始化

```c
esp_err_t ethernet_init(void)
{
    esp_netif_init();
    esp_event_loop_create_default();

    // RMII MAC 配置
    eth_esp32_emac_config_t emac_config = ETH_ESP32_EMAC_DEFAULT_CONFIG();
    emac_config.interface = EMAC_DATA_INTERFACE_RMII;
    emac_config.clock_config.rmii.clock_mode = EMAC_CLK_EXT_IN;
    emac_config.clock_config.rmii.clock_gpio = CONFIG_P4KVM_ETH_RMII_CLK_GPIO;
    
    // RMII 引脚映射
    emac_config.emac_dataif_gpio.rmii.tx_en_num = CONFIG_P4KVM_ETH_RMII_TX_EN_GPIO;
    emac_config.emac_dataif_gpio.rmii.txd0_num = CONFIG_P4KVM_ETH_RMII_TXD0_GPIO;
    emac_config.emac_dataif_gpio.rmii.txd1_num = CONFIG_P4KVM_ETH_RMII_TXD1_GPIO;
    emac_config.emac_dataif_gpio.rmii.crs_dv_num = CONFIG_P4KVM_ETH_RMII_CRS_DV_GPIO;
    emac_config.emac_dataif_gpio.rmii.rxd0_num = CONFIG_P4KVM_ETH_RMII_RXD0_GPIO;
    emac_config.emac_dataif_gpio.rmii.rxd1_num = CONFIG_P4KVM_ETH_RMII_RXD1_GPIO;
    
    // SMI (MDC/MDIO) 引脚
    emac_config.smi_gpio.mdc_num = CONFIG_P4KVM_ETH_MDC_GPIO;
    emac_config.smi_gpio.mdio_num = CONFIG_P4KVM_ETH_MDIO_GPIO;

    esp_eth_mac_t *mac = esp_eth_mac_new_esp32(&emac_config, &mac_config);
    esp_eth_phy_t *phy = esp_eth_phy_new_generic(&phy_config);
    
    esp_eth_config_t eth_config = ETH_DEFAULT_CONFIG(mac, phy);
    esp_eth_driver_install(&eth_config, &s_eth_handle);
    
    // DHCP + 主机名设置
    esp_netif_set_hostname(s_eth_netif, CONFIG_P4KVM_MDNS_HOSTNAME);
    
    // mDNS 广播
    mdns_init();
    mdns_hostname_set(CONFIG_P4KVM_MDNS_HOSTNAME);  // p4kvm.local
    mdns_service_add("P4KVM", "_http", "_tcp", 80, ...);
    
    esp_eth_start(s_eth_handle);
    return ESP_OK;
}
```

**逐段解析：**

- `esp_netif_init()` 和 `esp_event_loop_create_default()` 是网络栈和事件循环的标准初始化，必须在使用任何网络功能前调用。
- **RMII MAC 配置**：
  - `ETH_ESP32_EMAC_DEFAULT_CONFIG()` 取默认配置宏。
  - `interface = EMAC_DATA_INTERFACE_RMII`：使用 RMII（Reduced MII）接口，比 MII 引脚更少（2bit 数据而非 4bit）。
  - `clock_mode = EMAC_CLK_EXT_IN`：外部时钟输入模式，参考时钟由外部 PHY 提供（GPIO 50）。ESP32-P4 的以太网 MAC 不内置时钟发生器，需外部源。
  - 随后逐一映射 RMII 数据接口引脚（TX_EN/TXD0/TXD1/CRS_DV/RXD0/RXD1）和 SMI 管理引脚（MDC/MDIO）。所有引脚号来自 Kconfig，便于不同硬件适配。
- **MAC/PHY 实例化**：`esp_eth_mac_new_esp32` 创建内部 MAC 驱动，`esp_eth_phy_new_generic` 创建通用 PHY 驱动（通过 SMI 自动探测地址为 1 的 PHY）。
- `esp_eth_driver_install` 安装以太网驱动，返回句柄 `s_eth_handle`。
- **主机名**：`esp_netif_set_hostname` 设置 DHCP 请求时携带的 hostname，使路由器 DHCP 表中显示 p4kvm。
- **mDNS**：`mdns_init` 初始化 mDNS 守护，`mdns_hostname_set` 注册 `p4kvm.local` 域名，`mdns_service_add` 广播 `_http._tcp` 服务在 80 端口。这样用户无需记 IP，浏览器直接访问 `p4kvm.local` 即可。
- `esp_eth_start` 启动以太网，触发链路检测和 DHCP。

**设计要点：** 全部引脚可配置（Kconfig），适配不同 PCB 设计。mDNS 提供零配置访问体验，是消费级 IoT 产品的标配。

### 4.10 jpeg_frame.c — 三缓冲帧管理

```c
// 选择空闲编码槽位
int jpeg_frame_pick_encode_slot(void)
{
    for (int i = 0; i < JPEG_SLOT_COUNT; i++) {
        if (g_jpeg_frame.slot_ref[i] != 0) continue;  // 被引用中，跳过
        if (g_jpeg_frame.front_idx >= 0 && i == g_jpeg_frame.front_idx) continue;  // 跳过当前传输帧
        return i;
    }
    return -1;  // 无空闲槽位
}

// 通知流媒体客户端有新帧
void jpeg_frame_notify_new_frame(void)
{
    int n = s_stream_clients;  // 当前流媒体客户端数
    if (!g_jpeg_frame.frame_ready_sem || n <= 0) return;
    
    if (n > 16) n = 16;  // 限制信号量释放数量
    for (int i = 0; i < n; i++) {
        if (xSemaphoreGive(g_jpeg_frame.frame_ready_sem) != pdTRUE) break;
    }
}

// JPEG 质量持久化到 NVS
esp_err_t jpeg_quality_save_to_nvs(uint8_t q)
{
    nvs_handle_t h;
    nvs_open("p4kvm", NVS_READWRITE, &h);
    nvs_set_u8(h, "jpeg_q", q);
    nvs_commit(h);
    nvs_close(h);
}
```

**逐段解析：**

- **`jpeg_frame_pick_encode_slot`**：遍历所有槽位，跳过两类——引用计数非零的（正在被流传输读取）和当前 `front_idx` 指向的（刚发布、可能正被读取）。找到空闲槽位返回索引，全占用返回 -1。这是编码端无阻塞推进的保障。
- **`jpeg_frame_notify_new_frame`**：
  - 读取当前流客户端数 `s_stream_clients`。
  - 无客户端或信号量未初始化则直接返回（无人等待，省 CPU）。
  - `n > 16` 截断保护，避免信号量计数溢出（计数信号量有上限）。
  - 循环 `xSemaphoreGive` 为每个客户端释放一次信号量。多客户端各自 `Take` 一次，互不抢占。
- **`jpeg_quality_save_to_nvs`**：打开 NVS 命名空间 `p4kvm`，写入键 `jpeg_q` 的 uint8 值，`nvs_commit` 刷入 flash，关闭句柄。配合 `main.c` 中的 `jpeg_quality_load_from_nvs`，实现质量参数的持久化。用户通过 HTTP 调节质量后，重启仍保留。

**设计要点：** 三缓冲是经典的"双缓冲不能满足多消费者"的升级方案。两个缓冲区时，若一个被消费者持有、一个正在被生产者写入，就没有空闲缓冲接收新帧；三缓冲留出第三个缓冲始终可供生产者使用，最大化吞吐。引用计数让多个流客户端共享同一帧而互不干扰。

## 五、API 使用指南

### 5.1 WebSocket 键鼠控制协议

**端点：** `ws://p4kvm.local/ws`

WebSocket 连接建立后，客户端发送二进制帧控制键鼠。每帧 8 字节，格式如下。

#### 鼠标消息（8 bytes）

| 字节 | 字段 | 类型 | 说明 |
|------|------|------|------|
| byte[0] | type | uint8 | 固定 `0x01`（鼠标） |
| byte[1] | buttons | uint8 | 位掩码：bit0=左键，bit1=右键，bit2=中键 |
| byte[2-3] | X / dx | uint16/int16 | 小端序；绝对模式为 X 坐标，相对模式为 dx 位移 |
| byte[4-5] | Y / dy | uint16/int16 | 小端序；绝对模式为 Y 坐标，相对模式为 dy 位移 |
| byte[6] | wheel | int8 | 滚轮滚动量 |
| byte[7] | relative | uint8 | 0=绝对坐标，1=相对坐标 |

#### 键盘消息（8 bytes）

| 字节 | 字段 | 类型 | 说明 |
|------|------|------|------|
| byte[0] | type | uint8 | 固定 `0x02`（键盘） |
| byte[1] | modifier | uint8 | 位掩码：bit0=Ctrl，bit1=Shift，bit2=Alt，bit3=GUI |
| byte[2-7] | keycodes | uint8[6] | 最多 6 个 HID Usage Code（参考 HID Usage Table） |

**示例（相对鼠标移动 +10, +20）：**
```
0x01, 0x00, 0x0A, 0x00, 0x14, 0x00, 0x00, 0x01
```

**注意：** 服务端强制单客户端独占。后连接的 WebSocket 客户端若已有活跃会话，其消息将被忽略。

### 5.2 HTTP MJPEG 流

**端点：** `GET http://p4kvm.local/stream`

- **Content-Type：** `multipart/x-mixed-replace; boundary=frame`
- **响应：** 持续推送 JPEG 帧，每帧以 `--frame` 分隔。
- **用法：** 在 HTML 中直接用 `<img src="/stream">` 即可在浏览器中显示实时画面，无需 JavaScript 解码。
- **特性：** 支持多客户端同时拉流；帧率自适应（突发帧合并，只推最新帧）；客户端断开自动释放资源。

### 5.3 JPEG 质量调节

**端点：** `GET http://p4kvm.local/jpeg-quality?q=<value>`

- **参数：** `q` 为 1-100 的整数，值越大质量越高、带宽占用越大。
- **持久化：** 调用后自动保存到 NVS，重启后保持。
- **效果：** 下一次 JPEG 编码即生效，无需重启。适合根据网络带宽动态调节画质与帧率的平衡。

## 七、项目亮点与适用场景

### 技术亮点

1. **单芯片全链路 KVM**：采集、编码、传输、USB HID 控制全部在 ESP32-P4 单颗 SoC 上完成，无需外置编码芯片或 MCU，BOM 极简。

2. **硬件加速编码**：利用 ESP32-P4 内置 JPEG 硬件编码器，1080p RGB888 实时编码，CPU 负担极低。

3. **零拷贝 DMA pipeline**：CSI DMA 直接写入 PSRAM 帧缓冲，JPEG 编码直接读取，全程无内存拷贝，降低延迟与 CPU 开销。

4. **三缓冲 + 引用计数并发模型**：生产者（编码）与多消费者（流客户端）通过引用计数无锁协作，编码永不阻塞，丢帧而非积压。

5. **HDMI 热插拔自愈**：CSI 超时自动检测断连，八步恢复流程重连，无需人工干预。

6. **异步流媒体服务器**：HTTP 请求转为异步任务处理，长连接 MJPEG 流不阻塞其他 HTTP 服务。

7. **USB HID 消息合并**：高频鼠标事件在队列层合并，减少 USB 报告数，提升响应平滑度。

8. **二进制 WebSocket 协议**：8 字节定长键鼠包，解析开销极低，适合高频输入。

9. **参数持久化**：JPEG 质量等参数存于 NVS，断电不丢失。

10. **零配置网络**：mDNS + DHCP，即插即用，浏览器访问 p4kvm.local 即可。

### 适用场景

| 场景 | 说明 |
|------|------|
| **服务器远程管理** | 机房无头服务器通过 IP KVM 远程操作 BIOS/引导阶段 |
| **嵌入式开发调试** | 远程查看目标设备 HDMI 输出并控制 |
| **多主机切换** | 物理切换 HDMI/USB 线缆连接不同主机 |
| **教学/演示** | 低成本将一台主机画面广播到多终端 |
| **安全隔离环境** | 被控主机与控制端物理网络隔离，仅通过 KVM 交互 |
| **学习 ESP32-P4 视频/网络栈** | 优秀的参考实现，涵盖 CSI、ISP、JPEG、以太网、TinyUSB 全栈 |

## 八、总结

P4KVM 是一个精巧而完整的 ESP32-P4 视频与控制参考项目。它以 TC358743 HDMI 桥接芯片为输入，借助 ESP32-P4 的 MIPI CSI、硬件 JPEG 编码器和以太网 MAC，在单芯片上实现了 1080p HDMI 采集、MJPEG 实时编码、HTTP 流媒体推送以及 USB HID 键鼠回控的完整 IP KVM 链路。

从架构上看，项目将职责清晰拆分到 capture、http_server、usb_hid、ethernet、jpeg_frame 等模块，通过三缓冲帧管理器和引用计数机制协调生产者与消费者，既保证了编码端的无阻塞推进，又支持多客户端并发拉流。CSI DMA 回调采用 IRAM 中断 + 信号量的经典异步模型，HDMI 热插拔自愈机制提升了实际可用性。USB HID 侧的消息合并与分段发送则解决了协议限制下的输入平滑性问题。

虽然项目定位为"概念验证"（Stars 仅 94，创建于 2026-06-26），但其代码质量、模块化程度和工程细节已达到实用参考级别。对于希望深入理解 ESP32-P4 视频流水线、RMII 以太网、TinyUSB HID 以及实时流媒体架构的开发者而言，P4KVM 是一份极具学习价值的开源实现。随着 ESP32-P4 生态的成熟，这类单芯片 KVM 方案有望在远程管理、工业控制和安全隔离等场景中找到更多落地机会。
