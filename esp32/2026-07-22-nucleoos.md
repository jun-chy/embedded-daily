## 一、项目简介

NucleoOS 是一个运行在 M5Stack Cardputer（ESP32-S3）上的 Web 原生操作系统。它的核心理念是 **"The OS runs on the device. Your browser is the rich operator console."**（操作系统运行在设备上，浏览器是富操作控制台）——在仅 512KB RAM、无 PSRAM 的 ESP32-S3 上，构建了一个包含离线 AI 助手、双语语音、桌面级 Web Shell、原生游戏和安全实验室的完整操作系统。

项目最引人注目的创新是 **双界面架构**：原生固件 UI 运行在 240×135 物理屏幕上（智能手表式交互），同时设备自身充当 HTTP 服务器，向局域网内任何浏览器提供桌面级 Web Shell（文件管理器、电子表格、Paint+AI、游戏中心、IDE）。重计算工作交给浏览器（WebGPU + WASM），设备只负责存储和协调——这是在无 PSRAM 微控制器上实现大规模体验的核心策略。

### 核心特性一览

| 特性 | 说明 |
|------|------|
| **离线 AI 助手 ANIMA** | C 语言编写的真实离线 NLP 引擎，检索级联 L0/L1/L2，神经符号推理，知识图谱，数学/物理求解器 |
| **三运行时一致性** | ANIMA 大脑用 C 编写一次，同时在 ESP32 固件、浏览器 WASM、PC 宿主三处运行，经奇偶校验门保证字节级一致 |
| **双语离线语音** | 拼接式 TTS 引擎（IT/EN），运行时几乎零 RAM；PTT 语音命令使用 MFCC 特征提取 + DTW 模板匹配 |
| **Web 原生架构** | 设备作为 HTTP 服务器，浏览器执行重计算（WebLLM、ONNX、ffmpeg.wasm），设备仅存储结果 |
| **事件溯源总线** | 追加式状态增量日志，支持可观测性、回放、撤销；通过 ESP-NOW 扩展到网格节点 |

| **安全实验室** | WiFi 嗅探器（.pcap）、Evil Portal、WiFi 攻击套件、BLE 套件、W5500 以太网 L2/L3 引擎、BadUSB |
| **50+ 原生应用** | 游戏（Pong/Snake/Tanks/Pinball/Poker）、媒体播放器、传感器工具、SSH 终端等 |
| **OTA A/B 回滚** | 双分区 OTA 升级，新镜像在 PENDING_VERIFY 状态下启动，核心服务就绪后确认 |
| **通用二进制** | 单一固件自动检测原版 Cardputer 与 ADV 版本（TCA8418 键盘映射 + ES8311 编解码器 HAL） |

## 四、核心代码深度分析

### 4.1 启动序列与堆内存管理（main.c）

NucleoOS 的 `app_main()` 是整个系统最精心设计的启动序列之一。在 512KB RAM、无 PSRAM 的 ESP32-S3 上，每个启动阶段都必须精确跟踪堆内存状态。

```cpp
void app_main(void)
{
    ESP_LOGI(TAG, "NucleoOS booting — last reset: %s",
             reset_reason_str(esp_reset_reason()));
    // 版本横幅 — 串口控制台上的第一行，无需网络即可识别运行镜像
    {
        const esp_app_desc_t *appd = esp_app_get_description();
        ESP_LOGI(TAG, "NucleoOS %s | proj %s | built %s %s | idf %s | %s",
                 appd ? appd->version : "?", appd ? appd->project_name : "?",
                 appd ? appd->date : "?", appd ? appd->time : "?",
                 appd ? appd->idf_ver : "?", CONFIG_IDF_TARGET);
    }
    HMEM("boot-start");
    bootmark("boot-start");
```

**逐段解析：**

- `reset_reason_str(esp_reset_reason())` — 首先记录上一次重启的原因。`esp_reset_reason()` 返回 ESP32 的复位原因寄存器值。注释详细列出了每种原因的含义：`PANIC` = 崩溃或断言失败（通常是堆 OOM 下游）；`INT/TASK_WDT` = 挂起（任务阻塞过久）；`BROWNOUT` = 3V3 电压跌落（电源/电池问题，非软件）；`POWERON/SW` = 干净启动。这是设备异常重启时的第一个诊断线索。

- `esp_app_get_description()` — 从应用分区头部读取编译时注入的版本信息（版本号、项目名、编译日期、IDF 版本）。这确保即使没有网络连接，也能通过串口识别当前运行的固件版本。

- `HMEM("boot-start")` — 堆内存快照宏。展开为一条 ESP_LOGI 日志，记录当前空闲堆、最小空闲堆、最大连续空闲块和碎片率。`HLOG` 宏受 `NUCLEO_HEAPLOG` 编译标志控制——生产构建中设为 0 时，所有 `HLOG()` 调用编译为空，零 Flash 占用、零运行时开销。

```cpp
    // 回收 BLE 控制器的空闲 DRAM（~数十 KB）
    // 蓝牙默认关闭（仅安全工具使用），在 BLE 初始化前执行
    nucleo_ble_boot_reclaim();
    bootmark("bt-reclaim");

    // 掉电安全配置存储（内部 Flash 上的 LittleFS）
    if (nucleo_storage_mount_cfg() != ESP_OK)
        ESP_LOGW(TAG, "config store (littlefs) unavailable — falling back to NVS");
    bootmark("cfg");

    nucleo_event_init();
    nucleo_log_init();
    bootmark("event+log");
```

**解析：**

- `nucleo_ble_boot_reclaim()` — 这是一个关键的内存优化。ESP32-S3 的蓝牙控制器在启动时会预留数十 KB 的 DRAM，即使蓝牙未启用。由于 NucleoOS 默认关闭蓝牙（仅在安全工具中按需启用），在 BLE 初始化前回收这些内存，为操作系统和 ANIMA AI 引擎释放宝贵的堆空间。必须在任何 BLE 初始化之前调用。

- `nucleo_storage_mount_cfg()` — 在内部 Flash 上挂载 LittleFS 分区（标签 "cfg"），用于存储掉电安全的配置数据。选择 LittleFS 而非 SD 卡上的 FAT 文件系统，是因为 FAT 在写入中断时可能损坏，而 LittleFS 是掉电安全的日志结构文件系统。如果挂载失败，回退到 NVS（非易失性存储）。

```cpp
    // 诊断信息：上次重启原因 + ANIMA 在上次重启时所处的阶段
    // g_anima_stage 在热重启后存活（RTC 内存）
    {
        extern uint32_t g_anima_stage; extern uint8_t g_anima_phase;
        esp_reset_reason_t rr = esp_reset_reason();
        ESP_LOGW(TAG, "DIAG last-reset=%s anima_phase=0x%02X anima_stage=0x%02X",
                 reset_reason_str(rr), (unsigned)g_anima_phase, (unsigned)g_anima_stage);
        g_anima_stage = 0; g_anima_phase = 0;  // 消费面包屑
    }
```

**解析：** `g_anima_stage` 和 `g_anima_phase` 存储在 RTC 内存中——ESP32-S3 的 RTC FAST 内存区域在深度睡眠和热重启后保持不变。这样当设备崩溃重启后，日志中可以看到崩溃时 ANIMA AI 引擎正在执行的阶段和步骤，为远程诊断（通过 WiFi `/api/logs`）提供关键线索。使用 ESP_LOGW（WARN 级别）确保它进入日志环缓冲区。

```cpp
    // OTA 确认：新镜像在 PENDING_VERIFY 状态下启动
    // 核心服务就绪后确认，取消回滚
    if ((!solo || solo_rec) && sd_ok && nucleo_recorder_init() != ESP_OK)
        ESP_LOGW(TAG, "recorder init failed");

    // ... 更多服务初始化 ...

    ota_confirm_if_pending();
```

```cpp
static void ota_confirm_if_pending(void)
{
    const esp_partition_t *run = esp_ota_get_running_partition();
    esp_ota_img_states_t st;
    if (run && esp_ota_get_state_partition(run, &st) == ESP_OK
        && st == ESP_OTA_IMG_PENDING_VERIFY) {
        if (esp_ota_mark_app_valid_cancel_rollback() == ESP_OK)
            ESP_LOGI(TAG, "OTA image confirmed valid (rollback cancelled)");
    }
}
```

**解析：** ESP-IDF 的 OTA A/B 分区机制中，新固件写入后首次启动处于 `ESP_OTA_IMG_PENDING_VERIFY` 状态。如果在下一次重启前没有调用 `esp_ota_mark_app_valid_cancel_rollback()` 确认，引导加载器会回滚到上一个已知良好的固件镜像。NucleoOS 在核心服务（存储 + 网络 + HTTP 服务器）全部就绪后才确认——如果新固件在启动过程中崩溃，它永远不会被确认，下次重启自动回滚。这是防变砖的关键设计。

### 4.2 Solo 启动模式（main.c）

Solo 模式是 NucleoOS 在无 PSRAM 芯片上运行重型应用的核心创新。

```cpp
    // ANIMA Solo 个性化：用户从完整 OS 中打开了助手，设置了 RTC 标志并重启到此
    // 只启动助手需要的东西 — 显示 + SD + WiFi + 电源 + 仲裁器 + ANIMA/TTS/voice
    // 跳过 httpd / mDNS / recorder / auth / IR / 启动器应用注册
    // 堆内存因此大且未碎片化（没有 18KB httpd 块切割它），
    // 这是 PSRAM-less 芯片同时容纳在线 TLS + L1 + voice 的唯一方式
    const bool solo = nucleo_anima_solo_pending();
    const bool solo_rec = solo && (nucleo_app_solo_app() == 2);  // Recorder 配置
    const bool solo_srv = solo && nucleo_app_solo_is_server();   // Web Client 配置
```

**解析：** Solo 模式通过 RTC 内存标志触发——用户从完整 OS 中选择一个重型应用（如 ANIMA AI 助手），系统设置 RTC 标志并重启。重启后只加载该应用所需的组件，跳过 httpd（18KB）、mDNS（12KB）、recorder、auth 等不必要的服务。这确保了堆内存大且连续——在 PSRAM-less 的 ESP32-S3 上，同时运行在线 TLS 会话 + ANIMA L1 知识库 + 语音引擎的唯一方式就是避免 httpd 的 18KB 内存块碎片化堆空间。

```cpp
    // ANIMA Solo 在专用大栈任务上运行 nucleo_app_run()
    // （8KB 主任务栈在运行 ANIMA UI 时溢出）
    static void anima_solo_task(void *arg) {
        (void)arg;
        nucleo_app_run();
        vTaskDelete(NULL);
    }
```

**解析：** ANIMA AI 助手的 UI 栈深度超过了默认 8KB 主任务栈的容量。通过创建一个专用的大栈任务来运行 `nucleo_app_run()`，避免栈溢出。`vTaskDelete(NULL)` 是防御性代码——`nucleo_app_run()` 永不返回（Esc 键重启设备），但万一它返回了，任务会自删除而非崩溃。

### 4.3 板级抽象与引脚冲突管理（nucleo_board.h）

```cpp
// microSD over SPI。重要：M5GFX 在 SPI3_HOST 上驱动 Cardputer 显示器
// （LovyanGFX 在 ESP32-S3 上的默认值），而 nucleo_ui_init() 在 SD 挂载之前
// 运行 d.init() — 所以 SD 必须使用另一个主机（SPI2），否则
// spi_bus_initialize() 返回 ESP_ERR_INVALID_STATE 且卡永远不挂载
#define NUCLEO_SD_SPI_HOST   SPI2_HOST
#define NUCLEO_SD_PIN_SCLK   40
#define NUCLEO_SD_PIN_MISO   39
#define NUCLEO_SD_PIN_MOSI   14
#define NUCLEO_SD_PIN_CS     12
```

**解析：** 这是一个极其重要的硬件设计细节。ESP32-S3 的 SPI 外设有多个主机（SPI2_HOST 和 SPI3_HOST），但每个主机一次只能驱动一个设备。M5GFX/LovyanGFX 库默认使用 SPI3_HOST 驱动 ST7789 显示屏，且在 `nucleo_ui_init()` 中先于 SD 卡初始化。如果 SD 卡也使用 SPI3_HOST，`spi_bus_initialize()` 会返回 `ESP_ERR_INVALID_STATE`（总线已被占用），导致 SD 卡永远无法挂载。解决方案是 SD 卡使用 SPI2_HOST。由于 ESP32-S3 的 GPIO 矩阵可以路由任何引脚到任何 SPI 主机，引脚分配不受主机选择限制。

```cpp
// 内置 PDM 麦克风（SPM1423）。Cardputer 将 PDM 时钟连接到 GPIO43，
// PDM 数据线连接到 GPIO46。
#define NUCLEO_MIC_PIN_CLK   43
#define NUCLEO_MIC_PIN_DATA  46

// 内置扬声器放大器（I2S 标准模式）。
// 重要：扬声器 WS 线是 GPIO43 — 与上面的 PDM 麦克风时钟引脚相同。
// 因此录音和播放不能同时运行；nucleo_audio 在录音活跃时拒绝启动
#define NUCLEO_SPK_PIN_BCLK  41
#define NUCLEO_SPK_PIN_WS    43
#define NUCLEO_SPK_PIN_DOUT  42
```

**解析：** M5Stack Cardputer 的硬件设计中，PDM 麦克风的时钟引脚（GPIO43）与 I2S 扬声器的 WS（字选择）引脚共用。这意味着录音和播放不能同时进行——`nucleo_audio` 模块在录音活跃时拒绝启动，只在播放时才占用 I2S TX 引脚。这是一个硬件层面的限制，固件通过软件互斥来管理。这种设计在消费级嵌入式设备中很常见，是为了减少 GPIO 占用和 PCB 布线复杂度。

```cpp
// 诊断堆/RAM 日志门控
#ifndef NUCLEO_HEAPLOG
#define NUCLEO_HEAPLOG 1
#endif
#if NUCLEO_HEAPLOG
#define HLOG(tag, ...) ESP_LOGI(tag, __VA_ARGS__)
#else
#define HLOG(tag, ...) ((void)0)
#endif
```

**解析：** `HLOG` 宏是一个编译时门控的日志宏。在开发/调试构建中（`NUCLEO_HEAPLOG=1`），它展开为 `ESP_LOGI`，输出详细的堆内存跟踪信息。在生产构建中（`NUCLEO_HEAPLOG=0`），它展开为 `((void)0)`（空操作），编译器完全移除所有相关代码——零 Flash 占用、零 RAM 占用、零运行时开销。这确保了诊断日志不影响生产环境的性能和内存预算。注释详细解释了为什么需要这个门控：每次 `HLOG` 调用都会格式化文本到日志器的栈缓冲区，并搅动固定的 4KB RAM 环形缓冲区。

### 4.4 事件溯源总线（nucleo_eventbus.c）

事件总线是 NucleoOS 的架构核心——所有组件通过发布/订阅模式通信，事件被持久化到 SD 卡的 NDJSON 日志中，支持回放和远程可观测性。

```cpp
#define RING 16   // 重连订阅者的回放窗口；16 个事件在单用户设备上足够
                  // 32->16 释放 ~4.3KB .bss（event_t = 272B）给 PSRAM-less 堆
#define TOPIC_MAX   48     // 主题名最大长度
#define PAYLOAD_MAX 208    // 载荷最大长度
#define JOURNAL     "/sd/journal/events.ndjson"
#define JOURNAL_CAP (128 * 1024)  // 超过此大小则轮转

typedef struct {
    uint32_t seq;
    int64_t ts;
    char topic[TOPIC_MAX];
    char payload[PAYLOAD_MAX];
} event_t;

static event_t s_ring[RING];        // 环形缓冲区
static uint32_t s_seq;              // 全局序列号
static SemaphoreHandle_t s_lock;    // 保护 ring + seq
static SemaphoreHandle_t s_jlock;   // 序列化日志写入者（SD I/O 在 s_lock 外）
```

**解析：**

- `RING = 16` — 环形缓冲区容量从 32 降到 16，释放 4.3KB 的 `.bss` 段内存。每个 `event_t` 结构体占用 272 字节（4 + 8 + 48 + 208 + 对齐填充），16 个共约 4.3KB。在 PSRAM-less 的 ESP32-S3 上，这 4.3KB 对 ANIMA L1 知识库的连续内存至关重要。16 个事件的回放窗口对于单用户设备已经足够。

- 双锁设计是此模块最精妙之处：`s_lock` 保护内存中的环形缓冲区和序列号（持锁时间极短，绝不持有期间做 SD I/O）；`s_jlock` 序列化 SD 日志写入者。这种分离确保了 SD 卡 I/O 延迟不会阻塞事件发布者和其他读取者。

```cpp
// 高频/瞬态主题仅实时传播：不持久化到日志
static bool journal_skip(const char *topic)
{
    return !strcmp(topic, "rec.level")  || !strcmp(topic, "voice/state") ||
           !strcmp(topic, "voice/match") || !strcmp(topic, "system.busy");
}
```

**解析：** 某些事件（录音电平、语音状态、系统忙闲标志）频率极高——录音电平每分钟约 75 次，语音状态每次发声都触发。如果将这些事件写入 SD 卡日志，会在实时语音/录音循环中引入 SD I/O 延迟，且日志文件无限增长。这些事件仍然通过 WebSocket 实时传播（`s_sink` 回调），只是不持久化到磁盘。

```cpp
static void journal_append(const event_t *e)
{
    if (journal_skip(e->topic)) return;
    // O(1) 大小轮转：文件超过上限时重命名为 ".0"（覆盖旧备份），然后重新开始
    struct stat st;
    if (stat(JOURNAL, &st) == 0 && (long)st.st_size > JOURNAL_CAP) {
        remove(JOURNAL ".0");
        rename(JOURNAL, JOURNAL ".0");
    }
    FILE *f = fopen(JOURNAL, "a");
    if (!f) return;
    fprintf(f, "{\"t\":\"%s\",\"seq\":%u,\"ts\":%lld,\"d\":%s}\n",
            e->topic, (unsigned)e->seq, (long long)e->ts, e->payload);
    fclose(f);
}
```

**解析：** 日志轮转采用 O(1) 策略——当文件超过 128KB 上限时，直接重命名为 `.0` 备份（覆盖旧备份），然后创建新文件。这是一个 `rename` 操作，无需读取/解析/缓冲，因此历史日志在磁盘上有界（~2×上限），RAM 占用接近零。日志格式为 NDJSON（Newline Delimited JSON），每行一个事件对象，便于后续按行解析。

```cpp
uint32_t nucleo_event_publish(const char *topic, const char *payload_json)
{
    if (!payload_json || !payload_json[0]) payload_json = "{}";
    // sink 在锁外运行（它可能在 WS 层阻塞），所以必须获得栈拷贝：
    // 指向环形槽位的指针会在后续发布者环绕环形（RING=16）后失效
    char t_copy[TOPIC_MAX], p_copy[PAYLOAD_MAX];
    xSemaphoreTake(s_lock, portMAX_DELAY);
    uint32_t seq = ++s_seq;
    event_t *e = &s_ring[seq % RING];
    e->seq = seq;
    e->ts = time(NULL);
    strncpy(e->topic, topic, TOPIC_MAX - 1); e->topic[TOPIC_MAX - 1] = '\0';
    strncpy(e->payload, payload_json, PAYLOAD_MAX - 1); e->payload[PAYLOAD_MAX - 1] = '\0';
    memcpy(t_copy, e->topic, TOPIC_MAX);
    memcpy(p_copy, e->payload, PAYLOAD_MAX);
    int64_t ts_copy = e->ts;
    xSemaphoreGive(s_lock);
    // 日志写入在 s_lock 外：journal_append 做 SD I/O
    if (s_jlock && xSemaphoreTake(s_jlock, pdMS_TO_TICKS(500)) == pdTRUE) {
        event_t je; je.seq = seq; je.ts = ts_copy;
        memcpy(je.topic, t_copy, sizeof je.topic);
        memcpy(je.payload, p_copy, sizeof je.payload);
        journal_append(&je);
        xSemaphoreGive(s_jlock);
    }
    fan_sinks(seq, t_copy, p_copy, NULL);
    return seq;
}
```

**逐段解析：**

这是事件总线的核心发布函数，包含了多个精心设计的并发安全策略：

- **栈拷贝策略**：`t_copy` 和 `p_copy` 是发布者栈上的临时副本。为什么需要拷贝？因为 sink 回调（WebSocket 广播器）在锁外执行，它可能阻塞在网络 I/O 上。如果直接传递指向环形缓冲区槽位的指针，后续的发布者会覆盖该槽位（RING=16，17 个事件后回绕），导致 WS 帧携带另一个事件的载荷。约 256 字节的栈开销是值得的代价。

- **双锁分离**：`s_lock` 只在访问环形缓冲区时持有（极短），释放后才获取 `s_jlock` 进行 SD I/O。注释用意大利语和英语混合解释了一个曾经出现的 bug：如果 SD I/O 在 `s_lock` 下执行，慢速/争用的 SD 写入（如 ANIMA 日历写入后的同一任务）会持有环形锁整个 I/O 期间，导致所有其他发布/读取事件的任务阻塞在 `portMAX_DELAY` 上 = 设备完全冻结，无重启（IDLE 喂看门狗但不触发 WDT）。

- **限时等待**：`xSemaphoreTake(s_jlock, pdMS_TO_TICKS(500))` — 日志锁等待上限 500ms，而非 `portMAX_DELAY`。如果日志写入者卡在 SD I/O 上（卡慢/缺失/堆碎片化），其他发布者不应永远阻塞。超时后跳过磁盘写入，但事件仍通过实时 sink 传播。这是"尽力而为持久化"策略。

```cpp
int nucleo_event_copy_since(uint32_t since, char *out, size_t out_sz)
{
    xSemaphoreTake(s_lock, portMAX_DELAY);
    // 如果间隙大于环形缓冲区，客户端必须重新同步
    if (s_seq > since && s_seq - since > RING) {
        xSemaphoreGive(s_lock); out[0] = '\0'; return -1;
    }
    size_t n = 0; int count = 0;
    n += snprintf(out + n, out_sz - n, "[");
    for (uint32_t seq = since + 1; seq <= s_seq; seq++) {
        event_t *e = &s_ring[seq % RING];
        if (e->seq != seq) continue;  // 已被覆盖
        int w = snprintf(out + n, out_sz - n,
                         "%s{\"t\":\"%s\",\"seq\":%u,\"ts\":%lld,\"d\":%s}",
                         count ? "," : "", e->topic, (unsigned)e->seq,
                         (long long)e->ts, e->payload);
        if (w < 0 || (size_t)w + 1 >= out_sz - n) { out[n] = '\0'; break; }
        n += (size_t)w;
        count++;
    }
    snprintf(out + n, out_sz - n, "]");
    xSemaphoreGive(s_lock);
    return count;
}
```

**解析：** `copy_since` 用于 WebSocket 客户端重连后的状态同步——客户端发送它最后收到的序列号 `since`，服务器返回此后所有未确认的事件。如果 `s_seq - since > RING`（间隙超过 16），说明客户端离线太久，环形缓冲区已经回绕覆盖了它错过的事件，返回 -1 要求完全重新同步。`e->seq != seq` 检查是必要的——虽然索引计算指向某个槽位，但如果该槽位已被后续事件覆盖，其 `seq` 字段不等于请求的 `seq`，跳过即可。`snprintf` 的返回值检查确保不会输出截断的事件——如果剩余空间不足以容纳完整事件加闭合 `]`，立即停止。

### 4.5 语音命令引擎（nucleo_voice.h）

NucleoOS 实现了一个完整的端上语音命令系统，使用 MFCC 特征提取和 DTW 模板匹配，完全离线运行。

```cpp
// 端上 Push-To-Talk 语音命令引擎 (AVCEB)
// 在 FN 键按住时采集 I2S PCM 音频，实时提取 MFCC 特征，
// 并使用动态时间规整（DTW）与保存的模板进行比较。
// 匹配成功时向 nucleo_eventbus 发送事件。
```

**解析：** AVCEB（Audio Voice Command Engine - Built-in）是 NucleoOS 的语音命令引擎。与传统云端语音识别不同，它完全在 ESP32-S3 上运行：I2S 采集 PCM 音频 → 实时提取 MFCC（Mel-Frequency Cepstral Coefficients）特征 → DTW（Dynamic Time Warping）与用户预录的模板比较。这种方案的优点是完全离线、隐私安全、零网络延迟；缺点是需要用户预先录制每个命令的模板（通过学习模式），且识别词汇有限。

```cpp
// 初始化语音引擎后台任务。必须在 I2S 和键盘初始化后调用。
esp_err_t nucleo_voice_init(void);

// 手动开关引脚（粘性）。true 时延迟启动 16KB 引擎，false 时若无其他持有者则释放
void nucleo_voice_enable(bool enable);

// 应用/UI 持有：在前台应用需要时保持引擎运行；离开时传 false。
// 引擎在最后一个持有者离开时自删除（归还其 16KB）
void nucleo_voice_request(bool on);
```

**解析：** 语音引擎采用**延迟加载 + 引用计数**策略。引擎本身占用 16KB 堆内存，在无 PSRAM 的 ESP32-S3 上是一笔可观的开销。因此引擎不是开机即启动，而是按需加载：

- `nucleo_voice_enable(bool)` — 粘性开关，用户在设置中启用语音后持久化到 SD 卡。
- `nucleo_voice_request(bool)` — 引用计数式持有，多个应用可以同时请求保持引擎运行。当最后一个持有者释放时，引擎自动卸载并归还 16KB 堆内存。

这种设计确保了语音引擎只在真正需要时占用内存，不与重型应用争抢堆空间。

```cpp
// Push-to-Talk 触发，由主 UI 循环驱动（按钮时序的唯一所有者）。
// true = 立即开始监听：延迟启动引擎，播放"就绪"和弦音，然后按住时采集。
// false = 停止并识别。引擎在突发处理完毕后释放
// （除非 always-on 或前台应用保持固定）。
void nucleo_voice_ptt(bool on);

// 独占模式临时挂起：释放引擎但不清除持有者，
// 在 suspend(false) 时仅当仍有持有者时恢复
void nucleo_voice_suspend(bool suspend);
```

**解析：** PTT（Push-To-Talk）由主 UI 循环驱动而非引擎自身轮询按钮——这避免了手电筒按键和语音保持按键在同一按下时同时触发。`nucleo_voice_suspend` 是独占模式的配套接口：当重型应用需要最大堆内存时，临时挂起语音引擎（释放 16KB）但不清除持有者计数；应用退出后自动恢复。

```cpp
// 触发"学习模式"。下次使用 PTT 时，引擎不执行比较，
// 而是将提取的 MFCC 模板保存到指定文件名（如 "macro_L.tpl"）到 /sd/voice/
esp_err_t nucleo_voice_arm_learning_mode(const char *template_filename);

// 上次每词匹配遥测（DTW 距离）。seq 在每次新匹配时递增，
// 轮询者可据此检测新匹配
typedef struct {
    char    word[32];
    int32_t dist, second, radius;
    uint32_t seq;
} nucleo_voice_match_t;
bool nucleo_voice_last_match(nucleo_voice_match_t *out);
```

**解析：** 学习模式允许用户为每个命令录制自己的语音模板。`nucleo_voice_match_t` 结构体暴露了 DTW 匹配的内部遥测：`dist` 是最佳匹配的 DTW 距离，`second` 是第二最佳距离，`radius` 是置信半径。`dist / second` 的比值可以用来评估匹配置信度——如果最佳和第二最佳距离接近，说明匹配不确定。`seq` 字段递增让轮询者能检测新匹配（而非重复读取旧值）。

### 4.6 离线 AI 检索级联（ANIMA 引擎）

ANIMA 是 NucleoOS 的离线 AI 助手引擎，采用多级检索级联架构，在 ESP32-S3 的有限资源下实现智能问答。

```
查询 → L0  意图/FAQ 字符串匹配           (<1ms, 无SD, 无模型)   70-90% 的使用
      L1  二进制 Hamming (popcount) 粗筛 + int8 重排, 置信度门控
      L2  MOSAICO 片段拼接 — 逐字引用, 带来源标注 (绝不虚构)
      L3  可选云端 LLM — 否则诚实地回答"我不知道"
```

**解析：** ANIMA 的检索级联是一个成本递增的漏斗设计：

- **L0（<1ms）**：纯字符串匹配，检查意图和常见问题。无 SD 卡访问、无模型推理。覆盖 70-90% 的日常使用——"几点了"、"打开音乐"、"电池电量"等。这是零成本快速路径。

- **L1（二进制 Hamming）**：使用 Matryoshka **二进制**嵌入（约 64 字节/条），距离计算为 `popcount(a XOR b)`——纯整数运算，无需 FPU，理想适合 MCU。仅对 L0 粗筛后的少数候选进行 int8 精排（约数十 KB/查询）。Matryoshka 嵌入的特殊之处在于其前 16 维前缀可以预取可能的 SD 卡簇——在 Core 0 完成完整编码的同时，Core 1 并行预取存储，实现计算与 I/O 重叠。

- **L2（MOSAICO 片段拼接）**：从知识库中逐字提取相关片段并拼接，每个片段带来源标注。关键特性是**绝不虚构**——所有回答都基于知识库中的真实文本。

- **L3（可选云端）**：当本地检索置信度不足时，可选调用云端 LLM（Claude/Groq/Grok/Gemini）。如果用户未配置云端或选择纯离线模式，ANIMA 诚实地回答"我不知道"——这是**证据性弃权**（CRAG 风格门控）：高置信度 → 冻结答案，中置信度 → 拼接，低置信度 → 拒绝。

### 4.7 启动追踪与诊断（main.c）

```cpp
// 启动面包屑：记录到串口控制台并追加到 /sd/boot_trace.txt
// SD 挂载前的步骤缓冲在 RAM 中，由 bootmark_begin() 刷新
static char s_pre[480]; static int s_pre_n = 0;   // 挂载前面包屑缓冲
static bool s_sd_up = false;

static void bootmark(const char *step)
{
    unsigned fr = (unsigned)heap_caps_get_free_size(MALLOC_CAP_INTERNAL);
    unsigned lg = (unsigned)heap_caps_get_largest_free_block(MALLOC_CAP_INTERNAL);
    ESP_LOGW(TAG, "BOOTSTEP %s free=%u largest=%u", step, fr, lg);
    if (!s_sd_up) {                               // SD 未挂载 -> 缓冲
        int rem = (int)sizeof(s_pre) - s_pre_n;
        if (rem > 1) {
            int w = snprintf(s_pre + s_pre_n, rem, "%s  free=%u largest=%u\n", step, fr, lg);
            s_pre_n += (w > 0 && w < rem) ? w : (rem - 1);
        }
        return;
    }
    FILE *f = fopen(NUCLEO_SD_MOUNT "/boot_trace.txt", "a");
    if (f) { fprintf(f, "%s  free=%u largest=%u\n", step, fr, lg); fclose(f); }
}
```

**解析：** `bootmark` 是一个启动阶段追踪系统。每个关键启动步骤都调用 `bootmark("step-name")`，记录步骤名称和当时的堆内存状态（空闲字节数 + 最大连续块）。这些信息同时输出到串口控制台（`ESP_LOGW`）和 SD 卡文件（`/sd/boot_trace.txt`）。

关键设计：SD 卡挂载前的步骤无法写入文件，因此缓冲在 480 字节的 RAM 数组 `s_pre` 中。`bootmark_begin()` 在 SD 挂载后调用，创建新追踪文件并刷新缓冲区。这样即使设备在启动过程中挂起（如 SD 初始化卡住），用户也可以在 PC 上读取 SD 卡的 `boot_trace.txt` 文件，看到设备挂起前到达的最后一个步骤——这是无串口环境下的关键诊断手段。

### Python 示例（事件监控）

```python
import websocket
import json
import threading

class NucleoOSMonitor:
    def __init__(self, ip):
        self.url = f"ws://{ip}/api/events"
        self.last_seq = 0

    def on_message(self, ws, message):
        event = json.loads(message)
        self.last_seq = event.get("seq", 0)
        topic = event.get("t", "")
        data = json.loads(event.get("d", "{}"))

        if topic == "voice/match":
            print(f"Voice: {data.get('word')} (dist={data.get('dist')})")
        elif topic == "anima/state":
            print(f"ANIMA: {data.get('phase')} - {data.get('text','')}")
        elif topic == "system.busy":
            print(f"System busy: {data.get('app')}")

    def run(self):
        ws = websocket.WebSocketApp(
            self.url,
            on_message=self.on_message,
            header={"Cookie": "nucleo_session=YOUR_COOKIE"}
        )
        ws.run_forever(reconnect=5)

monitor = NucleoOSMonitor("192.168.1.100")
monitor.run()
```

## 七、项目亮点与适用场景

### 项目亮点

1. **无 PSRAM 上的完整 OS** — 在仅 512KB RAM、无 PSRAM 的 ESP32-S3 上实现了一个包含 50+ 应用、离线 AI、语音、游戏的完整操作系统。核心策略是"重计算交给浏览器"——WebGPU 图像生成、WASM LLM 推理、ffmpeg 视频处理都在浏览器中执行，设备仅负责存储和协调。

2. **三运行时 AI 一致性** — ANIMA AI 引擎用 C 编写一次，同时编译为 ESP32 固件、浏览器 WASM 和 PC 可执行文件三个目标。奇偶校验门确保三个运行时对同一查询给出字节级一致的回答。这不是简单的"移植"，而是同一份 C 代码的跨平台编译。

3. **Solo 启动模式** — 通过 RTC 标志在重启时跳过不必要的服务（httpd 18KB + mDNS 12KB），为单个重型应用提供大而连续的堆内存。每个游戏或 AI 会话都重启到一个全新的堆中，避免了长时间运行导致的堆碎片化。

4. **事件溯源架构** — 所有组件通过事件总线通信，事件持久化为 NDJSON 日志，支持回放、撤销和远程可观测性。双锁设计（内存锁 + 日志锁）确保 SD I/O 延迟不阻塞实时事件传播。

5. **离线语音命令** — 完全端上的 MFCC + DTW 语音识别，无需云端，隐私安全。延迟加载 + 引用计数策略确保 16KB 引擎只在需要时占用内存。

6. **安全实验室** — 清洁室实现的 WiFi/BLE/以太网攻击工具，包括 W5500 MACRAW L2/L3 引擎（自写 DHCP 客户端、ARP 欺骗、MITM），所有攻击均需用户明确同意（consent-gated）。

7. **启动追踪系统** — 每个启动阶段记录到 SD 卡的 `boot_trace.txt`，SD 挂载前的步骤缓冲在 RTC 内存中。即使设备在启动过程中挂起且无串口连接，用户也能从 SD 卡诊断挂起位置。

### 适用场景

| 场景 | 说明 |
|------|------|
| **嵌入式 OS 设计参考** | 学习在资源受限 MCU 上构建多应用操作系统的架构模式 |
| **ESP-IDF 高级应用** | OTA A/B 回滚、NVS 配置、LittleFS 掉电安全存储、堆碎片化管理 |

## 八、总结

NucleoOS 是一个在 ESP32-S3 上实现的令人惊叹的完整操作系统，它重新定义了微控制器能做什么的边界。在仅 512KB RAM、无 PSRAM 的硬件约束下，项目实现了离线 AI 助手、双语语音、桌面级 Web Shell、50+ 原生应用和完整的安全实验室——这在传统嵌入式开发中被认为是不可能的。

项目的核心创新在于**Web 原生架构**：设备固件负责 I/O 和协调，浏览器负责重计算（WebGPU、WASM、ONNX）。这种分工让无 PSRAM 的微控制器能够"承载"远超其硬件能力的体验——AI 图像生成在浏览器 GPU 上运行，LLM 推理在 WASM 中执行，设备只存储结果。

ANIMA 离线 AI 引擎是另一个技术亮点。检索级联 L0/L1/L2 架构通过成本递增的漏斗设计，让 70-90% 的查询在 <1ms 内完成（纯字符串匹配），只有少数复杂查询才需要 SD 卡 I/O 和模型推理。Matryoshka 二进制嵌入 + popcount 距离计算是专为无 FPU MCU 设计的方案。证据性弃权策略确保 AI 绝不虚构事实——高置信度回答，低置信度诚实拒绝。

事件溯源总线的设计也值得深入学习。双锁分离（内存锁 vs 日志锁）确保 SD I/O 延迟不阻塞实时事件传播；栈拷贝策略避免环形缓冲区回绕导致的 use-after-overwrite；限时日志锁防止卡死的日志写入者冻结整个系统。这些并发安全细节在资源受限的嵌入式环境中尤为关键。

对于嵌入式开发者而言，NucleoOS 是一个宝贵的参考实现——它展示了如何在极端资源约束下构建复杂系统，如何通过架构设计（Solo 模式、独占模式、延迟加载）在有限内存中运行多个重型应用，以及如何将嵌入式设备与现代 Web 技术（WASM、WebGPU、PWA）无缝集成。
